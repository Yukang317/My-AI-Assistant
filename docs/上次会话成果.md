## 本会话完成（6/28 第 29 次）

**7.0 Bug 根因分析与修复**

- **定位** `/chat/graph` **前端不显示回复的根因**：追踪日志 `agent_trace.jsonl` 揭示 `result_synthesis.py:67` `msg["role"]` 对 `HumanMessage` 下标访问崩溃 → `'HumanMessage' object is not subscriptable`
- **根因链**：`state.py:88` `add_messages` reducer 自动把 `load_context` 返回的 dict 转成 LangChain 对象 → `result_synthesis` 用 dict 语法访问对象 → 选择性 bug（新会话无历史不触发，复用会话有历史就炸）
- **修复两个文件**：`load_context.py`（源头转 LangChain 对象）+ `result_synthesis.py`（用 `isinstance` 兼容两种类型）
- **验证通过**：`run_graph('golden-mem-read-3f7c43b4', ...)` 从崩溃修复到 5.7s 成功返回

**前端验证**：用户最新测试"哦我的沟通偏好是什么？"前端正常返回内容 ✅

**本次核心讨论**：


| 问题                               | 结论                                                                                                                                                                                     |
| -------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 为什么 6月25日 没崩溃但 6月27日 崩溃？         | 6月25日是全新会话（PG无历史），for 循环未执行；6月27日复用会话（PG有历史），for 循环执行触发 bug                                                                                                                            |
| `add_messages` reducer 的类型转换机制   | 自动把 dict 转成 LangChain 消息对象，是便利但危险的设计                                                                                                                                                   |
| 修复为什么有效？                         | 修复后 `run_graph` 成功返回 content SSE 事件（而非 error 事件），前端正常显示（详见 dev_log 第四章"讲解修复原理"）                                                                                                        |
| MEMORY.md "沟通偏好" 3条重复为何没被 dedup？ | **用户实测确认**：新问题"我是太原理工毕业的"与"我的本科专业是软件工程哦"触发了"状态/背景"分区的更新+去重，可能的原因：但 `_dedup_bullets` 是**按 section 执行的**——只去重收到新补丁的那个 section。"沟通偏好"没收到新补丁，所以 3 条重复保留。`merge_patch_into_memory` 不是全文件去重。 |


**记录待修问题**：

1. 🟡 前端不展示 error SSE 事件（显示"思考中"而非错误信息）
2. 🟡 MEMORY.md "沟通偏好" 3 条重复未去重 —— **已确认根因**：`_dedup_bullets` 按 section 执行，不是全文件扫描。只有该 section 收到新补丁时才去重。

详情见 `dev_log/260628_阶段7.0_bug根因分析与修复.md`
