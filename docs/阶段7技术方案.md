## 项目下一阶段开发路线(阶段 7)

### 定位与目标

- **求职定位**:简历首条 = "用 LangGraph 构建了具备任务规划能力的个人 Agent,能分解多步任务并自主执行";面试主打 Plan-and-Execute 设计与 tradeoff。
- **自用定位**:灌入自己的 dev_log/docs,日常查资料和做学习规划真的走它。
- **节奏**:中等(1-2 个月),分 5 个子阶段,每阶段都产出可展示物。

### 当前真实状态(已核实)

- Agent 是「路由 → 单工具/循环 → 合成」,本质是 RAG + 工具调用,**不是真正的任务规划**。
- `agent/harness/` 目录为空,Harness 实际只有 [bound.py](personal_assistant/agent/bound.py) + [trace.py](personal_assistant/agent/trace.py)。
- [MEMORY.md](personal_assistant/memory/MEMORY.md) 有重复条目,[memory_update.py](personal_assistant/agent/nodes/memory_update.py) 的 `merge_patch_into_memory` 去重逻辑有 bug。
- 灵感引擎 Phase 1 代码完成,**从未在真实文档上端到端跑过**。
- RAG 全链路搭好,但库里没有真实文档。

### 阶段 7.1 — 旗舰:任务规划 Plan-and-Execute(约 2-3 周)

目的:核心差异化能力,简历首条 + 面试主打。

- 新增 `task` 意图:扩展 [router.py](personal_assistant/agent/router.py) 四分类 → 五分类。
- 新建 plan-execute 子图(参照灵感引擎的子图拆分方式 [inspire_diverge.py](personal_assistant/agent/inspire_diverge.py)/[inspire_converge.py](personal_assistant/agent/inspire_converge.py)):
  - `planner`:LLM 把任务分解为有序步骤(JSON,带兜底解析,复用 inspire 的 `_parse_queries_json` 模式)。
  - `step_executor`:逐步执行,**复用现有** `tool_execute` 循环 + [bound.py](personal_assistant/agent/bound.py) 安检 + [trace.py](personal_assistant/agent/trace.py) 打点。

  - `aggregate`:汇总各步结果为结构化交付。
- [state.py](personal_assistant/agent/state.py) 新增字段:`PLAN`(步骤列表)、`CURRENT_STEP`、`STEP_RESULTS`、`PLAN_STOP_REASON`。
- [main_graph.py](personal_assistant/agent/main_graph.py) 接线:`task` → 规划子图 → memory_update。
- mock 测试:仿 [agent_inspire_test260626.py](personal_assistant/test) 写 planner/executor/replan 场景。

### 附录:每次新开窗口如何自动加载"开发习惯 + 项目状态"

> 目的:无论用 Cursor 聊天框、Claude 还是其他工具,**每开一个新窗口/会话**,助手都能立刻知道「你是谁、怎么协作、项目现在到哪了、下一步做什么」,不用每次从头交代。这套逻辑你已具备,这里把它显式记录下来作为长期约定。

#### 三层文件机制(各管一件事)

- **习惯层 — `CLAUDE.md`(只读,always-applied)**
  - 新窗口自动加载(Cursor 把它作为 workspace 规则始终注入;Claude/其他工具按 `CLAUDE.md`/`AGENTS.md` 约定读取)。
  - 内容:中文作答、Mode E 骨架式教学流程或者Mode F、A-B-C-D-E-F 协作模型、代码状态核实规则(禁止用 grep 代替逐文件 Read)、智能体选择指南。
  - 定位:这是"我的开发习惯"的唯一权威来源,**只读、不随进度更新**。
- **状态层 — `personal_assistant/docs/上次会话成果.md`(唯一动态状态文件)**
  - 内容:上次会话日期/次数、当前阶段、路线总览、当前代码状态表、已知问题、**下一步**、dev_log 索引。
  - 定位:回答"现在在哪、什么状态、下一步做什么"。新窗口读它即可秒进状态。
- **细节层 — `personal_assistant/dev_log/YYMMDD_阶段X_*.md`(按时间倒序)**
  - 内容:每次会话的代码全貌、自提疑问与解答、踩坑记录、决策原因。
  - 定位:需要回溯某次具体怎么做的、为什么这么定时,按状态层的 dev_log 索引点进对应文件。

#### 启动加载顺序(新窗口开局必读)

```
新窗口开局
  1. 自动注入 CLAUDE.md          → 知道"怎么和我协作"(习惯)
  2. 读 personal_assistant/README.md        → 项目全貌(首次/陌生时)
  3. 读 docs/上次会话成果.md      → "现在在哪 + 下一步"(状态)
  4. 按需读 docs/阶段X技术方案.md / dev_log/  → 当前阶段细节
  → 进入开发,默认 Mode E(或本次约定的模式)
```

#### 收尾写回(让状态层始终最新)

- 由 **dev-logger 技能** 在"记录一下/收尾/结束会话"时触发:
  1. 在 `dev_log/` 下按 `YYMMDD_阶段X_描述.md` 规范新建当次日志;
  2. **只更新** `docs/上次会话成果.md`(日期+1、阶段、代码状态表、下一步、dev_log 索引);
  3. `CLAUDE.md`、`项目需求文档.md` 等已设为只读,不随日志更新。
- 闭环:**这次的"下一步"= 下次新窗口开局读到的起点**,状态在窗口之间无缝接力。

#### 跨工具一致性(Cursor / Claude / 其他)

- 整套机制是**纯文件驱动、与具体工具无关**:只要新工具能加载 `CLAUDE.md`(或等价的 `AGENTS.md`)并能读 `上次会话成果.md`,就能复用同一份"习惯+状态"。
- 因此用 Claude 按 Mode F 写代码、用 Cursor 讨论方案,看到的是同一套上下文,不会出现"换个工具就失忆"。

#### 一句话总结

> 新窗口 = 自动读 `CLAUDE.md`(习惯) + `上次会话成果.md`(状态),立刻进入状态;会话末尾用 dev-logger 把日志写进 `dev_log/` 并回写 `上次会话成果.md`,完成跨窗口接力。
