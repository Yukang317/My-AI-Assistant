# 开发日志记录器（dev-logger Skill）

> **来源**：从 Claude Code Skill `/root/.claude/skills/dev-logger/SKILL.md` 导出
> **用途**：复制到 Cursor 自建 Skill 使用
> **导出日期**：2026-06-19

---

## 一、何时触发

### 主动触发（用户关键词）
当用户说出以下任一关键词或短语时，立即执行日志记录流程：
- "记录一下" / "记录" / "写日志"
- "总结一下" / "总结"
- "结束会话" / "收尾" / "今天就到这"
- "更新日志" / "更新文档"
- "把今天的工作记下来"

### 自动提醒
当满足以下任一条件时，主动询问用户是否需要记录日志：
- 本次会话中完成了代码编写/修改（创建了新文件或修改了现有文件）
- 本次会话中有技术讨论（解释了新概念、解决了 bug、做了技术决策）
- 用户表示要结束会话但未提"记录"
- 会话内容明显是一次开发工作的结束（如"好了""完成""搞定"）

提醒方式：简短自然，如"老公，需要我把今天的工作记录到开发日志里吗？"

## 二、会话中要追踪什么

在会话进行中，留意并记住以下信息（不需要逐条记录，但要心里有数）：

1. **做了什么**：创建了哪些文件、修改了哪些代码、完成了什么功能
2. **讨论了什么**：用户提出了什么问题、做了解答、讨论了什么概念
3. **踩了什么坑**：遇到了什么 bug、原因是什么、怎么修复的
4. **做了什么决策**：技术选型、架构调整、方案变更
5. **协作模式**：用了 A/B/C/D 哪个模式、效果如何
6. **代码快照**：关键的代码片段（不需要完整，能说明问题即可）

## 三、日志文件编写规范

### 3.1 文件命名

```
格式：YYMMDD_阶段X_简短描述.md
示例：260602_阶段5_Embedding模块.md
     260602_阶段5_MinIO客户端.md
```

- 日期使用 6 位数字（如 260602）
- 阶段号和简短描述用下划线连接
- 同一天多个日志时，在描述中区分

### 3.2 文件位置

`personal_assistant/dev_log/`

### 3.3 日志内容结构

**必须包含以下部分**（按此顺序）：

```markdown
# 阶段 X — 模块名（文件名）

**日期**：YYYY-MM-DD
**模式**：A/B/C/D（说明）
**内容**：一句话描述本次做了什么

---

## 〇、做了什么 & 为什么

简要说明：
- 当前在项目中的位置（前面做了什么，这次做什么）
- 本次完成了什么
- 这个模块在整体架构中的作用

---

## 一、代码全貌（如有编码工作）

贴出本次写的关键代码，附简要注释。原则：
- 不需要逐行拆解（那是给新手看的），但关键逻辑要有注释
- 如果代码超过 60 行，只贴核心部分，省略模板代码
- 每个方法/类附一句话说明其作用

---

## 二、自提疑问 & 解答（如有讨论）

按 Q1、Q2... 编号，每个疑问包含：
- 问题本身（用粗体标出）
- 详细解答
- 如果有代码示例，贴出来

格式：
### Q1：问题标题
**背景**：为什么会有这个问题
**解答**：详细解释
**关键代码**（如有）：...

---

## 三、踩坑记录（如有 Bug）

用表格列出每个 bug：

| # | Bug 现象 | 根因 | 修复 |
|---|---------|------|------|
| 1 | ... | ... | ... |

对复杂 bug 可附详细分析。

---

## 四、协作模式评估（如适用）

对本次使用的协作模式做简短评价：哪里好、哪里需要调整。

---

## 五、文件状态

用表格列出本次涉及的文件及其当前状态：

| 文件 | 状态 | 说明 |
|------|------|------|
| xxx.py | ✅/🔄/⬜ | 简短说明 |

---

## 六、下一步

明确写出下一步要做什么、用什么模式、有什么注意事项。
```

### 3.4 写作风格要求

**核心原则：像学生在记笔记，不是在写文档。**

参照 `personal_assistant/dev_log/日志记录和笔记记录风格.md` 的风格（原始参考：`学习过程与学习过程中的项目/学习过程中的RAG项目实战/开发日志笔记.md`）：

1. **对话式语气**：用"你"而不是"用户"，像是对自己说话
2. **解释为什么**：不只写"做了什么"，更要写"为什么这样做"
3. **问题驱动**：以"遇到的问题"为线索组织内容，而不是以"代码结构"为线索
4. **自提疑问**：把自己当时疑惑的问题记录下来并给出解答，这是最有价值的部分
5. **概念卡片**：遇到新概念时，用简短的"一句话解释"或小表格说明
6. **代码+注释**：代码块里带注释，注释用口语化中文解释"这一行在干什么"
7. **踩坑实录**：每个 bug 记录现象→原因→修复，不要省略中间过程
8. **类比解释**：用生活化的类比解释技术概念（如"就像..."）

**但要注意**：参考日志是极详细模式（2374 行），本项目的日志用**核心讨论+代码+Bug**级别，不需要逐行拆解每一句已知代码。聚焦在：
- 本次新增/修改的代码
- 讨论中涉及的新概念
- 踩到的坑和修复过程
- 值得记录的决策和原因

## 四、项目状态文件更新规则

日志写完后，**只更新 `personal_assistant/上次会话成果.md`**。CLAUDE.md、准备工作总结.md、项目需求文档.md 已改为只读，不再由本技能更新。

### `personal_assistant/上次会话成果.md` — 每次必更新

这是唯一的动态状态文件。每次写日志后都要更新此文件：

```markdown
# 上次会话成果

**上次会话日期**：YYYY-MM-DD（第 N 次会话）
**当前阶段**：阶段 X — 描述（Step X-Y 完成，Step Z 待开始）

---

## 完成了什么

### 本会话：具体工作（日期）

#### Step X：文件名 — 描述（模式）

实现了...类/功能...
- 要点 1
- 要点 2

**本次核心讨论**：

| 问题 | 结论 |
|------|------|
| Q1 | A1 |
| Q2 | A2 |

**代码 Review 发现 N 个 bug**（如有）：列表

---

## 当前代码状态

| 文件 | 状态 | 说明 |
|------|------|------|
| ... | ✅/🔄/⬜ | ... |

---

## 关键讨论记录（新增 dev log）

| 文件 | 内容 |
|------|------|
| dev_log/xxx.md | ... |

---

## 下一步

（更新进度状态）

**下一文件**：...（Step X）— ...
**协作模式**：...
```

更新要点：
- **日期**：改为本次会话日期
- **会话次数**：递增
- **完成了什么**：追加本会话的新内容（保留历史摘要）
- **当前代码状态**：更新文件状态表
- **关键讨论记录**：追加新日志文件名
- **下一步**：更新进度（哪些 Step 从 ⬜→✅，下一个做什么）

## 五、执行流程

当触发日志记录时，按以下顺序执行：

```
1. 回顾本次会话内容
   ├── 做了什么（代码/文件）
   ├── 讨论了什么（概念/问题/决策）
   ├── 踩了什么坑（bug/修复）
   └── 用了什么协作模式

2. 在 dev_log/ 下创建日志文件
   ├── 按 3.1 规范命名
   └── 按 3.3 结构编写

3. 更新 上次会话成果.md

4. 告知用户已完成的更新
```

## 六、注意事项

1. **不要重复记录**：如果同一个工作在日志和状态文件中都有体现，日志写细节，状态文件写摘要
2. **保持一致性**：日期格式统一用 YYYY-MM-DD，文件名用 YYMMDD
3. **及时记录**：最好在会话结束前就写好日志，而不是下次会话开始时补
4. **增量更新**：状态文件是追加式更新，不要覆盖之前的内容（除非是修正错误）
5. **文件状态图标**：✅ 已完成、🔄 进行中、⬜ 待开始、🔴 有问题
6. **如果用户说"不用记录"**：尊重用户选择，不强制记录
