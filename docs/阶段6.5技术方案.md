## 一、目标（一句话）

把 `inspire` 路由从「临时走 `tool_execute` 单工具搜索」升级为 **发散检索 → 收敛合成** 两段式管线。

---

## 二、现状 vs 目标

### 现状（6.4 完成后）

```
intent=inspire → tool_execute（单工具单次/循环）→ result_synthesis → memory_update
```

路由器已能识别 `inspire`，但行为和 `use_tool` 几乎一样——LLM 选一个搜索工具，搜一次，通用 prompt 合成。

### 目标（6.5 Phase 1 垂直切片）

```
intent=inspire → inspire_diverge → inspire_converge → memory_update → END
                      ↑                    ↑
                  发散检索              收敛合成+证据门
              （轻量 LLM + 并行搜索）  （I think... and... 风格）
```

**不走** `tool_execute` 和 **不走** 通用 `result_synthesis`——灵感回复有专属 prompt，质量要求和普通问答不同。

## 三、数据流（一次 inspire 请求的完整路径）

```
用户输入："从心理学角度，我的 RAG 项目有什么启发？"
    │
    ▼
load_context
    ├─ PG 读历史消息 → history_messages
    └─ 读 MEMORY.md → memory_context
    │
    ▼
intent_route
    └─ LLM 四分类 → intent="inspire"
    │
    ▼
inspire_diverge（新节点）
    ├─ ① 轻量 LLM 调用（~50 tokens）
    │     输入：user_question + memory_context
    │     输出：JSON {"queries": ["...", "...", "..."]}  ← 3 条不同角度的查询
    │
    ├─ ② 并行搜索（ThreadPoolExecutor）
    │     每条 query × 2 工具（rag_search + exa_search）= 最多 6 次搜索
    │     每次搜索前过 check_bound（复用 H1 安检逻辑）
    │
    └─ ③ 写入 State
          inspire_queries: ["...", ...]
          inspire_evidence: [ToolResult, ...]   ← 全部搜索结果
    │
    ▼
inspire_converge（新节点）
    ├─ ① 格式化 evidence → 注入专属 System Prompt
    ├─ ② LLM 合成（"I think... and..." 风格，要求跨域连接）
    ├─ ③ 证据门检查（LLM 自检：有无引用支撑？有无跨域连接？）
    │     不通过 → 标记 gate_passed=False，仍输出 draft（Phase 1 不补搜）
    └─ ④ 写入 State
          final_response: "..."
    │
    ▼
memory_update（已有节点，不变）
    └─ 读 final_response + 对话 → 更新 MEMORY.md
```

### 三个存储系统的参与

| 存储 | 在 inspire 管线里的角色 |
|------|------------------------|
| **PostgreSQL** | load_context 读历史；memory_update 写消息；不涉及 inspire 专属表 |
| **Milvus + MinIO** | 仅通过 `rag_search` 工具间接参与（个人文档检索） |
| **外网 API** | 通过 `exa_search` 获取跨领域内容（Exa 神经语义搜索） |

LLM 预训练知识由 **converge prompt** 激活，不经过 RAG。

## 五、各文件改动细节

### 5.1 `agent/state.py` — 新增 2 个字段

在 `StateField` 类中追加：

```python
# 灵感引擎（阶段 6.5）
INSPIRE_QUERIES: str = "inspire_queries"       # 发散阶段生成的查询列表
INSPIRE_EVIDENCE: str = "inspire_evidence"     # 并行搜索收集的 ToolResult 列表
```

在 `MainState` TypedDict 中追加：

```python
# 灵感引擎（阶段 6.5，inspire 路径专用）
inspire_queries: NotRequired[list[str]]
inspire_evidence: NotRequired[list]            # list[ToolResult]，TypedDict 不做泛型
```

> **为什么不用 `tool_results`？** inspire 路径不再走 `tool_execute`，需要专用字段避免和 use_tool 路径混淆。H3 黄金路径测试不受影响。

### 5.5 `agent/main_graph.py` — 接线改动

**import 区追加**：

```python
from agent.inspire_diverge import inspire_diverge
from agent.inspire_converge import inspire_converge
```

**build_graph() 内新增适配节点**：

```python
def inspire_diverge_node(state: MainState) -> dict:
    return inspire_diverge(state, get_llm())

def inspire_converge_node(state: MainState) -> dict:
    return inspire_converge(state, get_llm())
```

**注册节点**：

```python
graph.add_node("inspire_diverge", inspire_diverge_node)
graph.add_node("inspire_converge", inspire_converge_node)
```

**改条件路由**（inspire 不再指向 tool_execute）：

```python
{
    "use_tool": "tool_execute",
    "inspire": "inspire_diverge",          # ← 改这里
    "general_chat": "result_synthesis",
}
```

**新增 inspire 专用边**：

```python
graph.add_edge("inspire_diverge", "inspire_converge")
graph.add_edge("inspire_converge", "memory_update")   # 跳过 result_synthesis
```

> `tool_execute → result_synthesis` 的边保留不动，只影响 use_tool 路径。

**memory_update 适配**：`inspire_converge` 直接写 `final_response`，但 `memory_update` 可能需要 PG 消息保存。检查 `memory_update.py`——如果它只读 `final_response` 更新 MEMORY.md，PG 消息保存需要在 `inspire_converge` 里补 `save_message()`，或让 converge 节点调用已有的保存逻辑。参考 `_ref_inspire_converge.py` 里的处理方式。

## 七、Phase 1 刻意不做的事（留到后续）

| 功能 | 原因 |
|------|------|
| 证据门不通过 → 补搜一轮 | Phase 1 垂直切片先跑通主路径，补搜是 Phase 2 增强 |
| 多 Agent 论坛（方案 C） | Token 成本 3-5×，架构预留接口即可 |
| 对立审视 Agent（Phase 3） | 用户手动触发，独立模块 |
| `inspire` 路径走 `result_synthesis` | 专属 prompt 质量更高，不混用 |
| Tavily 并行搜索 | Phase 1 只用 rag + exa；Tavily 留给 Phase 2 多样性增强 |
