## Q4：路由提示词为什么"太机械"？

### 用户观点

当前 router.py 中的路由规则：

```
- 区分 use_tool 和 inspire 的关键：用户是在**找答案**还是**找思路**？
- 「X 是什么」→ use_tool
- 「X 有什么启发」→ inspire
```

这种方式太机械了。

### Claude 分析（⚠️ 待讨论，未定案）

**这可能是提示词的问题，也可能是架构层面的问题——尚未确定。**

当前 Agent 本质上是**无状态的函数**：
```
f(用户问题) → 意图标签 → 工具调用 → 回复
```
每次对话结束后，Agent 对用户的理解归零。

但"深度对话者"需要的是**有状态的关系**：
```
f(用户问题, 对用户的累积理解) → 识别知识盲区 → 反问 → 理解加深 → 针对性回应
```

**讨论中的观点**：路由提示词的"机械感"可能不是靠改写 prompt 能解决的，可能需要架构升级——从"意图分类器"变成"用户模型驱动的对话决策器"。**此观点尚待验证，方案待定。**

## Q6：核心导航文件 + 分层记忆，这个组合对吗？

### Claude 分析（⚠️ 待讨论/未定案）

**为什么核心导航文件是对的**：

现有的 `MEMORY.md` 已经在用这个模式（`load_context` 节点读取 → 注入 System Prompt），但它是**人工维护的静态文件**——Agent 自己从不更新它。

核心导航文件的本质是把 `MEMORY.md` 从**静态档案**升级为**Agent 自主维护的动态索引**。

这是 Claude Code 和 OpenClaw 都在用的模式：

```
MEMORY.md (入口索引，~500 tokens)
  ├─ "我最近在学 Agent，对 ReAct 循环和 LangGraph 比较熟"
  ├─ "对 Embedding 原理还有疑惑，上次问过但没完全理解"
  ├─ "对前端 JS 不感兴趣，可以跳过"
  ├─ "当前阶段：阶段 6.2 Agent 升级"
  └─ → 详见 knowledge/manifest.md（知识图谱）
      → 详见 sessions/2026-06-13.md（上次深度对话）
```

**为什么不用纯向量数据库做这件事**：

向量检索擅长"语义相似"——你问"ReAct"，它找包含 ReAct 的段落。但它不擅长"这个用户对 ReAct 的理解到哪一层了"。后者是结构化知识——用 Markdown 文件存，LLM 一次性读完，比多次向量查询更准、更快。
