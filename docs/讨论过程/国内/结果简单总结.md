# 个人AI伙伴与助理项目架构方案 v1.0
本文档基于多轮技术研讨、头脑风暴与原型验证记录整理归档，完整覆盖项目定位、核心架构、模块设计、备选方案、已知问题与实施规划。所有研讨过程中的未定案内容、并行方案均已明确标注，已确认结论不做修改，仅补充细节与逻辑梳理。

**重点：以下内容均为叙述方案的讨论过程，下文任何决定性叙述均不成立！**


## 一、项目背景与演进历程
### 1.1 初始阶段：Agent 骨架与原生缺陷
项目阶段 6.1 已完成 LangGraph Agent 基础骨架搭建，包含 12 个核心文件、4 个执行节点、2 条主执行路径，实现了从上下文加载、意图路由到工具执行、结果合成的端到端跑通。但在验证与后续规划中，暴露了两类原生架构缺陷：
其一为意图路由的单维规则局限。测试发现“知识库里有内容吗”这类隐式检索意图的查询，会被基于显式动作词的路由逻辑误判为闲聊，无法正确触发 RAG 检索。本质上是单轮 LLM 同时承担语义理解与动作分发的模式，天然无法覆盖非祈使句的隐含需求，属于规则路由的必然瓶颈。
其二为无状态 Agent 的能力边界。现有 Agent 本质是无状态函数：`f(用户问题) → 意图标签 → 工具调用 → 回复`，每次对话结束后对用户的理解完全归零，无法累积用户认知背景、思维轨迹与偏好，仅能执行单轮任务，无法实现“深度对话者”的产品预期，也直接导致了项目方向的阶段性迷茫。

### 1.2 核心转折点：从个人助理到灵感引擎
经多轮研讨论证，通用个人助理品类本身存在“无具体场景则无真实用户价值”的瓶颈，继续沿该方向迭代将陷入功能堆砌但缺乏核心价值的困境。结合用户对现有 AI 产品的痛点感知，项目正式转向差异化定位——灵感引擎，即“能激活用户思考的对话伙伴”。
该方向精准命中现有 AI 产品的系统性缺陷，同时可复用已有的 RAG 检索、Agent 编排、记忆体系等全部技术基础，将技术能力从“服务执行”重新定向为“思考赋能”，是项目价值的核心转折点。

## 二、核心痛点与产品定位
### 2.1 现存 AI 产品的系统性缺陷
当前主流 AI 产品普遍存在三类不可忽视的体验问题，也是本项目的核心解决目标：
第一是无条件迎合用户。RLHF 对齐训练的导向使得模型天然倾向于服从用户观点，输出同质化的迎合内容，甚至出现脱离场景的尴尬偏好推荐，无法提供批判性、补充性的视角。
第二是启发能力缺失。用户需要发散思路时，检索结果天然偏向与用户观点一致的内容，同质化严重；模型也难以主动激活预训练中的跨领域知识，无法输出符合用户认知口味的启发性内容。
第三是深度交互受限。打字交互效率较低，难以承接临时性的突发想法；现有语音交互方案大多适配浅度问答，无法匹配深度思考类对话的需求。

### 2.2 产品愿景与核心定位
本产品定位为**能激活用户思考的对话伙伴**，核心运行逻辑为：接收用户输入后，自动关联同领域与跨领域的知识、案例与理论，输出用户未知但具备参考价值的内容，以“I think... and...”的叠加延伸模式拓展用户思路，全程不否定、不说教，仅做思路的补充与延伸。
与传统个人助理产品存在本质差异：目标从“完成用户任务”转向“深化用户思考”，交互姿态从服务顺从转向追问与补充，记忆体系从记录用户偏好转向追踪用户思维变化轨迹，检索逻辑从关键词匹配转向挖掘对立案例、边界场景与跨域关联，核心评估指标从任务完成率转向思考启发性。

## 三、整体技术架构体系
### 3.1 三层工程框架
项目整体遵循三层工程框架，明确各层职责边界，避免逻辑混杂：
- **Prompt Engineering 层**：聚焦单次对话的输出质量，负责人格设定、风格控制、结构化输出引导等单次交互的效果优化，是单轮回复质量的基础保障。
- **Context Engineering 层**：管控模型可获取的全部上下文信息，涵盖分层记忆系统、个人知识注入、跨域内容检索等内容供给环节，决定了模型“能看到什么”。
- **Harness Engineering 层**：负责全流程编排与质量约束，包含多段式管线调度、并行检索管理、对立审视模块、证据门校验等流程管控与质量保障机制，决定了系统的资源效率与输出底线。

### 3.2 MVP 核心架构流
MVP 阶段聚焦“懂用户的深度对话”核心能力，暂不接入外部检索工具，优先闭环验证记忆、动态思考策略与风格合成的核心体验。核心执行链路包含 4 个关键节点，全链路基于 LangGraph 状态机流转：
1. **理解层节点**：读取动态记忆索引，对用户输入做多维度解析，输出包含情绪、意图权重分布、知识盲区、工具需求判定的结构化标签，不做非黑即白的路径分支。
2. **记忆更新检查节点**：异步判断本轮对话是否存在值得记录的认知状态变更，提取增量信息生成结构化补丁，经用户确认后更新对应记忆层级。
3. **动态 CoT 生成器节点**：基于用户输入、理解层标签与用户记忆，生成定制化的思考策略指令，作为强制约束注入后续合成环节。
4. **对话合成层节点**：结合用户输入、动态思考指令、行为约束规则与记忆上下文，调用大模型生成符合产品定位的最终回复。

### 3.3 核心设计原则
- **统一理解原则**：废除多人格割裂架构，全系统共用同一套用户理解体系，根据场景标签动态混合回复风格，天然支持“闲聊带任务”等复合意图场景。
- **分层存储原则**：规避“全量存入向量库=全找不到”的陷阱，构建结构化数据与非结构化内容结合的分层存储体系，兼顾精确查询与语义溯源能力。
- **编排优化原则**：通过 Harness 层调度计算资源，轻量模型完成预处理、分发与抽取工作，重模型仅用于最终内容合成，平衡输出效果与 Token 成本。

## 四、核心模块详细设计
### 4.1 统一理解层
替代传统单维意图路由，从“分类器”升级为“用户模型驱动的对话决策器”。该层输出多维度结构化标签，涵盖情绪维度、多意图权重分布、用户知识盲区识别、工具需求判定四大类，不再执行硬性的路径分支，而是将标签传递至合成层，由合成层根据标签动态调整回复的风格、深度与工具调用逻辑。
该设计彻底解决了隐式意图、复合意图的识别难题，也为后续个性化回复提供了统一的判断依据。

### 4.2 分层记忆系统
采用“核心索引 + 三层存储”的组合架构，替代原有的静态 MEMORY.md 方案，兼顾读取效率与信息完整性：
- **Layer 1 结构化数据库**：存储用户知识图谱、技能熟练度、认知状态机、关系网络等结构化信息，提供精确的边界查询能力，支撑认知深度判断。
- **Layer 2 动态索引文件**：即升级后的 MEMORY.md，作为 Agent 每次交互的必读入口，控制在 500 Token 量级，记录用户当前认知焦点、已尝试方案、核心行为模式、待探索方向等核心信息，由 Agent 自主维护更新。
- **Layer 3 向量知识库**：存储原始对话记录、长文笔记、个人文档等非结构化原始数据，仅用于语义溯源与细节召回，不承担用户认知判断的职责。
记忆更新采用半自动机制：每轮对话结束后，由模型提取本轮增量信息，生成结构化补丁，经用户确认后写入对应存储层，避免错误信息污染记忆体系。同时记忆内容从规则约束转向语义映射，以“状态机+行为模式”的形式描述用户特征，替代生硬的禁止类规则，提升大模型的理解准确度。

### 4.3 动态 CoT 生成器
突破静态思考提示词的局限，以独立节点实现思考策略的动态生成，无需人工手写固定的 CoT 提示词：
- 输入为用户输入、理解层标签、记忆摘要，后续可扩展接入检索结果摘要。
- 执行逻辑为调用轻量大模型，仅负责生成思考步骤指南，不直接回答用户问题；思考指南可根据场景定制规则，比如必须关联跨领域原理、必须识别隐含前提、必须指定回复语气等。
- 输出为结构化的思考步骤指令，作为系统提示的一部分强制注入最终合成模型，约束模型遵循指定思考路径输出。
该设计将大模型的参数化知识激活转化为可程序化调度的能力，是实现个性化深度对话的核心机制。

### 4.4 两阶段发散-收敛管线
作为灵感引擎的核心检索编排方案，替代单轮检索架构，平衡启发性与资源成本：
- **阶段一：发散阶段**，通过轻量大模型生成多维度查询列表，并行执行多来源检索，涵盖个人知识库与网络检索；该阶段仅做关键段落抽取，不进行破坏性的内容压缩，保留原始片段用于后续用户溯源。
- **阶段二：收敛阶段**，将抽取的检索结果、用户输入、用户记忆共同注入重模型执行内容合成，要求输出包含至少 1 个跨领域关联、至少 1 项用户未知的参考内容，严格遵循“I think... and...”的表达风格。合成后经过证据门校验，若无有效引用支撑则触发一轮补搜。
该方案 Token 消耗约为单轮方案的 1.5-2 倍，是当前选定的核心落地方案。

### 4.5 可插拔扩展模块
- **对立面审视模块**：核心逻辑不是直接否定用户观点，而是揭露观点背后的隐含假设与前提条件，帮助用户自主完善思考。该模块为可插拔设计，仅在用户主动触发时启用，不默认开启，避免过度干扰正常对话。
- **证据门模块**：输出前的质量校验节点，检查输出内容是否具备检索结果或可信知识支撑，拦截无依据的主观迎合内容，保障输出的客观性与信息密度，是对抗模型迎合倾向的核心防线。

### 4.6 工具与路由系统
现有基础能力包含 rag_search 工具与三分类基础意图路由（general_chat/use_tool/unclear），存在隐式意图识别不足的已知问题。后续扩展规划为：新增 inspire 意图类型，明确区分“检索答案”与“获取灵感”两类核心需求；工具执行节点从单工具单次调用升级为循环调用模式，支持多轮工具编排与多工具组合调用。

## 五、备选方案与技术对比
以下为研讨过程中提出的并行方案，均未做最终取舍，同步保留作为后续迭代的参考依据。

### 5.1 整体架构方案对比
| 方案 | 架构逻辑 | Token 消耗 | 特点与适用阶段 |
|------|----------|------------|----------------|
| 方案A：轻量增强检索 | 用户输入→意图分析→多样查询生成→并行检索→单次 LLM 合成 | ~1× | 实现简单，创造力受单轮生成限制，适用于早期快速验证 |
| 方案B：两段式发散-收敛 | 发散阶段并行多维度检索，收敛阶段重模型合成+证据门校验 | ~1.5-2× | 平衡效果与成本，为当前选定的落地方案 |
| 方案C：多 Agent 论坛 | 视角分解后并行多个独立透镜 Agent 检索分析，最终统一合成 | ~3-5× | 多视角深度强，成本与复杂度高，为远期规划方向 |

### 5.2 记忆系统备选方案
研讨过程中共提出四类记忆落地路径，当前选定分层记忆+核心索引的组合方案，其余方案保留参考：
1. 单文件用户画像方案：以一份持续更新的画像文件存储全部用户认知，实现简单但扩展性弱。
2. 即时推断方案：仅在对话中实时推断用户状态，不做持久化存储，无历史认知累积能力。
3. 纯向量数据库方案：全量内容存入向量库，完全依赖语义检索，结构化认知能力弱，无法判断用户认知深度。
4. 分层记忆系统方案：短期会话记忆+中期学习轨迹+长期知识图谱，搭配核心索引文件导航，为当前选定方案。

### 5.3 意图路由优化备选方向
针对隐式检索意图误判问题，共提出四类优化方向，均处于待评估状态：
1. Prompt 优化：在路由提示词中补充隐式检索意图的规则与示例，强化存在性询问类句式的识别。
2. 关键词预筛选：在大模型判断前，通过“知识库”“资料”“文档”等关键词做前置过滤。
3. 二阶段检测：先判断内容是否与知识查询相关，再判断是否需要工具调用。
4. 案例积累：收集更多误判案例作为少样本示例，优化路由识别效果。

## 六、已知问题与待决策项
### 6.1 已确认的技术问题
- 意图检测误判问题：“知识库有X吗”类隐式检索查询会被误判为闲聊，影响 RAG 检索准确率，严重程度中等。当前通过使用更明确的提问方式临时绕过，待正式优化路由逻辑。
- 安全约束层未启用：当前已实现 BOUND 约束框架，包含危险区域、禁止操作、铁律三类规则，但因仅存在内部 RAG 工具无高危操作，暂无实际调用；后续接入网络请求、文件操作类工具时，需在工具执行前启用校验。

### 6.2 待决策与待补充事项
1. 分层记忆系统的落地阶段选择：可选三类路径，分别为最小可用版（升级记忆文件+会话总结，少量文件改动）、完整设计全套后分阶段实现、从 ReAct 循环开始升级核心执行模式，尚未最终确定。
2. 多 Agent 论坛架构的落地节奏：远期多视角能力的切入节点与资源投入规划待明确。
3. 语音交互能力的接入时机：语音输入输出能力的开发优先级待评估。
4. 垂直场景方向与额外创意想法暂未补充完善，作为后续方向迭代的灵感来源，在每次阶段规划时同步回顾。

## 七、实施路线与优先级
按照依赖关系与价值优先级，项目整体实施顺序如下：

| 优先级 | 任务项 | 核心价值 |
|--------|--------|----------|
| P0 | 实现 web_search 工具 | 灵感引擎跨领域内容检索的核心依赖 |
| P1 | 扩展意图路由，新增 inspire 意图类型 | 区分检索与灵感两类核心需求，支撑检索管线落地 |
| P2 | 实现发散阶段检索模块 | 完成多维度查询生成与并行检索编排 |
| P3 | 实现收敛阶段合成模块 | 完成跨域关联、风格控制与证据门校验 |
| P4 | 实现对立面审视模块 | 落地可插拔的批判性思考增强能力 |
| P5 | 落地记忆系统自动维护机制 | 实现记忆的半自动更新与动态索引维护 |
| P6 | 工具执行节点升级为循环调用模式 | 支持多轮工具调用，完善 Agent 核心执行能力 |

MVP 验证阶段建议优先落地理解层、动态 CoT 生成器、基础记忆体系的最小闭环，暂不接入外部工具，验证“懂用户”的核心体验达标后，再逐步扩展检索与工具能力。

## 八、实施原则与建议
第一，原型优先。MVP 阶段避免过度依赖重型框架，可先用简易脚本验证核心逻辑，确认体验达标后再接入完整的 LangGraph 状态机，避免胶水代码占用核心研发精力。
第二，明确北极星指标。MVP 的核心成功标准是用户感知到“系统真正理解自身当前困境与认知状态”，而非功能完备度。若核心体验不达标，优先迭代记忆体系与思考策略生成逻辑，而非新增功能。
第三，强化反对齐约束。针对大模型天然的迎合倾向，需在合成层加入强约束的负向提示，明确禁止无依据赞同、共情套话等迎合性行为，保障产品定位落地。
第四，结构化推进开发。可基于本架构文档先生成代码结构与函数签名，确认边界后再逐模块填充实现，降低架构走形的风险。
