## 一、核心痛点（为什么做）

用户 Yukang317 使用 AI 产品时发现了三个系统性缺陷：

### 1. AI 无条件迎合用户

- 现有的 AI（豆包等）只会顺着用户说话，翻旧账，给出尴尬的建议
- 这不是个别产品的问题——RLHF 对齐训练天然让模型学会了服从
- 后果：AI 输出"议论文式层层递进看似不可辩驳"，但**不分青红皂白迎合用户**

### 2. AI 不会给灵感启发

- 用户想要发散时，AI 找不到合适或符合口味的资料和案例
- 搜索结果是同质化的——和你想法一样的内容排名最高（RAG 天然偏向）
- AI 不会"激活"自身预训练中已知的跨领域知识

### 3. 无法随时随地深聊

- 打字效率低，语音只有豆包支持但豆包太傻
- 临时性想法无法立即展开深入讨论

## 二、产品愿景：灵感引擎

产品定位不是"聊天机器人"，而是**能激活你思考的对话伙伴**。核心公式：

```
用户说了什么
  → AI 把它跟相关领域和不同领域的东西联系起来
  → 展示你不知道但应该知道的理论/研究/实践
  → 用 "I think... and..." 的方式扩展你的思路（叠加延伸，而非否定或说教）
```

这是从"苏格拉底式挑战者"→ 被用户纠正为"I think... and..."后的最终形态。

## 三、三层工程框架（项目的学习与实践目标）

| 层 | 定位 | 这个项目里具体任务 |
|----|------|-------------------|
| **Prompt Engineering** | 单次对话的质量 | 人格设定、风格控制、JSON 输出引导 |
| **Context Engineering** | 模型能看到什么 | 记忆系统、知识注入、跨域内容检索 |
| **Harness Engineering** | 编排与约束 | 多段式管线、并行检索、对立审视、证据门 |

### 关键的认知纠正

**RAG ≠ 全人类知识库。** RAG 知识库存用户个人文档。跨领域知识通过三个渠道获取：

1. **激活大模型预训练参数**：通过 prompt 设计让模型"回忆"已学知识（心理学、哲学、科学等）
2. **联网搜索**：获取时事、最新研究
3. **特定外部资料库**：学术数据库、专业领域资源

## 四、参考模式与研究成果

讨论中深入研究了以下学术和工程成果：

| 来源 | 核心模式 | 借鉴价值 |
|------|---------|---------|
| **PERSPECTRA** (CHI 2025) | 论坛式多 Agent 审议，@-mention，thread branching，思维导图 | 多视角并行分析，每个"透镜 Agent"独立搜索+分析 |
| **Nova** | 迭代式、有目的的外部知识检索 | 3.4× 独特想法数量，2.5× 最高评分想法 |
| **BioSpark** (CHI 2025) | Tree-of-life 算法跨领域灵感搜索，映射机制，trade-off 分析 | 跨域关联的算法化 |
| **YES AND** (CHI 2025) | 即兴创作框架，acceptance-and-build，基于信心的动态发言权 | "I think... and..." 风格的形式化 |
| **Adversarial Trinity** | Generator-Auditor-Optimizer 三阶段 | 多遍生成 + 审计 + 优化 |
| **Harness 8 层架构** | Input/Context/Orchestration/Reasoning/Policy/State/Action/Observability | 整体框架 |
| **证据门 (Evidence Gate)** | 输出前检查：是否有引用支撑？是否有跨领域连接？ | 质量保障 |

## 五、架构方案对比

### 方案 A：轻量增强检索（Context-Engineering 为主）

```
用户输入 → 意图分析 → 多样查询生成 → 并行搜索(RAG+web) → 单次 LLM 合成
```

- Token 消耗：~1×
- 缺点：位置偏差（LLM 只抓前几段），创造力受限于单次生成

### 方案 B：两段式发散-收敛 ✅ 选定

```
Phase 1 — 发散
  用户输入 → LLM 轻量调用（输出 JSON 查询列表，~50 tokens）
  → 并行执行全部搜索（RAG + Web，每查询 2 来源）
  → 收集所有结果

Phase 2 — 收敛
  搜索结果 + 用户输入 → LLM 合成
  → 要求：≥1 跨领域连接 + ≥1 "不知道但该知道"的内容
  → "I think... and..." 风格展开
  → 证据门检查 → 通过输出 / 不通过补搜一轮
```

- Token 消耗：~1.5-2×
- Harness 负责：查询生成 + 并行搜索 + 结果聚合 + 证据门
- 核心优势：Phase1 轻量调用，Phase2 才是大 token 合成

### 方案 C：多 Agent 论坛（未来方向）

```
用户输入 → 视角分解器 → 并行 3-5 个"透镜 Agent"（各自独立搜索+分析）
  Agent 1（同域深入）Agent 2（跨域类比）Agent 3（批判/边界）...
→ 合成器跨 Agent 找关联→ 质量门
```

- Token 消耗：~3-5×
- 来源：PERSPECTRA 多角色并行模式
- 当前阶段成本过高

**选择：方案 B，架构预留升级到 C 的接口。**

## 六、对立面审视 Agent（Phase 3，可插拔）

两个相关但不同的需求：

### 需求 A："对立面"审视

不是让 AI 否定用户，而是揭露**隐含假设**。用户说的方案 → Agent 指出用户没意识到的前提条件 → 用户自己判断。触发方式：用户说"审一下"/"你怎么看"时才触发，不自动运行。

### 需求 B："客观事实"保障

AI 不能不分青红皂白迎合，必须基于实际情况给出客观建议。通过证据门实现——输出要求有检索结果引用支撑，不允许纯捏造。

两者都是可选、可插拔的模块，通过 feature flag / 用户命令触发。
