### 为什么不能合并？

| 需求 | 哪个系统负责 | 为什么不能用别的 |
|------|------------|----------------|
| "找和这个意思相似的文本" | **Milvus** | PostgreSQL 的 LIKE 做不到语义搜索；MinIO 没有搜索能力 |
| "这个文档上传时间、属于哪个分类" | **PostgreSQL** | Milvus 不懂外键和事务；MinIO 只能按对象名找 |
| "打开原始 PDF 文件" | **MinIO** | PostgreSQL 和 Milvus 不存原始文件 |
| "删除一个文档及其所有相关数据" | **PostgreSQL 级联删除 + Milvus 按 vector_id 清理 + MinIO 删除对象** | 三个系统协同工作 |

## 五、核心模块详细设计

### 5.1 文档解析器 — `rag/document_parser.py`

**借鉴来源：MilDoc 的 `simple_object_parser.py` + `document_parser.py`**

和 MilDoc 一致使用策略模式，但做三个调整：
1. 解析器减为 4 种（PDF/DOCX/MD/TXT）
2. 保留 `_get_parser()` 自动匹配逻辑
3. 输入为 bytes，从 MinIO 读取后直接解析

```python
class DocumentParser(ABC):          # 抽象基类（同 MilDoc）
    @abstractmethod
    def parse(self, data: bytes) -> str: ...
    @abstractmethod
    def supports(self, file_type: str) -> bool: ...

class PDFParser(DocumentParser):     # PyPDF2
class DocxParser(DocumentParser):    # python-docx
class MarkdownParser(DocumentParser): # 纯文本读取
class TextParser(DocumentParser):    # 兜底解析器

class DocumentParserCoordinator:     # 协调器（同 MilDoc 的 SimpleObjectParser）
    def __init__(self):
        self.parsers = [PDFParser(), DocxParser(), MarkdownParser(), TextParser()]

    def _get_parser(self, file_type: str):
        """遍历解析器列表，返回第一个 supports() 返回 True 的"""
        for parser in self.parsers:
            if parser.supports(file_type):
                return parser
        return TextParser()  # 兜底
```

### 5.2 文档切分器 — `rag/chunker.py`

**借鉴来源**：
- RAG项目实战的 `DocumentProcessor.create_parent_child_chunks()`（父子块设计）
- LlamaIndex RAG 的 `MarkdownNodeParser` + `SentenceSplitter`（语义分块）🆕

**策略**：融合两种分块方式

```
原始文档
    │
    ▼
第1步：语义分块（按 Markdown 标题层级 + 段落边界切分）
    ├─ MarkdownNodeParser：识别 H1-H6 标题层级，保留结构关系
    ├─ SentenceSplitter(chunk_size=512, overlap=50, separator="\n\n")
    └─ 输出：语义完整的段落块（不会在句子中间切断）
    │
    ▼
第2步：父子块构建
    ├─ parent_splitter(2048/128) → 父块（大块，保留完整上下文）
    └─ child_splitter(512/64)  → 子块（小块，精准语义匹配）

检索时：子块做语义匹配（精度高）→ parent_id 回溯父块（上下文完整）
```

**与纯固定大小分块的区别**：
- 固定大小：按字符数切，"在句子中间"切断 → 语义碎片化
- 语义分块：按标题+段落边界切 → chunk 自带结构信息 → 检索更精准

### 5.7 RAG 评估 — `rag/evaluate.py` 🆕

**借鉴来源：LlamaIndex RAG 的 `test/evaluate_rag.py`（用户已完成）**

**评估指标**（对标 RAGAS）：

| 指标 | 含义 | 计算方式 |
|------|------|---------|
| **Faithfulness**（忠实度） | 回答是否完全基于检索到的文档，有没有编造 | LLM 逐句判断 → 忠实句数 / 总句数 |
| **Context Recall**（召回率） | 检索到的文档覆盖了多少正确答案 | 标注答案中的信息点命中率 |
| **Context Precision**（精确率） | 检索结果中有多少是真正相关的 | 相关文档数 / 检索文档数 |
| **Answer Relevance**（答案相关性） | 回答是否直接回应了问题 | LLM 判断 |

```python
class RAGEvaluator:
    def __init__(self, rag_service):
        self.rag = rag_service
```

### 5.9 PostgreSQL 数据库设计

**替代原有 SQLite chat.db**，在 PostgreSQL 中新建 `personal_assistant` 数据库：

```sql
-- 文档主表
CREATE TABLE documents (
    id          SERIAL PRIMARY KEY,
    filename    TEXT NOT NULL,
    file_type   TEXT NOT NULL,          -- pdf/docx/md/txt
    object_key  TEXT NOT NULL,          -- MinIO 对象路径（bucket/key）
    file_size   INTEGER,
    file_md5    TEXT,                   -- 文件MD5去重
    chunk_count INTEGER DEFAULT 0,
    created_at  TIMESTAMPTZ DEFAULT now()
);

-- 父块表
CREATE TABLE parent_chunks (
    id          SERIAL PRIMARY KEY,
    document_id INTEGER NOT NULL REFERENCES documents(id) ON DELETE CASCADE,
    parent_id   TEXT NOT NULL UNIQUE,   -- 业务ID: "{doc_name}_parent_{i}"
    content     TEXT NOT NULL,
    vector_id   TEXT                    -- Milvus 中的向量ID
);

-- 子块表
CREATE TABLE child_chunks (
    id              SERIAL PRIMARY KEY,
    document_id     INTEGER NOT NULL REFERENCES documents(id) ON DELETE CASCADE,
    parent_chunk_id INTEGER NOT NULL REFERENCES parent_chunks(id) ON DELETE CASCADE,
    child_id        TEXT NOT NULL UNIQUE, -- 业务ID: "{parent_id}_child_{j}"
    content         TEXT NOT NULL,
    vector_id       TEXT                  -- Milvus 中的向量ID
);

-- 保留原有的 messages 和 sessions 表（从 SQLite 迁移）
CREATE TABLE sessions (...);
CREATE TABLE messages (...);
```

**级联删除链路**：
```
DELETE FROM documents WHERE id=5
  → CASCADE DELETE parent_chunks WHERE document_id=5
    → CASCADE DELETE child_chunks WHERE parent_chunk_id IN (...)
      → 应用层: Milvus.delete(vector_ids=child_chunks.vector_id)
  → 应用层: minio_client.remove(documents.object_key)
```

## 八、防踩坑清单

| # | 坑 | 预防 |
|---|-----|------|
| 1 | 子块分隔符含空字符 → 语义破坏 | `child_splitter` 的 `separators` 不含 `""` |
| 2 | `parent_chunk_map` key用错 → 子块找不到父块 | 用 `parent_id` 做 key |
| 3 | `session.flush()` 忘记 → 自增ID为None | flush拿doc.id → flush拿parent.id → commit |
| 4 | 重排序把最相关文档排没 | MilDoc安全兜底：向量检索第1名强制保留 |
| 5 | Milvus 内存不足 → OOM | 索引后 unload collection，日常不驻留 |
| 6 | BGE模型占用2GB → 索引时OOM | 索引完成后调用 `unload()` 释放 |
| 7 | 🆕 BM25+向量结果不兼容（不同分数字段） | RRF 统一排名融合，不直接比较原始分数 |
| 8 | 🆕 语义分块在纯文本上退化 | 无 Markdown 标题时回退到段落边界分块（`separator="\n\n"`） |
| 9 | 🆕 RRF 融合后 chunk 重复（向量和BM25都命中） | 融合前按 chunk_id 去重 |
| 10 | 🆕 评估数据与生产数据同源 → 评估失真 | 评估用例独立创建，不和上传文档重复 |
