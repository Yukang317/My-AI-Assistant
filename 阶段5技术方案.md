### 为什么不能合并？

| 需求 | 哪个系统负责 | 为什么不能用别的 |
|------|------------|----------------|
| "找和这个意思相似的文本" | **Milvus** | PostgreSQL 的 LIKE 做不到语义搜索；MinIO 没有搜索能力 |
| "这个文档上传时间、属于哪个分类" | **PostgreSQL** | Milvus 不懂外键和事务；MinIO 只能按对象名找 |
| "打开原始 PDF 文件" | **MinIO** | PostgreSQL 和 Milvus 不存原始文件 |
| "删除一个文档及其所有相关数据" | **PostgreSQL 级联删除 + Milvus 按 vector_id 清理 + MinIO 删除对象** | 三个系统协同工作 |

## 五、核心模块详细设计

### 5.1 文档解析器 — `rag/document_parser.py`

**借鉴来源：MilDoc 的 `simple_object_parser.py` + `document_parser.py`**

和 MilDoc 一致使用策略模式，但做三个调整：
1. 解析器减为 4 种（PDF/DOCX/MD/TXT）
2. 保留 `_get_parser()` 自动匹配逻辑
3. 输入为 bytes，从 MinIO 读取后直接解析

```python
class DocumentParser(ABC):          # 抽象基类（同 MilDoc）
    @abstractmethod
    def parse(self, data: bytes) -> str: ...
    @abstractmethod
    def supports(self, file_type: str) -> bool: ...

class PDFParser(DocumentParser):     # PyPDF2
class DocxParser(DocumentParser):    # python-docx
class MarkdownParser(DocumentParser): # 纯文本读取
class TextParser(DocumentParser):    # 兜底解析器

class DocumentParserCoordinator:     # 协调器（同 MilDoc 的 SimpleObjectParser）
    def __init__(self):
        self.parsers = [PDFParser(), DocxParser(), MarkdownParser(), TextParser()]

    def _get_parser(self, file_type: str):
        """遍历解析器列表，返回第一个 supports() 返回 True 的"""
        for parser in self.parsers:
            if parser.supports(file_type):
                return parser
        return TextParser()  # 兜底
```

### 5.2 文档切分器 — `rag/chunker.py`

**借鉴来源**：
- RAG项目实战的 `DocumentProcessor.create_parent_child_chunks()`（父子块设计）
- LlamaIndex RAG 的 `MarkdownNodeParser` + `SentenceSplitter`（语义分块）🆕

**策略**：融合两种分块方式

```
原始文档
    │
    ▼
第1步：语义分块（按 Markdown 标题层级 + 段落边界切分）
    ├─ MarkdownNodeParser：识别 H1-H6 标题层级，保留结构关系
    ├─ SentenceSplitter(chunk_size=512, overlap=50, separator="\n\n")
    └─ 输出：语义完整的段落块（不会在句子中间切断）
    │
    ▼
第2步：父子块构建
    ├─ parent_splitter(2048/128) → 父块（大块，保留完整上下文）
    └─ child_splitter(512/64)  → 子块（小块，精准语义匹配）

检索时：子块做语义匹配（精度高）→ parent_id 回溯父块（上下文完整）
```

**与纯固定大小分块的区别**：
- 固定大小：按字符数切，"在句子中间"切断 → 语义碎片化
- 语义分块：按标题+段落边界切 → chunk 自带结构信息 → 检索更精准

#### 5.4.1 Milvus Collection Schema 设计

**数据库**：`personal_assistant`（通过配置创建/切换）

**Collection 1：`parent_chunks`**（父块，存完整上下文）

| 字段名 | 类型 | 说明 |
|--------|------|------|
| `id` | INT64 | 主键，自增（`auto_id=True`） |
| `doc_name` | VARCHAR(500) | 来源文档名（用于删除时的过滤条件） |
| `doc_path_name` | VARCHAR(1000) | 文档在 MinIO 的完整 object_key（唯一标识，删除/更新的 key） |
| `doc_type` | VARCHAR(50) | 文件类型（pdf/docx/md/txt） |
| `doc_md5` | VARCHAR(32) | 文件 MD5（去重判断） |
| `parent_id` | VARCHAR(200) | 业务 ID：`{doc_name}_parent_{i}`（唯一索引） |
| `content` | VARCHAR(65535) | 父块完整文本（2048 tokens） |
| `content_vector` | FLOAT_VECTOR(1024) | BGE-large-zh-v1.5 向量 |
| `embedding_model` | VARCHAR(100) | 记录生成向量的模型名（模型切换时过滤旧向量！） |

**Collection 2：`child_chunks`**（子块，用于语义检索）

| 字段名 | 类型 | 说明 |
|--------|------|------|
| `id` | INT64 | 主键，自增 |
| `doc_name` | VARCHAR(500) | 来源文档名 |
| `doc_path_name` | VARCHAR(1000) | MinIO object_key |
| `parent_id` | VARCHAR(200) | 关联父块业务 ID（检索后回溯父块） |
| `child_id` | VARCHAR(200) | 业务 ID：`{parent_id}_child_{j}`（唯一索引） |
| `content` | VARCHAR(65535) | 子块文本（512 tokens） |
| `content_vector` | FLOAT_VECTOR(1024) | BGE-large-zh-v1.5 向量（L2 归一化后） |
| `embedding_model` | VARCHAR(100) | 生成向量的模型名 |

> **`embedding_model` 字段的重要性**（来自 mildoc 实践）：
> 将来换 Embedding 模型（如 BGE → M3E → API），新旧向量维度/语义空间不同，混在一起搜索会出问题。
> 有此字段可精确过滤：`expr = f'embedding_model == "bge-large-zh-v1.5"'`，不影响旧数据。

#### 5.4.2 Milvus 索引配置

```python
# 索引类型
INDEX_TYPE = "IVF_FLAT"       # 倒排索引 + 平坦搜索，适合中小规模（<100万向量）

# ⚠️ 关键决策：度量方式必须与 Embedding 的归一化策略匹配
# BGE embedding.py 已做 L2 归一化（normalize_embeddings=True）
# 归一化后向量模长=1，此时 内积(IP) 等价于 余弦相似度(COSINE)，但计算量减半
METRIC_TYPE = "IP"            # 内积（不是 COSINE！因为已做 L2 归一化）

# 索引参数
INDEX_PARAMS = {
    "nlist": 1024,            # 聚类中心数（√n 经验值，n≈100万时用1024）
}
# 搜索参数
SEARCH_PARAMS = {
    "nprobe": 64,             # 搜索时探测的聚类数（nlist 的 6.25%，平衡速度/精度）
}
```

**nlist/nprobe 调优参考**（来自 mildoc 生产经验）：

| 数据量 | nlist | nprobe | 说明 |
|--------|-------|--------|------|
| <1万 | 128 | 16 | 个人文档起步阶段 |
| 1-10万 | 1024 | 32-64 | 日常使用推荐 |
| >10万 | 2048+ | 64-128 | 精度优先时可适当提高 nprobe |

#### 5.4.3 向量存储类设计


#### 5.4.4 增量索引与 MD5 去重

**借鉴来源：mildoc 的 force_update + MD5 检查 + llamaindex 的 IngestionCache**

```
文档上传
  │
  ├── 1. 计算文件 MD5
  ├── 2. 查 PG：SELECT id, file_md5 FROM documents WHERE object_key = ?
  │     ├── 不存在 → 新文档，走完整入库流程
  │     ├── 存在且 MD5 相同 → 跳过（去重），返回 "已索引，无需重复上传"
  │     └── 存在但 MD5 不同 → 文件内容已更新
  │         ├── 删 Milvus 旧向量：delete_by_doc_path(old_object_key)
  │         ├── 删 PG 旧 chunks（CASCADE）
  │         ├── 更新 PG documents 记录（file_md5, file_size, chunk_count）
  │         └── 走正常入库流程（重新解析+分块+向量化）
  └── 3. 返回索引结果（新增/更新/跳过）
```

#### 5.4.5 三种索引模式 🔴 阶段5实现

**借鉴来源：mildoc 的 full-refresh / backfill / listen 三种运行模式**

| 模式 | 触发方式 | 用途 | 阶段5实现 |
|------|---------|------|----------|
| **upload**（上传同步） | POST `/api/documents/upload` 调用 | 日常使用：用户上传一个文件 → 即时索引 | ✅ 默认模式 |
| **backfill**（断点续传） | CLI 命令 `uv run rag/storage/vector_store.py --mode backfill` | 运维：Milvus 数据丢了/挂了，从 MinIO 全量重建（只索引 Milvus 里没有的文件） | ✅ CLI 参数 |
| **full-refresh**（全量重建） | CLI 命令 `uv run rag/storage/vector_store.py --mode full-refresh` | 运维：换 Embedding 模型后，强制重建全部向量 | ✅ CLI 参数 |

```python
# CLI 使用示例
# 断点续传（最常用）：MinIO 有 50 个文件，Milvus 只有 30 个 → 只索引缺失的 20 个
uv run rag/storage/vector_store.py --mode backfill

# 全量重建：换了 Embedding 模型（如 BGE → M3E），删光旧向量重新索引
uv run rag/storage/vector_store.py --mode full-refresh
```

> **backfill 模式的价值**（来自 mildoc 踩坑经验）：Milvus 是独立 Docker 容器，数据可能因磁盘满/升级/误操作丢失。没有 backfill 就得手动重新上传所有文件，有 backfill 一行命令从 MinIO 全量恢复。

## 八、防踩坑清单

| # | 坑 | 预防 |
|---|-----|------|
| 1 | 子块分隔符含空字符 → 语义破坏 | `child_splitter` 的 `separators` 不含 `""` |
| 2 | `parent_chunk_map` key用错 → 子块找不到父块 | 用 `parent_id` 做 key |
| 3 | `session.flush()` 忘记 → 自增ID为None | flush拿doc.id → flush拿parent.id → commit |
| 4 | 重排序把最相关文档排没 | MilDoc安全兜底：向量检索第1名强制保留 |
| 5 | Milvus 内存不足 → OOM | 索引后 unload collection，日常不驻留 |
| 6 | BGE模型占用2GB → 索引时OOM | 索引完成后调用 `unload()` 释放 |
| 7 | 🆕 BM25+向量结果不兼容（不同分数字段） | RRF 统一排名融合，不直接比较原始分数 |
| 8 | 🆕 语义分块在纯文本上退化 | 无 Markdown 标题时降级到 `RecursiveCharacterTextSplitter`（`separators=["\n\n","\n","。","."]`） |
| 9 | 🆕 RRF 融合后 chunk 重复（向量和BM25都命中） | 融合前按 chunk_id 去重 |
| 10 | 🆕 评估数据与生产数据同源 → 评估失真 | 评估用例独立创建，不和上传文档重复 |
| 11 | 🆕 Milvus 度量方式用错（COSINE vs IP） | BGE 已做 L2 归一化（`normalize_embeddings=True`），Milvus 必须用 `IP`（内积），不用 `COSINE` |
| 12 | 🆕 Milvus 空路径删除 → 误删全库 | `delete_by_doc_path("")` → 拒绝空字符串（mildoc 的 `_validate_path` 安全检查） |
| 13 | 🆕 换 Embedding 模型后新旧向量混搜 | `embedding_model` 字段标记每个向量 → 搜索时过滤 `expr='embedding_model=="bge-large-zh-v1.5"'` |
| 14 | 🆕 增量索引时 PG 写入成功但 Milvus 写入失败 → 数据不一致 | 先写 PG（flush）→ 写 Milvus → Milvus flush 成功 → PG commit；任一步失败回滚 |
| 15 | 🆕 文档重命名后旧 chunk ID 对不上 | 用 `object_key`（MinIO 路径）做稳定标识，不用文件名 |

> → 向量找到语义相关的笔记段落 + BM25 精确命中 `deepseek-chat` 模型名

#### 11.2.2 RecursiveCharacterTextSplitter 降级分块器 — `rag/chunker/recursive_splitter.py`（阶段 6）

**来源**：mildoc `simple_object_parser.py` 的 `RecursiveCharacterTextSplitter(chunk_size=2048, chunk_overlap=128)`

当前方案主打语义分块（`MarkdownNodeParser`），但纯文本文件（TXT、代码文件）没有 Markdown 标题结构。`RecursiveCharacterTextSplitter` 按优先级递归切分：

```
separators = ["\n\n", "\n", "。", ".", " ", ""]
            ↑ 段落  ↑ 换行 ↑ 中文句号 ↑ 英文句号 ↑ 空格 ↑ 字符
              优先级高 ←······························→ 优先级低
```

**与语义分块的关系**：互补而非替代。`semantic_splitter.py` 处理有结构文档（MD/PDF），`recursive_splitter.py` 处理无结构纯文本。

#### 11.2.3 SSE 事件类型设计（阶段 6）

**来源**：llamaindex-RAG-repo `core/application.py` 的 `query_documents_stream()`

当前 `step22_session.py` 的 SSE 只推 `text` delta。llamaindex 项目定义了多种事件类型：

```python
# 事件类型（丰富 SSE 推送内容）
async def query_documents_stream(query):
    # 1. sources 事件：来源文档（先于文本推送！）
    yield {"type": "sources", "documents": [
        {"file_name": "xxx.md", "score": 0.92, "preview": "前200字..."}
    ]}

    # 2. delta 事件：文本逐 chunk（当前已有）
    async for chunk in llm_stream:
        yield {"type": "delta", "content": chunk}

    # 3. complete 事件：结束信号 + token 统计
    yield {"type": "complete", "token_usage": {...}}

    # 4. error 事件：错误信息
    yield {"type": "error", "content": "检索失败，已降级为纯LLM模式", "done": True}
```

**核心改进**："来源先返回"——用户在等待 LLM 打字时，已经在看参考来源了，体验更好。

#### 11.2.5 Reranker 提供商抽象 — `rag/retrieval/reranker.py`（阶段 6）

**来源**：mildoc `rerank_service.py` 的双提供商设计

当前方案只计划本地 `bge-reranker-large`（~1.3GB），参考 `embedding.py` 的双模式设计，reranker 也应支持：

```python
class RerankerService:
    """重排序服务 — local/API 双模式"""
    def __init__(self, mode="local"):
        if mode == "local":
            self.model = SentenceTransformer("BAAI/bge-reranker-large")
        elif mode == "api":
            self.provider = "dashscope"  # gte-rerank-v2
            # 或 "siliconflow"          # BAAI/bge-reranker-v2-m3

    def rerank(self, query: str, documents: List[str], top_n=5) -> List[Dict]:
        """返回 [{"content": ..., "score": 0.92}, ...]"""

    # 安全兜底（来自 mildoc）：向量检索第1名强制保留
    def _safety_retain_top1(self, original_top1, reranked_docs):
        if original_top1 not in reranked_docs:
            reranked_docs.insert(0, original_top1)
        return reranked_docs
```
