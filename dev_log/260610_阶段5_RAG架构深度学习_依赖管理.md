## 一、RAG 两大核心模块的职责分离

### 1.1 模块对照表

| 模块 | 类名 | 职责 | 触发时机 |
|------|------|------|---------|
| **文档摄入** | `DocumentIndexer` | 解析 → 分块 → 向量化 → 存入 Milvus/MinIO | 用户上传文档时（离线） |
| **知识检索** | `RagService` | BM25 + 向量检索 → RRF 融合 → LLM 生成答案 | 用户提问时（在线） |

**关键原则**：两者**独立存在**，不互相包含。共享底层组件（`MilvusVectorStore`、`EmbeddingService`）。数据流向：`DocumentIndexer` 写入 → Milvus → `RagService` 读取。

## 二、懒加载（Lazy Loading）模式

### 2.1 模板代码

```python
_rag_service: RagService | None = None
_indexer: DocumentIndexer | None = None

def get_rag_service() -> RagService:
    global _rag_service
    if _rag_service is not None:  # 已初始化，直接返回
        return _rag_service

    # 首次调用：执行耗时的初始化逻辑
    _rag_service = RagService(...)
    return _rag_service
```

### 2.2 为什么要懒加载？

- `RagService` 初始化耗时 ~1-2 秒（构建 BM25、连接 Milvus、加载模型）
- `DocumentIndexer` 初始化也需要连接多个服务
- **整个应用生命周期只初始化一次**，后续请求直接复用
- 如果用户不用 RAG 功能，不会浪费资源

### 2.3 懒加载生命周期图

```
应用启动
    ↓
_rag_service = None
_indexer = None
    ↓
用户第一次发起 RAG 请求
    ↓
get_rag_service() 被调用
    ↓
检查 _rag_service is None? → 是 → 执行初始化（~1-2 秒）
    ↓
_rag_service = RagService(...)  ← 缓存到全局变量
    ↓
返回 _rag_service → 执行检索，返回答案
    ↓
─────────────────────────────────────
用户第二次发起 RAG 请求
    ↓
get_rag_service() 被调用 → _rag_service is None? → 否
    ↓
直接返回 _rag_service（~0ms）→ 执行检索（快！）
```

## 三、BM25 vs 向量检索的本质区别

### 3.1 对比表

| 特性 | BM25Index | VectorRetriever |
|------|-----------|-----------------|
| **索引位置** | 内存中（`self._bm25_model`） | Milvus 数据库（磁盘持久化） |
| **是否需要预建** | ✅ 必须手动 `build_index()` | ❌ 不需要（Milvus 内部管理） |
| **重启后** | 索引丢失，需重建 | 数据依然存在 |
| **搜索原理** | 倒排索引 + IDF 统计 | 向量相似度（Cosine/IP） |
| **类比** | 图书馆卡片目录（需预先整理） | 快递员（去仓库取货） |

### 3.2 为什么 VectorRetriever 不需要构建逻辑？

- Milvus 是专业向量数据库，内部自动管理 HNSW/IVF 索引
- `VectorRetriever` 只是客户端，调用 `vs.search()` 即可
- BM25 是轻量级库（`rank_bm25`），没有内置持久化，必须自己维护索引

### 3.3 数据流对比

```
【BM25（内存索引）】
DocumentIndexer 写入 Milvus → 应用重启 → get_rag_service()
→ 从 Milvus 读取所有 chunks → bm25.build_index()（内存中构建）
→ 可以搜索 → 应用再次重启 → 索引丢失 ❌ → 需重建

【向量检索（Milvus 持久化）】
DocumentIndexer 写入 Milvus → Milvus 内部：向量写磁盘 + 构建 HNSW 索引
→ 应用重启 → VectorRetriever 重新连接 → 可以直接搜索 ✅
→ 应用再次重启 → 依然可以搜索 ✅（数据还在 Milvus）
```

## 四、从 Milvus 反查构建 BM25 的设计考量

```python
all_chunks = vs.client.query(
    collection_name=Config.MILVUS_CHILD_COLLECTION,
    filter=f'embedding_model == "{Config.get_embedding_model_name()}"',
    output_fields=["child_id", "content"],
    limit=2000,
)
bm25.build_index(
    [c["content"] for c in all_chunks],
    [c["child_id"] for c in all_chunks],
)
```

**设计初衷**：
1. **单一数据源原则**：Milvus 是"唯一真相源"，避免数据不一致
2. **简化架构**：不需要额外的持久化层（如 Redis、磁盘文件）
3. **支持动态更新**：重启时自动从 Milvus 重建，无需手动同步

**是否标准做法？**
- ❌ 不是最常见的（生产环境通常 BM25 独立持久化）
- ✅ 但对学习项目是合理权衡（数据量小、一致性优先、实现简单）

**性能影响**：
- 首次 RAG 请求延迟 ~1-2 秒（Milvus 查询 + jieba 分词 + BM25 构建）
- 后续请求无额外开销（索引缓存在内存）

### 为什么用子块（child_chunks）而非父块？

- **子块**：细粒度（~512 tokens），适合 BM25 关键词精准匹配
- **父块**：粗粒度（~2048 tokens），用于 LLM 生成时的完整上下文

## 五、知识检索全流程（RagService.query）

```
用户提问："DeepSeek API 怎么调用？"
    ↓
第1步：双通道检索
  ├─ BM25Index.search() → 关键词检索 → top-10 chunks
  └─ VectorRetriever.search() → 语义检索 → top-10 chunks
    ↓
第2步：RRF 融合排序
  RRFFusion.fuse(bm25_results, vector_results) → top-10
    ↓
第3步：重排序（可选，Config.USE_RERANK=True）
  bge-reranker 重新打分 → top-5
    ↓
第4步：构建 Prompt
  _build_context() → 拼接参考文档
  _build_prompt() → 填充模板
    ↓
第5步：LLM 生成答案
  llm_client.chat.completions.create() → DeepSeek 生成回答
    ↓
返回 {answer, sources, token_usage, latency_ms}
```

## 六、文档摄入全流程（DocumentIndexer.index_document）

```
用户上传文档
    ↓
第1阶段：准备工作
  1. 提取文件类型 (.md/.pdf/.txt)
  2. 计算 MD5（内容去重）
  3. 构建 object_key (documents/2026/06/..)
  4. MD5 去重检查：新文件→继续 / 内容相同→跳过 / 内容更新→删旧数据
    ↓
第2阶段：分块 + 向量化
  1. Parser.parse() → 纯文本
  2. split_by_type() → 语义块列表
  3. build_parent_child() → 父子块
  4. EmbeddingService.embed() → 向量
    ↓
第3阶段：双存储写入
  1. MinIO.upload() → 存原始文件
  2. Milvus.insert_parent_child() → 存向量
  3. Milvus.flush() → 刷盘
    ↓
返回 {status, object_key, parent_count, child_count}
```

## 七、已识别 Bug

### Bug 1：`update_bm25_after_uploads()` 调用不存在的方法

```python
# ❌ 错误：BM25Index 没有 update_index() 方法
_rag_service.bm25_index.update_index(chunks_text, chunk_ids)

# ✅ 正确应该是 add_chunks()
_rag_service.bm25_index.add_chunks(chunks_text, chunk_ids)
```

**更根本的问题**：外部拿不到 `chunks_text` 和 `chunk_ids`，因为 `DocumentIndexer.index_document()` 没有返回这些信息。

**解决方案**：
- 短期：删除该函数，采用"重启时全量重建"策略
- 长期：在上传接口中从 Milvus 读取新 chunk，调用 `add_chunks()`

## 八、依赖管理

### 8.1 pyproject.toml 缺失的包

初始 `pyproject.toml` 缺少以下依赖，导致 `uv run step23_rag.py` 报 `ModuleNotFoundError`：

| 缺失的包 | 用在哪里 | 安装方式 |
|----------|---------|---------|
| `pymilvus` | Milvus 向量数据库客户端 | `uv add pymilvus` |
| `rank-bm25` | `rag/retrieval/bm25.py` BM25 关键词检索 | 手动加到 pyproject.toml |
| `sentence-transformers` | `rag/embedding.py` BGE 向量化 | 手动加到 pyproject.toml |

### 8.2 解决过程

1. `uv add pymilvus --index-url https://pypi.tuna.tsinghua.edu.cn/simple` → 装上了但 index 不持久化到 pyproject.toml
2. 用户手动在 pyproject.toml 的 `dependencies` 列表中添加 `rank-bm25` 和 `sentence-transformers`
3. `uv sync` 一次性解析并安装 **138 个包**，包括 `torch`、`transformers`、`scikit-learn`、`pymilvus` 等大型依赖

### 8.3 `uv add --index-url` 的坑

`--index-url` 是临时参数，不会写入 pyproject.toml。下次 `uv sync` 在其他机器上不会使用清华源。正确做法是用 `--default-index` 或手动在 pyproject.toml 中配置 `[[tool.uv.index]]`。
