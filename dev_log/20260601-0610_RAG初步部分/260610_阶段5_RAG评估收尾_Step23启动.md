## 三、核心知识点

### Q1：为什么 BM25 更新需要"全量重建"？

**背景**：用户上传新文档后，调用 `bm25.add_chunks(new_texts, new_ids)`。

**解答**：BM25 的核心公式是：
```
BM25(d, q) = Σ IDF(qi) × TF(qi, d) × ...
```

其中 **IDF（逆文档频率）= log((N - n(qi) + 0.5) / (n(qi) + 0.5))**。N 是**文档总数**。每加一个新文档，N 变了，每个词的 IDF 都要重新算。

这就是为什么 `add_chunks()` 内部调的是 `build_index(旧+新)`——必须全量重建 `BM25Okapi` 模型。幸好对于几千个 chunk 规模（个人知识库），这是毫秒级的。

对比：向量检索不存在这个问题——新向量插入 Milvus 后立刻可被 ANN 搜到，不需要重建索引。

### Q2：为什么叫"懒加载"？跟直接初始化有什么区别？

**背景**：`get_rag_service()` 和 `get_indexer()` 都用了 lazy init 模式。

**解答**：
- **直接初始化**（放在模块顶层）：`uvicorn.run(app)` 之前就必须连接 Milvus、加载 BGE 模型。如果 Milvus 没启动，**整个服务都起不来**——连普通聊天都用不了。
- **懒加载**：服务照常启动。只有第一个 RAG 请求/文档上传才触发初始化。Milvus 不在线时，普通聊天功能完全不受影响。

这在小内存 ECS（3.5GB）上尤其重要——BGE 模型占 1.3GB，平时不加载，只在需要时加载。

### Q3：`update_bm25_after_uploads` 里为什么要检查 `_rag_service is None` 就跳过？

**背景**：函数第一行 `if _rag_service is None: return`

**解答**：分两种情况：

1. **用户还没调过 RAG 聊天**：`_rag_service` 是 None，BM25 索引从未构建。跳过是对的——因为下次 RAG 请求触发 `get_rag_service()` 时，会从 Milvus **全量读取**所有 child_chunks 构建 BM25，新上传的文档自然会被包含。

2. **用户已经调过 RAG 聊天**：`_rag_service` 已初始化，BM25 索引已构建（100 个 chunk）。此时上传新文档（+3 chunk），必须更新内存中的 BM25 索引，否则这 3 个 chunk 搜不到。

### Q4：`object_key` 和 `filename` 的区别？

**背景**：文档存储中用 `object_key` 做唯一标识而非 `filename`。

**解答**：
- `filename`：用户上传时的文件名，如 `学习笔记.md`。不同用户可能上传同名文件。
- `object_key`：MinIO 中的全路径，如 `documents/2026/06/学习笔记.md`。包含年月目录前缀，天然唯一。

用 `object_key` 做唯一标识的优点：即使两个用户上传了同名的 `学习笔记.md`，只要上传时间不同（不同月份/目录），就不会冲突。
