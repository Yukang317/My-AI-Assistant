## 一、向量检索底层原理

### 🔑 知识点

**1. L2 归一化的本质：从"模长"到"角度"**

归一化把所有向量压到单位超球面上（模长=1），消除文本长度差异对检索的干扰。归一化后，余弦相似度 = 内积（因为分母 |A|×|B| = 1×1 = 1），计算量减半。

**2. 余弦相似度 vs 欧氏距离：RAG 为什么选前者**

| 度量 | 对长度敏感 | RAG适用 |
|------|-----------|---------|
| 欧氏距离 | ✅ 敏感 | ❌ 短查询 vs 长文档会误判 |
| 余弦相似度 | ❌ 不敏感 | ✅ 只关心"方向"（语义） |

关键洞察：RAG 关心的是"意思相近"，不是"数值空间位置相近"。

**3. ANN（近似搜索）的本质：牺牲 1% 精度换 100 倍速度**

KNN = 逐一算所有向量距离，O(N)；ANN = 建索引后只搜附近几个簇，毫秒级。对于 RAG，漏掉一两个不太相关的块，LLM 不敏感。

**4. IVF 索引 = 先 K-means 聚类分桶 + 搜索时精排**

- `nlist`：建索引时定，分多少个簇
- `nprobe`：搜索时定，探测多少个簇
- `nlist=1024, nprobe=64` → 只搜 6.25% 的向量，这就是"近似"的来源

### ❓ 你的疑问

> **Q：IVF_FLAT 已经建了索引，为什么还需要 BM25、向量检索、RRF 融合、父子分块？这不是重复了吗？**

**答**：它们在不同层面，完全不重复。

| 层次 | 做的事 | 类比 |
|------|--------|------|
| 父子分块 | 文档怎么切 | 菜怎么切 |
| BM25+向量 | 用不同方式找 | 按菜名找 vs 按口味找 |
| RRF | 合并排序 | 两份菜单综合排名 |
| IVF_FLAT | 加速向量搜索 | 书架分类上架 |

IVF_FLAT 是 Milvus 内部的加速技巧，BM25/RRF/父子分块是应用层逻辑。

> **Q：L2 归一化会不会降低检索精度？**

**答**：不会，反而提升。因为余弦相似度本质不关心模长，归一化消除了长短文本匹配时的假阴性。

> **Q：ECS 没有 GPU，能不能在笔记本 GPU 跑 embedding 传结果到 ECS？**

**答**：不要这样做，不是行业标准做法。CPU 推理个人知识库绰绰有余（索引 100 篇文档 2-5 分钟，单次查询 0.1-0.3 秒）。服务应该 7×24 独立运行，不依赖开发者电脑。

## 二、父子分块策略

### 🔑 知识点

**1. 父子分块解决的核心矛盾：检索粒度 vs 上下文长度**

- 用**父块**（2048 tokens）做检索 → 语义被稀释，匹配不精确
- 用**子块**（512 tokens）直接返回 → 上下文太短
- **父子分块**：子块检索（精确命中）+ 父块返回（完整上下文）= "精确召回 + 完整供给"

**2. "不均匀"是常态，不是意外**

每个父块的子块数量不固定——最后一个子块几乎总是填不满。加上语义分块会主动在自然边界（段落、句子）切割，更增加了不均匀性。

**3. Token ≠ 字符**

- 1 token ≈ 4 个英文字符
- 1 个中文字符 ≈ 1.33 个 token
- 分块必须以 token 为单位，否则跨语言失去统一标准

### ❓ 你的疑问

> **Q：搜索到的子块返回父块，会不会返回大量无关内容？如果子块只覆盖了父块的 80%，是不是应该只返回子块？**

**答**：覆盖率阈值方案有三个致命缺陷：
1. 无法知道父块总共有几个子块（需要额外存 `total_children`）
2. 命中的那个子块可能就是最核心的一段，但它只占 12.5%，按规则就被丢弃了
3. 现代 LLM 对 70% 无关 + 30% 相关容忍度很高

行业标准做法是**去重**（同一父块多个子块命中 → 合并为一条）+ **Reranker**（重排序过滤）。

## 三、技术方案设计

### 🔑 知识点

**1. 三存储分工**

```
MinIO (:9000)  → 文件柜，存原始 PDF/Word/MD
Milvus (:19530) → 搜索引擎，存向量+chunk文本
PostgreSQL (:15432) → 档案目录，存文档台账+父子块映射
```

**2. LlamaIndex / LangChain / LangGraph 三层分工（你的追问）**

| 框架 | 定位 | 在 RAG 中的位置 |
|------|------|----------------|
| LlamaIndex | 数据驱动 | 上游 R（Retrieve）：数据加载→分块→索引→检索 |
| LangChain | 流程驱动 | 中游 AG：Prompt 组装→工具调用→LLM 输出 |
| LangGraph | 图编排 | 下游：意图路由→条件分支→HITL→子图调度 |

三者不是竞品，是上下游协作关系。

**3. 框架引入时机："疼了才引入"**

| 阶段 | 用什么 | 为什么 |
|------|--------|--------|
| Parser | 纯原生 Python | bytes→str 不需要框架 |
| Chunker | LangChain 分块器 | 语义分隔符手写要 200+ 行 |
| RRF | 先手写 | 20 行代码，理解原理后再用框架 |
| RAG Service | LangChain | Prompt 组装+工具调用就该用框架 |
| Agent | LangGraph | 你已有 SAGT 实战经验 |

### ❓ 你的疑问

> **Q：策略模式嵌套策略模式会不会显得没必要？**

**答**：现阶段确实没必要。嵌套会导致组合爆炸（3 文档类型 × 3 PDF 子策略 = 9 种路径）。正确做法是**平面化注册**——所有解析器注册到协调器同一层，按优先级排序。未来加 MinerU 只需加一行。

> **Q：LlamaIndex 和 LangChain 各自的优势是什么？为什么企业级系统要一起用？**

**答**：见上方三层分工表。你在 SAGT 项目里已经验证了"LangGraph 主图 + 多个子图/工具"的模式。

> **Q：LangGraph 在本项目的正确定位？**

**答**：LangGraph 是整个个人 AI 助理的**核心骨架**（主程序），RAG 只是它身上的一个器官（子图/工具）。和 SAGT 一样：主图做意图检测+条件路由，子图独立开发/测试/部署。

## 四、策略模式 & 架构设计

### 🔑 知识点

**1. Parser 为什么"薄"**

Parser 只做 `bytes → str`（格式消除），真正的"理解文本结构"被故意推到 Chunker 层。PDF/DOCX 的二进制解析复杂度被 pymupdf4llm 和 markitdown 吃掉了。

Parser 薄不是缺陷——它的价值在架构上：
- 消除 if/elif 分支
- 未来加新格式：写一个新文件 + coordinator 列表加一行
- Coordinator 不需要知道哪个解析器在处理

**2. 管道模式（Pipeline Pattern）**

每一步的输出是下一步的输入。好处：出问题时精确定位到"第几步挂了"，每步抛出的异常就是天然的断点。

**3. 防腐层（Anti-Corruption Layer）**

`_prepare_chunks_for_milvus` 就是防腐层：上游产出 `ParentChunk` dataclass，下游期望 `Dict[str, Any]`。中间夹一层转换，两边都不需要为对方妥协自己的数据结构。

**4. 依赖注入 vs 自管理**

| 模式 | 示例 | 适用场景 |
|------|------|---------|
| 依赖注入 | `VectorRetriever(embedding_service, vector_store)` | 共享重型对象（BGE 1.3GB） |
| 自管理 | `BM25Index()` 自己 new 对象 | 轻量对象，独立使用 |

**5. MinIO-before-Milvus 写入顺序**

如果 MinIO 上传失败，直接抛异常，不污染 Milvus。如果先写 Milvus 再上传 MinIO 失败，需要回滚——Milvus 不支持事务回滚。

**6. zip() 的"拉链"比喻**

```python
for chunk, vec in zip(parents, parent_vectors):
    # 左边一个齿（chunk）对应右边一个齿（vector），一一咬合
```

### ❓ 你的疑问

> **Q：Parser 感觉没什么有价值的东西，就调了一下库？**

**答**：Parser 确实薄——因为它的职责被严格限定为"消除格式差异"。厚的是策略模式架构本身：开闭原则（对扩展开放，对修改关闭）。

## 五、BM25 关键词检索

### 🔑 知识点

**1. `_bm25_model` 里面存的不是文本，是统计模型**

```
BM25Okapi 内部结构：
  corpus_size = 500          # 总 chunk 数
  avgdl = 128.5              # 平均词数
  idf = {                    # 每个词的"稀有度权重"
      "DeepSeek": 3.2,       # 稀有词 → 权重高
      "的": 0.02,            # 常见词 → 基本忽略
  }
  doc_freqs = [...]          # 每个 chunk 的词频统计
  doc_len = [...]            # 每个 chunk 的长度
```

它是数学引擎：知道 `chunk_index → 分数`，但没有原文。

**2. 并行数组模式**

```
_bm25_model[i]  → 分数（统计模型）
corpus[i]       → 原文（返回文本内容）
doc_ids[i]      → 来源（标注文件路径）
```

三个数组用同一个 index 串联，一一对应。

**3. BM25 全量重建约束**

BM25 的 IDF 依赖全量语料库——增删任何 chunk 后，所有词的"稀有度"都可能变化，必须全量重建。这是 BM25 算法的固有限制，不是实现 bug。

**4. 用空间换时间**

建索引慢（分词+统计，3 分钟），但检索极快（纯数学运算，0.01 秒）。所有搜索引擎（Google、Elasticsearch）都这么干。

### ❓ 你的疑问

> **Q：`self._bm25_model` 里存的是啥？把上传的文件全部转为纯文本再分词，这个工作量好大，这是在为什么做准备？**

**答**：见上方——它存的是 IDF 表 + 词频统计 + 文档长度。工作量大的原因是"用空间换时间"：建索引时辛苦一次，以后每次查询都是毫秒级。

> **Q：`self.corpus` 和 `self.doc_ids` 在整个数据流过程中是什么角色？**

**答**：
- `_bm25_model` = 数学引擎（`index → 分数`，没有原文）
- `corpus` = 原文索引（`index → text`，返回文本内容）
- `doc_ids` = 来源索引（`index → 文件路径`，标注来源）

一次 `search()` 调用链：分词 → `get_scores()` 拿分数 → 用分数最高的 index 去 `corpus` 和 `doc_ids` 取原文和来源。

## 六、RRF 融合排序

### 🔑 知识点

**1. RRF 公式：`score = Σ 1/(k + rank)`**

不关心原始分数大小，只关心排名——天然免疫 BM25 和余弦相似度之间的尺度差异。

**2. k=60 阻尼器的含义**

- k=0 → rank=1 的文档垄断所有分数（不民主）
- k=60 → rank=1 贡献 1/61 ≈ 0.0164，rank=3 贡献 1/63 ≈ 0.0159，差距被压缩（民主化）
- k 越大越"民主"，来自 SIGIR 2009 论文实验结论

**3. 文档级融合 vs Chunk 级融合**

必须用**文档级融合**（以 doc_id 为键）：BM25 用 `chunk_index`（整数），Vector 用 `chunk_id`（字符串），两者无法对齐。`doc_id` 是唯一公共键。

**4. RRF 为什么优于分数归一化**

| 方法 | 问题 |
|------|------|
| Min-Max 归一化 | 受异常值影响大 |
| Softmax | 对分布形状敏感 |
| **RRF** | 分布无关，跨系统天然可比 |

### ❓ 你的疑问

> **Q：`_build_rank_map` 函数是干什么的？rank 和 score 什么关系？**

**答**：输入是按 score 降序排列的 chunk 列表，`enumerate(..., start=1)` 给排名（第 1 名 rank=1，第 2 名 rank=2...）。然后按 `doc_id` 去重：同一个文档可能多个 chunk 被命中，只保留排名最靠前的那个。输出是 `{doc_id: {rank, text, ...}}`。RRF 不关心原始 score 数值，只看 rank。

## 七、Milvus 实战关键细节

### 🔑 知识点

**1. `search()` 返回的数据结构——双重列表**

```python
results = self.client.search(...)
# results 是 List[List[dict]]，不是 List[dict]！
# results[0] 才是第一条查询向量的命中列表
# results[0] 为空时是 [[]]，不是 []，所以 `if not results` 不会触发
```

**2. filter 表达式的引号坑**

Python `str(list)` 产生单引号 `['a', 'b']`，Milvus 要求双引号 `["a", "b"]`。必须用 `json.dumps()` 转换。

**3. `delete()` 自带返回计数**

Milvus 的 CUD 操作都返回 `{"delete_count": N}` / `{"insert_count": N}`，不需要先 delete 再 query 拿计数（删完数据已经没了）。

**4. `flush()` 的必要性**

Milvus 写入后数据在内存缓冲，重启会丢。`flush()` 强制刷盘。生产实践：索引完成后一定要 flush。

### ❓ 你的疑问

> **Q：`search_params` 为什么只有 `nprobe`，没有 `nlist`？**

**答**：nlist 在建索引时就定了（`_create_index_if_not_exists`），search 阶段只能调 nprobe。nlist 决定分多少个簇，nprobe 决定搜几个簇。

> **Q：`anns_field="content_vector"` 是什么意思？**

**答**：Milvus 一个 Collection 可以有多个向量字段。指定 `anns_field` 就是告诉它"在这个字段上做 ANN 搜索"。字段名必须和 Schema 定义一致。

> **Q：child schema 要不要加 `doc_type` 字段？**

**答**：要加。不是因为和 parent 一致，而是需求上需要支持按文件类型过滤搜索。如果 child 没这个字段，过滤只能后置做（破坏 top_k 语义）。

## 八、内存管理

### 🔑 知识点

**1. Python 两层内存回收**

| 机制 | 触发 | 局限 |
|------|------|------|
| 引用计数 | `del` 瞬间，计数归零立即释放 | 处理不了循环引用 |
| GC | 自动定期扫描 | 时机不确定，可能延迟 |

**2. `gc.collect()` 为什么需要**

BGE 模型内部大量嵌套对象（tokenizer、model、config），不排除有隐蔽的循环引用。`del self.model` 释放大部分 1.3GB，`gc.collect()` 手动触发 GC 扫描清理循环引用——多花 0.01 秒，确保不给 ECS 3.5GB 内存留隐患。

**3. BGE CPU 推理的性能现实**

- 批量索引 100 篇：2-5 分钟（一次性）
- 增量索引 1 篇：2-5 秒
- 查询向量化 1 条：0.1-0.3 秒

个人知识库场景 CPU 绰绰有余。

## 九、跨模块的架构概念

| 概念 | 出现位置 | 一句话 |
|------|---------|--------|
| 管道模式 | Indexer | 上一步输出→下一步输入，异常天然断点 |
| 防腐层 | Indexer, VectorRetriever | 中间转换层，上下游玩各自的数据结构 |
| 依赖注入 | VectorRetriever | 构造函数只接收不创建，共享重型对象 |
| 策略模式 | Parser | 消除 if/elif，开闭原则 |
| 并行数组 | BM25 | 三个数组同一 index 串联 |
| 用空间换时间 | BM25 | 建索引慢，检索极快 |
| 幂等删除 | Indexer, MinIO | 删不存在的东西和删存在的东西对外表现一致 |
| 契约信任+防御校验 | Indexer | 信任 embed() 的契约，写入前再校验数量 |
| 编排层模式 | Indexer, VectorRetriever | 自己不干活，调别人协调 |

**最隐蔽的两类 bug**：
