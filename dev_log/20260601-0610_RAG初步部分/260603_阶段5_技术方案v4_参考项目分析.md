## 二、自提疑问 & 解答

### Q1：BGE 做了 L2 归一化，Milvus 应该用 IP 还是 COSINE？

**背景**：`embedding.py` 里写了 `normalize_embeddings=True`，但技术方案里没提 Milvus 的度量方式应该怎么选。

**解答**：
- L2 归一化后向量模长=1
- COSINE 相似度公式：`cos(A,B) = (A·B) / (|A|×|B|)`，当 |A|=|B|=1 时，退化为 `A·B`
- IP（内积）公式：`IP(A,B) = A·B`
- **两者在归一化后完全等价**，但 IP 计算量减半（不用算模长除法）
- **结论**：用 `IP`，不是 `COSINE`。mildoc 用 COSINE 是因为它用的阿里云 API 没有做归一化。

### Q2：为什么需要 `embedding_model` 字段？

**背景**：mildoc 的 Schema 里有这个字段，当前方案没有。

**解答**：
- 将来可能换 Embedding 模型（BGE → M3E → API → 更新的模型）
- 不同模型的向量维度/语义空间不同，混在一起搜索会出问题
- 有此字段可以精确过滤：`filter='embedding_model == "bge-large-zh-v1.5"'`
- 换模型时：新模型写新向量（带新标记），旧向量可以 gradual migration 或 one-shot full-refresh

### Q3：为什么需要三种索引模式，平时不就用"上传时索引"吗？

**背景**：mildoc 有 full-refresh/backfill/listen 三种模式，看起来有点 over-engineering。

**解答**：
- **upload（上传同步）**：日常用，够了
- **backfill（断点续传）**：Milvus 是 Docker 容器，数据可能因磁盘满/升级/误操作丢失。没有 backfill 就得手动重新上传所有文件
- **full-refresh（全量重建）**：换了 Embedding 模型后，旧向量全废，需要一键重建

这两个"看起来用不上"的模式，**一旦需要就是救命稻草**。mildoc 团队显然是在生产环境踩过坑才加了这些。

### Q4：语义分块这么好，为什么 mildoc 只用 RecursiveCharacterTextSplitter？

**背景**：你的方案主打语义分块（MarkdownNodeParser），mildoc 却只用固定递归分块。

**解答**：
- mildoc 是企业客服场景，文档以 FAQ/手册为主，结构简单
- `RecursiveCharacterTextSplitter(2048/128)` 对这类文档够用，且稳定可预测
- 语义分块的优势在**有结构的文档**（Markdown 笔记、技术文档），这正是个人知识库的主要内容
- **你的方案在分块策略上已超越 mildoc**

## 五、关键概念卡片

| 概念 | 一句话解释 | 来自 |
|------|----------|------|
| `IVF_FLAT` | 先聚类再暴力搜索的索引类型，适合百万级以下向量 | mildoc |
| `nlist` | IVF 的聚类中心数，越大索引越慢搜索越快 | mildoc |
| `nprobe` | 搜索时探测的聚类数，越大精度越高速度越慢 | mildoc |
| `IP`（内积） | L2 归一化后等价余弦相似度，但少一次模长除法 | 本次分析 |
| `IngestionCache` | 基于 Redis 的文档去重缓存，避免重复索引 | llamaindex |
| `DocstoreStrategy.UPSERTS_AND_DELETE` | 增量索引策略：存在则更新，不存在则插入 | llamaindex |
| `backfill` | 断点续传式全量索引，只处理缺失的文档 | mildoc |
| `RRF`（Reciprocal Rank Fusion） | 多路检索结果融合算法：`score = Σ 1/(k+rank)` | llamaindex |
| `Argon2` | 密码哈希算法，比 bcrypt 更安全 | llamaindex |
| `Langfuse` | 开源 LLM 全链路追踪平台 | mildoc |
