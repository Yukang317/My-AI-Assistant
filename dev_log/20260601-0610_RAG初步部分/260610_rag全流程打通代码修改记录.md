# 详细修复记录（分步骤）

## 1. 基础设施层

**现象**：`step23_rag.py` 启动后，首次 RAG 请求时进程卡死无响应。

**排查**：
```bash

# → 空，所有容器都不在运行
```

**根因**：上次会话结束后 Docker 未启动。`MilvusVectorStore.__init__()` 中 `MilvusClient(uri="http://127.0.0.1:19530")` 无显式超时，连接到不存在的端口会挂起。

**修复**：
```bash
docker compose up -d
```

**验证**：
```bash
docker compose ps
# NAME          STATUS
# pa-etcd       Up (healthy)
# pa-minio      Up (healthy)
# pa-milvus     Up (healthy)
# pa-postgres   Up (healthy)
```

```python
elif self.mode == "api":
```

## 3. 向量存储层（共 5 个 pymilvus SDK 兼容问题）

> **本节 Bug 在 RAG 完整链路中的位置：**
> 
> ```
> 上传文档 ──→ 解析 ──→ 分块 ──→ Embedding ──→ MinIO 写 ──→ Milvus 写 ──→ flush ──→ ✅
>                                                       ↑            ↑          ↑
>                                                     3.4 在这里     │       3.5 在这里
>                                                       Schema       │       list→str
>                                                      缺 doc_md5     │       参数变更
>                                                                    │
>                                           初始化（首次请求）─────────┘
>                                           ↑
>                                       3.1 在这里
>                                       Collection 有壳无索引
>                                       load_collection 失败
> ```
> 
> **关键区别：**
> - **3.1** 是初始化门槛——连 Collection 都加载不了，写入和检索都无从谈起
> - **3.4** 是写入中段的 Schema 校验——parent 能写、child 不能写，导致父子数据不完整
> - **3.5** 是写入末端的持久化——数据已在内存但刷不进磁盘，重启即丢失

### 3.1 已存在 Collection 不创建索引 → load_collection 失败

**现象**：`MilvusException: index not found[collection=parent_chunks]`

**文件**：`rag/storage/vector_store.py` → `_create_collection_if_not_exists()`

**根因**：之前的 buggy 运行创建了 Collection 但索引创建失败。代码逻辑是「Collection 存在 → 直接 load」，不会补建索引。

**修复前**：
```python
if not self.client.has_collection(collection_name):
    self.client.create_collection(collection_name, schema=schema)
    self._create_index_if_not_exists(collection_name)
self.client.load_collection(collection_name)
```

**修复后**：
```python
if not self.client.has_collection(collection_name):
    self.client.create_collection(collection_name, schema=schema)
# 无论新建还是已存在，确保索引已建好再 load
self._create_index_if_not_exists(collection_name)
self.client.load_collection(collection_name)
```

> **📍 链路阶段：初始化（⭐ 首次请求触发）。** 这是写入/检索前的必经门槛。`_create_collection_if_not_exists()` 在 `MilvusVectorStore.__init__()` 中被调用（懒加载），前一次 buggy 运行成功执行了 `create_collection()` 但 `create_index()` 崩溃，留下"有壳无索引"的 Collection。修复前的 `if not exists` 逻辑只在新建时建索引，已存在的 Collection 直接 `load_collection()` 就炸了。修后把 `_create_index_if_not_exists()` 提到 `if` 块外，保证无论新建还是已存在都补建索引再 load。

### 3.2 `has_index()` 方法不存在

**现象**：`AttributeError: 'MilvusClient' object has no attribute 'has_index'`

**文件**：`rag/storage/vector_store.py` → `_create_index_if_not_exists()`

**根因**：pymilvus 2.4.x 的 `MilvusClient`（HTTP 客户端）没有 `has_index()` 方法，只有 `list_indexes()`。

**修复前**：
```python
if not self.client.has_index(collection_name, "content_vector"):
```

**修复后**：
```python
indexes = self.client.list_indexes(collection_name)
if "content_vector" not in indexes:
```

---

### 3.3 `create_index()` 参数格式错误

**现象**：`ParamError: wrong type of argument [index_params], expected type: [IndexParams], got type: [dict]`

**根因**：新版 `MilvusClient.create_index()` 不接受 `index_type`/`metric_type`/`params` 分散参数，必须打包成 `IndexParams` 对象（`list` of `IndexParam`）。

**修复前**：
```python
self.client.create_index(
    collection_name, field_name="content_vector",
    index_type=Config.MILVUS_INDEX_TYPE,
    metric_type=Config.MILVUS_METRIC_TYPE,
    params={"nlist": Config.MILVUS_INDEX_NLIST}
)
```

**修复后**：
```python
index_params = self.client.prepare_index_params(
    "content_vector",
    index_type=Config.MILVUS_INDEX_TYPE,
    metric_type=Config.MILVUS_METRIC_TYPE,
    index_name="content_vector_index",
    params={"nlist": Config.MILVUS_INDEX_NLIST},
)
self.client.create_index(
    collection_name=collection_name,
    index_params=index_params,
)
```

> `prepare_index_params()` 是 `BaseClient` 的类方法，自动构造合法的 `IndexParams` 对象，推荐使用。

---

### 3.4 child_chunks Schema 缺少 `doc_md5` 字段

**现象**：`DataNotMatchException: Attempt to insert an unexpected field 'doc_md5' to collection without enabling dynamic field`

**根因**：parent_chunks 的 Schema 有 `doc_md5`，但 child_chunks 的 Schema 漏掉了。`insert_parent_child()` 对子表插入时也带了 `doc_md5`，被 Milvus 拒绝。

**修复**：在 `_build_child_schema()` 中补字段：
```python
child_schema.add_field("doc_md5", DataType.VARCHAR, max_length=32)
```

**重要**：改 Schema 后必须删除旧 Collection（Milvus 不支持 ALTER TABLE ADD COLUMN）：
```python
from pymilvus import MilvusClient
c = MilvusClient(uri='http://127.0.0.1:19530', db_name='personal_assistant')
c.drop_collection("parent_chunks")
c.drop_collection("child_chunks")
```

> **📍 链路阶段：Milvus 向量写入（insert_parent_child 对 child 子表插入时）。** 数据已过分块和 Embedding，在"落库"这一步被 Schema 挡下。分块器产出的 child_chunks 字典带 `doc_md5`，`insert_parent_child()` 对父子两表传入同一份数据结构，parent Schema 有 `doc_md5` 字段 ✅，child Schema 漏定义了 → Milvus 拒绝 `unexpected field`。修复后 child Schema 补齐第 3 个字段 `doc_md5`，两表写入一致。

---

### 3.5 `flush()` 参数类型变更

**现象**：`ParamError: wrong type of argument [collection_name], expected type: [str], got type: [list]`

**根因**：旧版 pymilvus `flush()` 接受 `List[str]`，新版 `MilvusClient` 只接受单个 `str`。

**修复前**：
```python
self.client.flush([Config.MILVUS_PARENT_COLLECTION, Config.MILVUS_CHILD_COLLECTION])
```

**修复后**：
```python
self.client.flush(Config.MILVUS_PARENT_COLLECTION)
self.client.flush(Config.MILVUS_CHILD_COLLECTION)
```

> **📍 链路阶段：flush 持久化（写入链路最后一步）。** 数据已通过 `insert()` 写入 Milvus 内存，`flush()` 的作用是将内存中的未持久化数据刷入磁盘。pymilvus 2.4.x 的 `MilvusClient.flush()` 只接受单个 `str`，不能再传 `List[str]` 批量刷。不 flush 的后果：容器重启后数据丢失。修复后拆成两次独立调用，数据安全落盘。

# 核心教训

### 1. SDK 版本迁移是陷阱重灾区

本次 10 个问题中，7 个是 pymilvus / openai SDK 的 breaking changes。代码按旧版 API 风格编写，但 `pyproject.toml` 锁定了新版（pymilvus 2.4.x, openai 2.38.0）。

**关键 API 变化速查表**：

| SDK | 旧版写法 | 新版写法 |
|-----|----------|----------|
| pymilvus | `has_index(coll, field)` | `list_indexes(coll)` |
| pymilvus | `create_index(..., index_type=, metric_type=, params=)` | `create_index(..., index_params=IndexParams)` |
| pymilvus | `flush([coll_a, coll_b])` | `flush(coll_a); flush(coll_b)` |
| openai | `AsyncOpenAI(..., retry_on_status_codes=)` | 参数已移除 |

### 2. Lazy Init 的双刃剑

懒加载避免了启动时的连接失败，但也把错误推迟到首次请求。建议在服务启动后打印 "RAG 服务就绪" 或用 health check 端点主动探测依赖。

### 3. 去重逻辑救了数据

MinIO 的 MD5 去重机制使半失败的残留可以用新文件名绕过去重做，同时保护已成功的数据不被覆盖——这在反复调试时非常实用。

### 4. 排查方法论

遵循「自底向上」原则：基础设施 → API 层 → 存储层 → 服务层，每层修复后验证，避免跳层猜测。本次调试如果不先启动 Docker，后面所有错误都是误导性的连接失败。
