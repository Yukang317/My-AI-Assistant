## 一、代码全貌

### 1.1 双格式 SSE 解析（sendMessage 的核心改动）

这是整个前端最关键的改动。后端有两种 SSE 格式：

```
旧版（普通聊天，step22 时代）：
  {"content": "你好", "finished": false, "created_at": "..."}
  {"content": "", "finished": true, "created_at": "..."}

新版（RAG 聊天，step23 新增）：
  {"type": "sources",  "sources": [...], "finished": false}
  {"type": "delta",    "content": "你好", "finished": false}
  {"type": "complete", "content": "", "finished": true, "token_usage": {...}}
```

**兼容策略：用 `data.type` 是否存在来分流**。两种格式都有 `finished` 字段，但 RAG 格式多了 `type`：

```javascript
if (data.type === "sources") {
    // RAG：在助手气泡前插入来源文档卡片
    if (!sourcesShown && data.sources && data.sources.length > 0) {
        showSourcesCard(data.sources, wrapper);
        sourcesShown = true;
    }
} else if (data.type === "delta") {
    // RAG：LLM 逐 token 输出 → 累积后 Markdown 渲染
    accumulated += data.content;
    bubble.innerHTML = renderMarkdown(accumulated);
} else if (data.type === "complete") {
    // RAG：流结束，设置时间戳
    if (data.created_at) timeDiv.textContent = formatTime(data.created_at);
} else if (data.finished && data.type === undefined) {
    // 普通：流结束（旧格式，无 type 字段）
    if (data.created_at) timeDiv.textContent = formatTime(data.created_at);
} else if (data.content !== undefined && data.type === undefined) {
    // 普通：文本增量（旧格式，无 type 字段）
    accumulated += data.content;
    bubble.innerHTML = renderMarkdown(accumulated);
}
```

**为什么不用 `data.finished` 来判断结束？** 因为 RAG 格式的 `sources` 和 `delta` 事件里 `finished` 都是 `false`，只有 `complete` 才是 `true`。如果只靠 `finished`，RAG 格式下收到 `sources`（finished=false）会错误地当成普通文本内容来处理。

### 1.2 RAG Toggle 开关

纯 CSS 实现的滑块开关，不依赖任何 JS 库：

```css
/* 隐藏原生 checkbox，用 span 模拟滑块 */
.rag-toggle input { display: none; }

/* 轨道 */
.rag-toggle-slider {
    width: 38px; height: 20px;
    background: rgba(255,255,255,0.3);  /* 关闭时半透明白 */
    border-radius: 10px;
    position: relative;
    transition: background 0.2s;
}

/* 圆形滑块 */
.rag-toggle-slider::after {
    content: "";
    width: 16px; height: 16px;
    background: #fff;
    border-radius: 50%;
    position: absolute;
    top: 2px; left: 2px;
    transition: transform 0.2s;
}

/* 开启状态：轨道变绿 + 滑块右移 */
.rag-toggle input:checked + .rag-toggle-slider {
    background: #2ecc71;
}
.rag-toggle input:checked + .rag-toggle-slider::after {
    transform: translateX(18px);
}
```

JS 只做两件事：更新全局变量 `useRag`，控制输入区提示标签的显示/隐藏：

```javascript
function toggleRag() {
    useRag = ragToggle.checked;
    if (useRag) {
        ragIndicator.classList.add("active");
    } else {
        ragIndicator.classList.remove("active");
    }
}
```

`useRag` 的值在 `sendMessage()` 调用时被捕获进请求体——用户发消息后切换开关不影响正在进行的流式请求。

### 1.4 文档上传流程

```
用户点击 [+] → triggerUpload() → fileInput.click()
→ 系统文件对话框
→ 用户选择文件 → handleFileSelect(event)
→ uploadDocument(file)
→ new FormData() + fetch POST /api/documents/upload
→ 成功后 loadDocuments() 刷新列表
```

隐藏的 `<input type="file">` 是关键——浏览器的文件选择对话框只能由用户点击 `<input type="file">` 触发（安全限制），不能由 JS 直接打开。所以"+"按钮只是触发 `fileInput.click()` 的代理。

```javascript
function triggerUpload() {
    fileInput.click();  // 触发隐藏的 <input type="file">
}
```

### 1.5 文档删除与 `encodeURIComponent`

删除端点的路由是 `DELETE /api/documents/{object_key:path}`，后端用了 `:path` 转换器。

`object_key` 如 `documents/2026/06/笔记.txt`，含有两个 `/`。如果直接拼 URL：

```
DELETE /api/documents/documents/2026/06/笔记.txt
```

中间件可能把 `/` 解析为路径分隔符，或者特殊字符导致 404。

**解决**：前端用 `encodeURIComponent()` 转义：

```javascript
const response = await fetch(
    "/api/documents/" + encodeURIComponent(objectKey),
    { method: "DELETE" }
);
```

`encodeURIComponent("documents/2026/06/笔记.txt")` → `"documents%2F2026%2F06%2F%E7%AC%94%E8%AE%B0.txt"`。FastAPI 收到后自动 decode 回原始值传给 `:path` 参数。

### 1.6 来源文档卡片

RAG 检索完成后、LLM 文本生成之前，前端收到 `type: "sources"` 事件。此时在助手气泡前插入一张淡黄色卡片：

```
┌─────────────────────────────────────┐
│ 📖 参考了 3 个文档片段          ▼   │  ← 点击折叠
├─────────────────────────────────────┤
│ 📄 笔记.txt                   85%   │
│ 📄 技术文档.md                 72%   │
│ 📄 年报.pdf                    68%   │
└─────────────────────────────────────┘
```

卡片默认展开，用户可点击标题栏折叠（`classList.toggle("collapsed")`）。

`showSourcesCard(sources, wrapper)` 的第二个参数 `wrapper` 是插入位置参考——卡片通过 `messagesDiv.insertBefore(card, wrapper)` 插在助手气泡之前。

这就是为什么 `createStreamingAssistantBubble()` 的返回值从 `{bubble, timeDiv}` 改成了 `{wrapper, bubble, timeDiv}`。

## 二、自提疑问 & 解答

### Q1：为什么 SSE 解析里要区分"有没有 `type` 字段"？

**背景**：后端两套 SSE 格式天然不兼容。旧版用 `content`+`finished`，新版用 `type`（sources/delta/complete）。前端必须同时支持两种。

**解答**：判断依据选 `data.type` 而非 `data.finished`，因为：

```
RAG sources 事件：{"type": "sources", "finished": false}
普通 content 事件：{"content": "你好", "finished": false}
```

两者 `finished` 都是 `false`！如果只看 `finished`，RAG 的 sources 事件会被误当成普通文本内容，`undefined + rendered` 导致气泡显示 "undefined"。

用 `data.type` 做第一级判断：有 type → RAG 格式 → 按类型分发；无 type → 普通格式 → 按 finished 判断。

### Q2：`createStreamingAssistantBubble` 为什么多返回一个 `wrapper`？

**背景**：之前只返回 `{bubble, timeDiv}`。现在 RAG 需要在助手气泡"之前"插入来源卡片，需要知道气泡外层容器的 DOM 引用。

**解答**：

```
DOM 树：
  messagesDiv
    ├── user wrapper       ← 用户消息
    ├── sources card       ← 🆕 插在这里（insertBefore(card, wrapper)）
    └── assistant wrapper  ← wrapper 引用（createStreamingAssistantBubble 返回的）
          ├── bubble
          └── timeDiv
```

如果不返回 `wrapper`，就只能 `messagesDiv.appendChild(card)`，卡片会出现在助手气泡"之后"，逻辑上不对——参考文档应该在回答之前展示。

### Q3：`encodeURIComponent` 为什么在删除 API 调用时需要，但上传 API 不需要？

**背景**：上传是 POST `/api/documents/upload`（固定路径），删除是 DELETE `/api/documents/{object_key}`（动态路径）。

**解答**：上传路径没有变量部分，URL 是固定的。删除的 `object_key` 是 URL 路径的一部分，含 `/` 和中文等特殊字符。如果不转义，HTTP 请求可能被路由到错误的路径。

**类比**：`object_key` 就像快递单号，里面可能有特殊符号。如果不编码，扫码枪可能把 `AB/C` 扫成两个包裹 `AB` 和 `C`。

### Q4：为什么文件上传用 `FormData` 而不是 JSON？

**背景**：所有其他 API 都发 `Content-Type: application/json`，只有上传用 `FormData`。

**解答**：JSON 只能传文本，文件是二进制数据。`FormData` 是浏览器原生支持的多部分表单格式（`multipart/form-data`），能高效传输二进制文件。后端 FastAPI 的 `UploadFile` 类型也只接受 `FormData` 格式。
