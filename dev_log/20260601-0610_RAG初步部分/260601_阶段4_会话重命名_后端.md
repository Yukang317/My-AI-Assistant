## 〇、做了什么 & 为什么

### 当前情况

会话标题固定为"第一条用户消息"。`list_sessions()` 用子查询从 messages 表取：
```sql
(SELECT content FROM messages WHERE session_id = m.session_id AND role = 'user' ORDER BY id LIMIT 1) as title
```

用户想自定义标题（比如"面试准备"、"旅行计划"），需要：
1. 一个地方**存**自定义标题 → 新增 `sessions` 表
2. 一个 API **改**自定义标题 → 新增 `PUT /sessions/{id}/rename`
3. 查询时**优先用**自定义标题 → 改 SQL 用 COALESCE

### 为什么新增 sessions 表而不是在 messages 表加字段？

- messages 表存的是"消息"，一行一条
- 标题是"会话级别"的属性，放 messages 里不合适（放哪条消息？第一条？删了怎么办？）
- 新建 sessions 表是标准做法，以后还可以扩展（会话颜色、置顶、归档等）

### 为什么用 COALESCE 而不是 IFNULL？

`COALESCE(a, b, c)` 取第一个非 NULL 的值。SQLite 里 `IFNULL` 功能一样，但 `COALESCE` 是 SQL 标准，更通用。

> 2. `title` 必须是字符串类型
> 3. 如果缺了或类型不对，FastAPI 自动返回 422 错误

### 改动 ③ `list_sessions()` — 查询改用 LEFT JOIN + COALESCE

**位置**：`list_sessions()` 函数里的 SQL 查询（约第 176 行）

**原来的 SQL**：
```sql
SELECT session_id,
    (SELECT content FROM messages WHERE session_id = m.session_id AND role = 'user' ORDER BY id LIMIT 1) as title,
    COUNT(*) as msg_count,
    MAX(created_at) as updated_at
FROM messages m
GROUP BY session_id
ORDER BY updated_at DESC
```
```python
#   rows = [
#       ("abc123", "你好我叫张三", 3, "2026-05-27 10:02"),   # 行 0
#       ("def456", "今天天气如何", 2, "2026-05-27 11:01"),   # 行 1
#   ]
```

**改成**：
```sql
SELECT
    m.session_id,
    COALESCE(
        s.custom_title,
        (SELECT content FROM messages WHERE session_id = m.session_id AND role = 'user' ORDER BY id LIMIT 1)
    ) as title,
    COUNT(*) as msg_count,
    MAX(m.created_at) as updated_at
FROM messages m
LEFT JOIN sessions s ON s.session_id = m.session_id
GROUP BY m.session_id
ORDER BY updated_at DESC
```

**逐行解释**：

| 部分 | 解释 |
|------|------|
| `LEFT JOIN sessions s ON s.session_id = m.session_id` | 左连接：messages 的每一行都保留，sessions 表里能匹配上的就拼过来，匹配不上的填 NULL |
| `COALESCE(s.custom_title, 子查询)` | 优先取 `s.custom_title`（自定义标题），如果是 NULL（没重命名过），就用子查询取第一条用户消息 |
| 子查询和原来一样 | 取 `role='user'` 的第一条消息内容作为默认标题 |

**为什么用 LEFT JOIN 而不是 INNER JOIN**：INNER JOIN 要求两边都有记录。如果 sessions 表里没有某个会话的记录（新建的、旧的），INNER JOIN 会直接**丢掉**这行。LEFT JOIN 则保留 messages 的所有行，匹配不上的填 NULL。

**Python 代码部分不变**：原来的 `row[1] or "新会话"` 兜底还在——如果 COALESCE 返回 NULL（意味着既没有自定义标题，也没有用户消息——空会话），Python 侧填"新会话"。

> ⚠️ 注意：SQL 改了之后，`row[0]` 还是 session_id，`row[1]` 还是 title，`row[2]` 还是 msg_count，`row[3]` 还是 updated_at——顺序没变，后面的 Python 代码不用改。

### 改动 ④ `create_session()` — 同步写 sessions 表

**位置**：`create_session()` 函数（约第 278 行）

**原来的代码**：
```python
@app.post("/sessions")
def create_session():
    new_id = uuid.uuid4().hex
    return {"session_id": new_id}
```

**改成**：
```python
@app.post("/sessions")
def create_session():
    new_id = uuid.uuid4().hex
    conn = sqlite3.connect(DB_PATH)
    conn.execute(
        "INSERT INTO sessions (session_id) VALUES (?)",
        (new_id,),
    )
    conn.commit()
    conn.close()
    return {"session_id": new_id}
```

**为什么现在要写数据库了**：之前"空会话"不存在于 messages 表，GET /sessions 查不到它。现在 sessions 表里有了记录，list_sessions 用 LEFT JOIN 时能关联上，但因为没有 messages 记录… 等等，不对！

> 🤔 **自提疑问**：空会话在 sessions 表有记录，但 messages 表没消息。LEFT JOIN 从 messages 出发，messages 没这个 session_id 的行，所以 LEFT JOIN 结果里根本不会有这个空会话。**所以创建会话时写 sessions 表其实不影响侧边栏显示**——空会话仍然不会出现在列表里。
>
> **那为什么要写？** 为了 `PUT /sessions/{id}/rename` 里的 UPSERT。
> - 如果 sessions 表已有行：`ON CONFLICT DO UPDATE` 直接更新
> - 如果 sessions 表没有行（旧会话，在这个功能上线前创建的）：`INSERT` 创建新行
>
> 两种都能正常工作。但既然每次创建会话都写一行是最干净的做法，我们就这么做。

### 改动 ⑤ `delete_session()` — 同时删 sessions 行

**位置**：`delete_session()` 函数（约第 199 行）

**原来的代码**：
```python
def delete_session(session_id):
    conn = sqlite3.connect(DB_PATH)
    conn.execute(
        "DELETE FROM messages WHERE session_id = ?",
        (session_id,),
    )
    conn.commit()
    conn.close()
```

**改成**（加一行 DELETE）：
```python
def delete_session(session_id):
    conn = sqlite3.connect(DB_PATH)
    conn.execute("DELETE FROM messages WHERE session_id = ?", (session_id,))
    conn.execute("DELETE FROM sessions WHERE session_id = ?", (session_id,))
    conn.commit()
    conn.close()
```

**为什么要删两处**：messages 和 sessions 是独立的表，外键没有设 CASCADE。如果只删 messages 不删 sessions，sessions 表会留下孤儿行。虽然不影响功能（LEFT JOIN 以 messages 为准），但脏数据留着不好。

### 改动 ⑥ 新增 `PUT /sessions/{session_id}/rename` 端点

**位置**：放在 `DELETE /sessions/{session_id}` 后面（约第 295 行）

**完整代码**：
```python
@app.put("/sessions/{session_id}/rename")
def rename_session(session_id: str, req: RenameRequest):
    """
    PUT /sessions/{session_id}/rename：重命名会话

    请求体：{"title": "新标题"}
    成功返回：{"ok": true, "title": "新标题"}
    失败返回：{"ok": false, "error": "标题不能为空"}
    """
    # ---- 1. 校验标题 ----
    title = req.title.strip()
    if not title:
        return {"ok": False, "error": "标题不能为空"}

    # ---- 2. UPSERT：有则更新，无则插入 ----
    # INSERT ... ON CONFLICT DO UPDATE 是 SQLite 的 UPSERT 语法：
    #   - session_id 已存在 → 更新 custom_title
    #   - session_id 不存在 → 插入新行（兼容旧会话）
    conn = sqlite3.connect(DB_PATH)
    conn.execute(
        "INSERT INTO sessions (session_id, custom_title) VALUES (?, ?) "
        "ON CONFLICT(session_id) DO UPDATE SET custom_title = ?",
        (session_id, title, title),
    )
    conn.commit()
    conn.close()

    return {"ok": True, "title": title}
```

**逐行解释**：

| 行 | 解释 |
|----|------|
| `@app.put(...)` | PUT 方法表示"更新资源"（REST 惯例） |
| `req: RenameRequest` | FastAPI 自动从请求体解析 JSON 并校验 |
| `title = req.title.strip()` | 去掉首尾空格 |
| `if not title:` | 空字符串 → 拒绝。注意只用 `strip()` 不用 `len()`，Python 里空字符串本身就是 falsy |
| `INSERT ... ON CONFLICT DO UPDATE` | SQLite 的 UPSERT。`session_id` 是 PRIMARY KEY，冲突时就更新 |
| `(session_id, title, title)` | 三个 `?` 占位符的值：INSERT 用前两个，UPDATE 用第三个 |

**为什么返回 `{"ok": True}` 而不是抛 HTTPException**：
- 这是内部 API，前端是咱自己写的
- 返回统一格式 `{ok, error}` 让前端处理更简单，不用区分 200/422/500
- 和现有的 `/sessions` 系列 API 风格一致（它们都返回简单 JSON）

## 三、⚠️ 踩坑预警

### 坑 1：chat.db 已有旧数据

如果你的 `chat.db` 已经有一些会话，改完 `init_db()` 后 sessions 表是空的。这不影响功能——`LEFT JOIN` 下 `s.custom_title` 全是 NULL，`COALESCE` 自动用子查询取第一条消息。

**验证**：启动服务后，侧边栏应该正常显示所有旧会话。

### 坑 2：SQL 里 COALESCE 的拼写

是 `COALESCE`，不是 `COALESCE`（只有一个 L）。SQLite 和 PostgreSQL 都是这个拼写。如果写错了，SQLite 报错 `no such function`。

### 坑 3：sqlite3 模块每次要 connect + close

Python 的 `sqlite3` 不是连接池——每次操作都要 `connect(DB_PATH)` 然后 `.close()`。不要在模块顶部创建全局 connection 对象——多线程访问会炸（SQLite 默认不支持跨线程共享连接）。

现在代码里每个函数都是 `connect → 操作 → close` 的模式，保持一致就好。

### 坑 4：ON CONFLICT 需要 PRIMARY KEY 或 UNIQUE 约束

`sessions` 表的 `session_id` 是 `PRIMARY KEY`，所以 `ON CONFLICT(session_id)` 能识别冲突。如果把建表语句写成 `session_id TEXT` 不加 PRIMARY KEY，UPSERT 会报错。

### 坑 5：COALESCE 的括号闭合（真实踩坑 2026-06-01）

**现象**：`GET /sessions` 返回 500，终端报 `sqlite3.OperationalError: near "as": syntax error`。

**根因**：`COUNT(*)` 和 `MAX(...)` 被错误地写在了 `COALESCE(...)` 的括号里头：

```sql
-- ❌ 错误：COALESCE 吃掉了后面所有列
COALESCE(
    s.custom_title,
    (SELECT content ...) as title,     -- as title 在 COALESCE 里面 → 语法错误！
    COUNT(*) as msg_count,             -- 也变成了 COALESCE 的参数
    MAX(m.created_at) as updated_at
)
FROM messages m ...

-- ✅ 正确：COALESCE 只有两个参数，右括号及时闭合
COALESCE(
    s.custom_title,
    (SELECT content ...)
) as title,                            -- as title 在 COALESCE 外面，是独立的列别名
COUNT(*) as msg_count,                 -- 和 COALESCE 平级的独立列
MAX(m.created_at) as updated_at
FROM messages m ...
```

**一句话理解**：`COALESCE(a, b)` 是一个函数调用——它的括号只包含参数。`as title` 是给整个 `COALESCE(...)` 结果起别名，必须写在 `COALESCE` 的右括号外面。同理 `COUNT(*)` 和 `MAX(...)` 是 SELECT 的独立列，不能写在 COALESCE 的参数列表里。

| 概念 | 一句话解释 |
|------|-----------|
| LEFT JOIN | 保留左表所有行，右表匹配不上的填 NULL |
| COALESCE(a, b) | 返回第一个非 NULL 的值 |

## 六、前端改动 — Markdown 渲染

### 6.1 为什么需要 Markdown 渲染

DeepSeek 返回的文本天然是 Markdown 格式。比如：
```markdown
当然可以！Python 的 hello world 如下：

```python
print("Hello, World!")
```

你只需要：
1. **安装 Python**（[官网下载](https://python.org)）
2. **保存代码**到 `.py` 文件
3. **运行** `python 文件名.py`
```

以前用 `bubble.textContent = text` 显示，`**粗体**` 会原样显示为星号，代码块也没有暗色背景。现在用 `bubble.innerHTML = renderMarkdown(text)`，渲染出真正的富文本。

### 6.2 核心函数：`renderMarkdown()`

```javascript
function renderMarkdown(text) {
    if (!text) return "";
    const rawHtml = marked.parse(text, { gfm: true, breaks: true });
    return DOMPurify.sanitize(rawHtml);
}
```

**流水线**：Markdown 文本 → `marked.parse()` → HTML 字符串 → `DOMPurify.sanitize()` → 安全 HTML。

**两个库的职责**：
| 库 | 干嘛的 | 例子 |
|----|--------|------|
| `marked.js` | Markdown → HTML | `**粗体**` → `<strong>粗体</strong>` |
| `DOMPurify` | 过滤 XSS 攻击 | `<script>alert(1)</script>` → 删掉 |

**为什么需要 DOMPurify**：`innerHTML` 会执行 HTML 里的 `<script>` 标签。如果 DeepSeek 被诱导返回恶意脚本，`textContent` 是安全的（当纯文本显示），但 `innerHTML` 会真的执行。DOMPurify 把 `<script>`、`onerror`、`onclick` 等危险标签/属性全过滤掉，只保留安全的格式化标签。

### 6.3 改了三处渲染

| 位置 | 原来 | 改为 |
|------|------|------|
| `addMessageBubble()` | `bubble.textContent = text` | 助手消息用 `innerHTML = renderMarkdown(text)`，用户消息保持 `textContent` |
| `sendMessage()` 流式更新 | `bubble.textContent = accumulated` | `bubble.innerHTML = renderMarkdown(accumulated)` |
| `sendMessage()` 错误提示 | `bubble.textContent = ...` | 不动，错误消息是纯文本 |

### 6.4 流式 Markdown 的特殊行为

流式过程中，Markdown 可能不完整。比如 streaming 到一半：
```
**你好世界
```
还没收到闭合的 `**`，`marked.parse()` 会把它当成普通文本渲染（不会报错）。下一个 chunk 收到闭合标记后，再次渲染就正常加粗了。这是正常的——所有主流聊天应用（Discord、Slack）都是这样。

### 6.5 新增的 CSS

给 `.message-bubble` 内部的 Markdown 元素加了样式：
- 代码块 `pre` — 暗色背景（`#1e1e1e`）+ 等宽字体
- 行内代码 `code` — 半透明灰底 + 圆角
- 引用 `blockquote` — 左侧蓝色竖线
- 表格 — 边框 + 表头灰底
- 列表、标题、链接 — 基础间距和颜色

所有样式都加了 `.message-bubble` 前缀，不会影响页面其他区域。

## 七、前端改动 — 会话重命名

### 7.1 交互逻辑

```
用户双击侧边栏标题
  → 标题进入编辑模式（contentEditable = "true"）
  → 文字自动全选，方便直接输入覆盖
  → 两种保存方式：
      - 按 Enter → 保存
      - 点击别处（失焦）→ 保存
  → 调后端 PUT /sessions/{id}/rename
  → 成功：标题更新（当前会话的话顺手更新顶部标题栏）
  → 失败/空标题：恢复原标题
```

### 7.2 核心 JS 概念：`contentEditable`

```javascript
titleDiv.contentEditable = "true";   // 让 div 像输入框一样可编辑
titleDiv.contentEditable = "false";  // 退出编辑模式
```

`contentEditable` 是浏览器内置的 HTML 属性。不需要 `<input>` 元素——直接把一个 `<div>` 变成可编辑的文本框。优点是：
- 不需要创建/销毁输入框，只要切换属性
- 编辑时保持原有样式（字体、颜色），体验无缝
- 退出编辑后它还是那个 div，不需要替换 DOM

### 7.3 两个保存触发器

```javascript
// Enter 键保存
function keyHandler(e) {
    if (e.key === "Enter") {
        e.preventDefault();  // 阻止换行（contentEditable 里 Enter 本来会换行）
        saveTitle();
    }
}
// 失焦保存（用户点了别的地方）
function blurHandler() { saveTitle(); }

titleDiv.addEventListener("blur", blurHandler);
titleDiv.addEventListener("keydown", keyHandler);
```

**为什么两种保存方式**：
- `Enter` — 键盘操作，双手不用离开键盘
- `blur`（失焦）— 直觉操作，点别的地方就是"确认"。如果只有 Enter 才能保存，用户改了标题后直接点聊天区，改动就丢了

**别忘了 removeEventListener**：绑定的事件在保存后要解绑，否则下次编辑会绑两套事件，saveTitle 被执行两次。

### 7.4 与后端的通信

```javascript
const resp = await fetch("/sessions/" + sessionId + "/rename", {
    method: "PUT",
    headers: { "Content-Type": "application/json" },
    body: JSON.stringify({ title: newTitle }),
});
const data = await resp.json();
// data = { ok: true, title: "新标题" }
// 或     { ok: false, error: "标题不能为空" }
```

后端返回 `ok: true/false`，前端根据这个判断是否成功，不用处理 HTTP 状态码。

### 7.5 CSS：编辑态的视觉反馈

```css
.session-title[contenteditable="true"] {
    background: rgba(255, 255, 255, 0.15);  /* 半透明白底 */
    outline: 1px solid #3498db;              /* 蓝色外框 */
    cursor: text;                            /* 文字光标 */
}
```

用了 CSS 属性选择器 `[contenteditable="true"]`——只匹配正在编辑状态的元素。编辑结束后 `contentEditable` 变回 `"false"`，样式自动恢复。
