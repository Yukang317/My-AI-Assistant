**日期**：2026-05-31

## 〇、为什么需要流式输出？

现在的 `/chat` 端点流程：

```
用户发消息 → 服务器调 DeepSeek → 等 3~5 秒 → 完整回复一次性返回 → 前端一次性显示
```

用户看到的是：点了发送 → "思考中..." 等几秒 → 突然冒出一大段文字。

流式输出的目标：

```
用户发消息 → 服务器调 DeepSeek(stream=True) → 每生成1-2个字就立刻发给前端 → 前端逐字显示
```

用户体验从"等半天突然出现"变成"实时打字效果"。

## 一、核心概念：SSE（Server-Sent Events）

### 1.1 SSE 是什么？

SSE 是一种**服务器向客户端单向推送数据**的 HTTP 协议。它的本质是：

> 服务器不关连接，每隔一小段时间就往客户端发一行数据，直到全部发完才关。

类比：不是"写一封长邮件一次发出去"，而是"开着电话免提，边说边让对方听到"。

### 1.2 SSE 的数据格式

极简——每行以 `data: ` 开头，后面跟 JSON，结尾用两个换行 `\n\n`：

```
data: {"content":"你好","finished":false}\n\n
data: {"content":"，我","finished":false}\n\n
data: {"content":"是AI","finished":false}\n\n
data: {"content":"","finished":true}\n\n
```

前端读到 `finished: true` 就知道流结束了。

### 1.3 SSE vs WebSocket

| | SSE | WebSocket |
|------|-----|-----------|
| 方向 | 服务器 → 客户端（单向） | 双向 |
| 协议 | 普通 HTTP | 升级为 ws:// |
| 复杂度 | 极简单 | 需要握手、心跳 |
| 断线重连 | 浏览器自动 | 需要自己写 |
| 适用场景 | **AI 打字效果**、股票推送、日志流 | 聊天、游戏、协作编辑 |

**我们就用 SSE**——AI 回复是单向的（服务器生成文字推给用户），不需要双向。

## 二、核心概念：Python 异步生成器

### 2.1 回顾：普通生成器（你已经知道的）

```python
def count_to_3():
    yield 1    # 暂停，返回 1，等下次调用
    yield 2    # 暂停，返回 2
    yield 3    # 暂停，返回 3
    # 函数结束

for num in count_to_3():
    print(num)  # 1, 2, 3 —— 一个一个出来，不是一次返回 [1,2,3]
```

**`yield` 不终止函数**——它让函数"暂停"，把值返回给调用者，等调用者下次 `next()` 时从暂停点继续。

### 2.2 异步生成器 = 生成器 + await

```python
async def stream_from_deepseek():
    stream = client.chat.completions.create(stream=True)  # 返回迭代器
    for chunk in stream:                                 # 逐块读取
        if chunk.choices[0].delta.content:
            yield chunk.choices[0].delta.content         # 逐块发送
            await asyncio.sleep(0.01)                    # 让出控制权
```

关键区别：
- **`def`** → **`async def`**：这个函数内部可以用 `await`
- **`for chunk in stream`**：OpenAI SDK 的 `stream=True` 返回一个可迭代对象，每次迭代拿到一个 token
- **`yield`**：把当前 chunk 发送给调用者（FastAPI），然后暂停等下次迭代
- **`await asyncio.sleep(0.01)`**：让出 CPU 控制权，允许其他协程跑——好习惯

### 2.3 为什么后端用 `async def`？

原 `/chat` 是同步的 `def chat()`：
```python
def chat(req):
    response = client.chat.completions.create(...)  # 阻塞：等 3 秒才返回
    return ChatResponse(...)                        # 一次性返回 JSON
```

流式版需要用 `yield` 逐块返回 → 有 `yield` 的函数是生成器 → FastAPI 需要异步生成器来支持 SSE → 必须是 `async def`。

`yield` 是关键：它让函数"返回但不结束"，可以在一个 HTTP 响应里多次发送数据。

## 三、核心概念：OpenAI SDK 的 `stream=True`

### 3.1 非流式 vs 流式

**非流式**（现在的代码）：
```python
response = client.chat.completions.create(
    model="deepseek-chat",
    messages=messages,
    temperature=0.7,
    # stream 默认 False
)
# response 是完整的 ChatCompletion 对象
# response.choices[0].message.content  → "Python是一种简洁易读的高级编程语言..."
```

**流式**：
```python
stream = client.chat.completions.create(
    model="deepseek-chat",
    messages=messages,
    temperature=0.7,
    stream=True,          # ← 关键！
)
# stream 是一个迭代器，不是一个完整对象
# 每次迭代得到一个 chunk（可能只有1-2个字符）

for chunk in stream:
    print(chunk.choices[0].delta.content)  # "Python" → "是一种" → "简洁" → ...
```

### 3.2 `delta.content` vs `message.content`

| 属性 | 非流式 (`stream=False`) | 流式 (`stream=True`) |
|------|------------------------|-------------------|
| 内容位置 | `response.choices[0].message.content` | `chunk.choices[0].delta.content` |
| 内容量 | 完整回复 | 增量文本（每次1-2个字） |
| 类型 | `str` | `str` 或 `None`（空 chunk 返回 None） |

**注意 `delta` 不是 `message`——这是最容易写错的地方！**

### 3.3 空 chunk 处理

DeepSeek 流式响应中有一些 chunk 不包含文本内容（如流开始的元数据）。需要跳过：

```python
if chunk.choices and chunk.choices[0].delta.content:
    # 有有效内容才处理
    content = chunk.choices[0].delta.content
```

## 六、实际踩坑记录 🔴

以下是在手写代码过程中真实遇到的 bug，按发现顺序排列。

### Bug 1：`NameError: name 'event' is not defined`

**现象**（终端输出）：
```
data:{"error": "name 'event' is not defined", "finished": true, ...}
```

**根因**：在 `yield` 之前根本没有定义 `event` 变量。

**错误代码**：
```python
for chunk in stream:
    if chunk.choices and chunk.choices[0].delta.content:
        content = chunk.choices[0].delta.content
        accumulated += content
        yield f"data:{json.dumps(event,ensure_ascii=False)}\n\n"  # ← event 哪来的？
```

**正确代码**：先构建 `event` dict，再用它：
```python
for chunk in stream:
    if chunk.choices and chunk.choices[0].delta.content:
        content = chunk.choices[0].delta.content
        accumulated += content

        event = {                              # ← 先定义
            "content": content,
            "finished": False,
            "created_at": datetime.datetime.now().isoformat(),
        }
        yield f"data: {json.dumps(event, ensure_ascii=False)}\n\n"  # ← 再用
```

**教训**：`yield` 语句中引用的变量必须在同一作用域的前面定义。这和 `return` 不一样——`return` 返回之后函数就结束了，`yield` 还会继续，所以变量的定义时机和位置很重要。

---

### Bug 2：`Ture` 拼写错误

**代码**：
```python
end_event = {
    "content": "",
    "finished":Ture,    # ← Python 里没有 Ture 这个关键字
    ...
}
```

**正确**：`True`（T-r-u-e，不是 T-u-r-e）

**为什么第一次测试没报这个错？** 因为 Bug 1 在前面就炸了，代码进了 `except` 块，根本没执行到 Bug 2 的位置。**bug 会互相掩盖**——这是为什么修完一个 bug 之后要重新测试。

---

### Bug 3：`response_model=ChatResponse` 与 `StreamingResponse` 冲突

**代码**：
```python
@app.post("/chat", response_model=ChatResponse)  # ← 声明返回类型是 ChatResponse
async def chat(req: ChatRequest):
    ...
    if req.stream:
        return StreamingResponse(...)  # ← 但这里返回的是 StreamingResponse
```

**问题**：FastAPI 看到 `response_model=ChatResponse`，会在返回时尝试把返回值序列化为 `ChatResponse`，但 `StreamingResponse` 是原始 HTTP 响应对象，不能序列化为 Pydantic 模型。去掉 `response_model` 即可，FastAPI 会自动判断返回类型。

**正确**：
```python
@app.post("/chat")  # ← 不指定 response_model
async def chat(req: ChatRequest):
```

---

### Bug 4：`end_event` 被锁在 `if accumulated:` 里面

**代码**：
```python
if accumulated:
    save_message(session_id, "assistant", accumulated)
    end_event = { ... }
    yield f"data:..."  # ← 如果 accumulated 为空，永远不会执行
```

**问题**：如果 DeepSeek 返回空内容（虽然罕见，但可能发生），`accumulated` 是空字符串，`end_event` 永远不会发送。前端会一直等待 `finished: true`，页面上的"思考中..."永远不会消失。

**正确**：把 `end_event` 的发送放在 `if` 外面：
```python
if accumulated:
    save_message(session_id, "assistant", accumulated)

# end_event 不管有没有内容都要发
end_event = { ... }
yield f"data:..."
```

**教训**：控制信号（如"流结束"）和业务数据（如"保存回复"）的生命周期不同。数据可能为空，但信号必须发送。

---

### Bug 5：SSE 格式缺空格

**代码**：`f"data:{json.dumps(...)}"` → 少了一个空格  
**正确**：`f"data: {json.dumps(...)}"` — SSE 规范要求 `data: ` 后面有空格

虽然大部分前端解析器能容错，但规范的格式能避免未来换前端框架时踩坑。

---

### Bug 6：`json.dumps` 参数间缺空格

**代码**：`json.dumps(event,ensure_ascii=False)`  
**正确**：`json.dumps(event, ensure_ascii=False)` — PEP 8 要求逗号后面加空格

这个不影响运行，但 Claude review 时被指出来了。

## 八、最终代码快照

```python
async def generate_stream_response(session_id: str, messages: list[dict], model_name: str):
    """流式调用 DeepSeek，逐块产出 SSE 事件字符串"""
    try:
        stream = client.chat.completions.create(
            model=model_name,
            messages=messages,
            temperature=0.7,
            stream=True,
        )

        accumulated = ""
        for chunk in stream:
            if chunk.choices and chunk.choices[0].delta.content:
                content = chunk.choices[0].delta.content
                accumulated += content

                event = {
                    "content": content,
                    "finished": False,
                    "created_at": datetime.datetime.now().isoformat(),
                }
                yield f"data: {json.dumps(event, ensure_ascii=False)}\n\n"
                await asyncio.sleep(0.01)

        if accumulated:
            save_message(session_id, "assistant", accumulated)

        end_event = {
            "content": "",
            "finished": True,
            "created_at": datetime.datetime.now().isoformat(),
        }
        yield f"data: {json.dumps(end_event, ensure_ascii=False)}\n\n"

    except Exception as e:
        error_event = {
            "error": str(e),
            "finished": True,
            "created_at": datetime.datetime.now().isoformat(),
        }
        yield f"data: {json.dumps(error_event, ensure_ascii=False)}\n\n"
```

## 九、本次学到的新概念

| 概念 | 一句话解释 |
|------|-----------|
| **SSE** | 服务器不关连接，用 `data: {json}\n\n` 格式逐条推送数据 |
| **`async def` + `yield`** | 异步生成器——函数可以"暂停→返回值→继续"，适合流式场景 |
| **`stream=True`** | OpenAI SDK 参数，让 API 返回迭代器而非完整响应 |
| **`delta.content` vs `message.content`** | 流式用 `delta.content`（增量），非流式用 `message.content`（全量） |
| **`StreamingResponse`** | FastAPI 内置的 SSE 响应类，`media_type="text/event-stream"` |
| **`ensure_ascii=False`** | `json.dumps()` 参数，保留中文原文不转义 |
| **Bug 互相掩盖** | 前面的 bug 让代码进 except，后面的 bug 永远执行不到——修一个测一次 |
| **控制信号 vs 业务数据** | `end_event`（流结束信号）必须无条件发送，`save_message`（存数据）只发生在有内容时 |

### Q1：为什么把 `generate_stream_response` 放在数据库操作区？

**答**：放那里只是因为方便（在 `save_message()` 和 `list_sessions()` 之间），不是一个好的设计决策。

`generate_stream_response` 本质是一个 **API 辅助函数**（调用 LLM + 格式化 SSE + 顺带存数据库），不是数据库操作函数。更合理的位置应该是在 API 端点区，放在 `@app.post("/chat")` 的紧上面，这样调用者和被调用者离得近，读代码的人不用跳来跳去。

当前文件分区可以优化为：
```
# 数据模型区    → ChatRequest / ChatResponse / SessionInfo
# 数据库操作区  → init_db / load_messages / save_message / list_sessions / delete_session
# API 辅助函数  → generate_stream_response   ← 应该放这里
# API 端点区    → @app.get("/") / @app.post("/chat") / @app.post("/sessions") ...
```

### Q2：`if chunk.choices and chunk.choices[0].delta.content:` 是什么意思？

**答**：逐层检查 chunk 内部是否包含有效文本，利用 Python 的**短路求值**防止访问不存在的属性。

流式响应的每个 chunk 结构是一层套一层的：

```
chunk                           # ChatCompletionChunk 对象
  .choices                      # list[Choice] — 可能为空列表 []
    [0]                         # Choice 对象 — 列表非空才能取
      .delta                    # Delta 对象 — 流式才有 delta（非流式是 message）
        .content                # str | None — 有内容 / None / 空字符串
```

这行 `if` 的执行顺序：

```
第 1 步：chunk.choices 是 []（空列表）吗？
    → 是 → 整个 if 为 False，跳过（[0] 不会执行，避免了 IndexError）
    → 否 → 进入第 2 步

第 2 步：chunk.choices[0].delta.content 是 None 或 "" 吗？
    → 是 → 整个 if 为 False，跳过（空 chunk，如流开始的元数据）
    → 否 → 整个 if 为 True，进入循环体处理这个 chunk
```

**为什么需要这个检查？** DeepSeek 流式响应中约 5-10% 的 chunk 是空包：
- 第一个 chunk 有时只有 `role: "assistant"` 没有 content
- 最后一个 chunk 有时是 `finish_reason: "stop"` 没有 content
- 网络波动时偶发空 choices

不检查的后果：
- `choices` 为空 → `IndexError`（列表索引越界）
- `delta.content` 为 None → 拼接时 `TypeError`（`"" += None`）

### Q3：去掉 `response_model=ChatResponse`——"两种类型"是哪两种？

**答**：同一个 `/chat` 端点，根据 `stream` 字段返回完全不同的两类 HTTP 响应。

| | 非流式 | 流式 |
|------|--------|------|
| 触发条件 | `stream: false`（默认） | `stream: true` |
| 返回类型 | `ChatResponse`（Pydantic 模型） | `StreamingResponse`（原始 HTTP 响应对象） |
| Content-Type | `application/json` | `text/event-stream` |
| 响应结构 | `{"created_at":"...","reply":"..."}` | `data: {...}\n\ndata: {...}\n\n` |
| 连接行为 | 一次性返回，返回后关闭 | 持续推送，流结束后才关闭 |

`response_model=ChatResponse` 的作用是告诉 FastAPI"不管函数 return 了什么，最终都按 `ChatResponse` 来序列化和校验"。但 `StreamingResponse` 已经是最终的 HTTP 响应对象——它不需要也不应该再被 `ChatResponse` 序列化。两者冲突，所以必须去掉。

去掉后 FastAPI 的行为：
- 看到 `return ChatResponse(...)` → Pydantic 模型，自动转 JSON
- 看到 `return StreamingResponse(...)` → 原始 HTTP 响应，直接返回不做任何处理
