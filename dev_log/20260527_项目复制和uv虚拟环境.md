## 1. 背景与目标
为了在不影响原销售智能体项目（`sagt_202601`）运行的前提下，进行向“通用个人 AI 助理”的转型开发，需要在同一台云服务器上创建一个完全一致的项目副本 `assistant`。在此过程中，深入研究了 Linux 磁盘存储查看方法及 Python 现代化工具 `uv` 的运行机制。

## 2. 磁盘存储空间分析

### 2.1 常用查看命令
*   **`df -h`**：查看文件系统整体容量。重点关注 `Avail`（可用空间）和 `Use%`（使用率）。
    *   *实例*：服务器总容量 79G，已用 27G，剩余 49G，空间充足。
*   **`du -sh <目录>`**：查看指定目录的实际占用大小。
    *   *实例*：原项目 `sagt_202601` 占 302M，另一项目 `mildoc_202601` 占 6.4G。
*   **`find / -type f -size +500M`**：查找系统中大于 500M 的文件，定位“消失”的空间。

### 2.2 空间占用疑点排查
发现两个项目代码仅占约 6.7G，但系统显示已用 27G。经分析，剩余空间主要被以下部分占用：
1.  **Docker 资源**：镜像、容器、构建缓存及数据卷（通常占用最大）。
2.  **系统目录**：`/usr`（程序库）、`/var`（日志与缓存）。
3.  **隐藏文件**：如 `.langgraph_api` 检查点、Python 编译缓存 `__pycache__` 等。

## 3. 项目复制实践

### 3.1 复制过程中的空间异常
初次使用 `cp -r` 复制项目后，发现副本 `assistant` (421M) 比原项目 `sagt_202601` (302M) 大了约 119M。
*   **原因分析**：Linux 文件系统中的硬链接在普通复制时会断开，导致原本共享 inode 的文件变成了独立副本，从而增加了物理存储占用。此外，`.venv` 虚拟环境中的大量小文件加剧了这一现象。

### 3.2 推荐的复制方案
为了保证副本的轻量化和独立性，建议采用 `rsync` 排除非必要文件：
```bash
rsync -av --exclude='.venv' --exclude='__pycache__' --exclude='*.log' \
    --exclude='.langgraph_api' sagt_202601/ assistant/
```
*   **优势**：不复制庞大的依赖包和缓存，副本体积更小，且避免了旧环境的潜在冲突。

## 4. UV 与虚拟环境机制深度解析

### 4.1 核心角色定义
*   **UV (工具)**：全局安装的超高速 Python 包管理器。它负责读取配置、下载包并管理版本。
*   **虚拟环境 (.venv)**：每个子项目独立的“房间”。它包含独立的 Python 解释器和第三方库，确保不同项目间的依赖互不干扰。
*   **配置文件**：
    *   `pyproject.toml`：需求清单（声明需要哪些库）。
    *   `uv.lock`：采购单（锁定精确的版本号，保证环境一致性）。

### 4.2 项目中的虚拟环境分布
当前多模块项目结构中，每个子模块都拥有独立的虚拟环境：
1.  `sagt_agent/.venv` (约 251M)：AI 核心逻辑依赖。
2.  `sagt_sidebar/.venv` (约 84M)：FastAPI 网关依赖。
3.  `sagt_admin/.venv`：Flask 管理后台依赖。
4.  `sagt_client/.venv`：测试客户端依赖。

### 4.3 UV 的工作流
1.  **全局安装**：服务器上只需安装一次 `uv`。
2.  **创建环境**：在项目目录下执行 `uv venv .venv`。
3.  **同步依赖**：执行 `uv sync`，UV 会根据 `pyproject.toml` 和 `uv.lock` 将包安装到当前的 `.venv` 中。
4.  **运行项目**：使用 `uv run python app.py`，无需手动 `source activate`，UV 会自动调用虚拟环境中的解释器。
