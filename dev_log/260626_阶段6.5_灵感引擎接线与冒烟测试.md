|------|---------------------|-------------------|
| 搜索策略 | 单工具、单查询、精准命中 | **多角度并发搜索**，跨领域类比 |
| 回复风格 | 基于事实的客观回答 | "I think... and..." 叠加延伸 |
| 质量检查 | 不需要 | **证据门**自检（跨域连接 + 引用 + 风格） |
| PG 保存 | result_synthesis 做 | inspire 路径不走 result_synthesis，**converge 补做** |

所以专属管线 = `inspire_diverge`（多角度并发搜索）→ `inspire_converge`（专属 prompt 合成 + 证据门检查 + 补写 PG）。

### 📌 知识点 1：Facade 模式 — 语义化打点封装

**在哪**：`agent/trace.py` L288-326

`trace_inspire_diverge()` 和 `trace_inspire_converge()` 这俩函数，本质上都是在调底层的 `log_event()`——区别只是传不同的 `event` 字符串和不同的 data 字段。

**为什么不直接在 main_graph 里调 log_event？**

因为这是 Facade（外观）模式——给底层通用函数穿一件语义化的"外衣"：

```python
# ❌ 直接调：调用方需要知道 event 字符串拼写、data 字段名
log_event(trace_id, session_id, "inspire_diverge",
          queries=queries, evidence_count=len(evidence), latency_ms=elapsed_ms)

# ✅ 通过语义化包装：IDE 自动补全参数名，写错参数名会报 TypeError
trace_inspire_diverge(
    trace_id, session_id,
    queries=queries,
    evidence_count=len(evidence),
    latency_ms=elapsed_ms,
)
```

好处：
1. **调用方零认知负担**——不需要知道 event 名叫 `"inspire_diverge"` 还是 `"inspire_divergent"`
2. **参数强约束**——`*` 强制关键字参数，`queries` 是 `list[str]` 类型，IDE 能检查
3. **改 event 名只改一处**——将来如果 event 改叫 `"diverge_search"`，只改 facade 函数体，调用方不动

类比：就像餐厅菜单（facade）vs 直接跟厨师说原料配比（log_event）。顾客只需要说"麻婆豆腐"，不需要知道花椒几克、豆腐几块。

### 📌 知识点 2：Bare Star (`*`) 参数 — 强制关键字传参

**在哪**：`trace.py` L111 的 `log_event()`，以及所有 facade 函数（L163、L178、L189、L208、L293、L312）

```python
def log_event(
    trace_id: str,
    session_id: str,
    event: str,
    *,                  # ← 这个单独的星号
    latency_ms: Optional[int] = None,
    **data: Any,
) -> None:
```

**`*` 在这里是什么意思？**

它不是 `*args`（收集多余位置参数），而是一个**分隔符**——`*` 之前的参数可以按位置传，`*` 之后的参数**必须用关键字形式**传。

```python
# ✅ 正确：* 后面的参数必须写名字
log_event("tid", "sid", "run_start", latency_ms=100, question="你好")

# ❌ 错误：* 后面的参数不能按位置传
log_event("tid", "sid", "run_start", 100, "你好")
# TypeError: log_event() takes 3 positional arguments but 5 were given
```

**为什么这样设计？**

`trace_id`、`session_id`、`event` 这三个是"必定有的身份信息"——每个打点都要写，按位置传最自然。但 `latency_ms` 和各种 data 字段（`intent`、`tool`、`reason`...）是"可选的业务信息"——不同事件带不同字段，如果按位置传，调用方根本记不住第 4 个参数是 latency 还是 intent。

**这和 `trace_route_decision` 等 facade 函数的风格一致**——它们也在自己的签名里用 `*`，把所有业务字段强制为关键字参数：

```python
def trace_route_decision(
    trace_id, session_id,
    *,                   # ← 统一风格：身份信息按位置，业务信息必须写名字
    intent, target_tool, reason, latency_ms,
):
```

类比：去银行办业务——身份证号和姓名是必须的（位置参数），但"取款金额""转账账号""是否加急"这些各不相同，柜员会问"您要取多少？"（关键字参数）而不是让你按固定顺序报。

### 📌 知识点 3：LangGraph 适配节点模式 — 桥接业务函数和图节点

**在哪**：`main_graph.py` L325-367

LangGraph 注册节点时，只认这种签名：

```python
def some_node(state: MainState) -> dict:
    ...
```

但我们的业务函数签名各不相同：

```python
# route_intent 需要 model + tools（不只是 state）
def route_intent(question, model, tools) -> RouteResult: ...

# inspire_diverge 需要 model（state 里没有）
def inspire_diverge(state, model) -> dict: ...

# memory_update 需要 model
def memory_update(state, model) -> dict: ...
```

**解决方案：适配节点（Adapter Node）**

在 `build_graph()` 里写**闭包适配函数**——把 LangGraph 的 `(state) -> dict` 签名"翻译"成业务函数的实际调用：

```python
def build_graph() -> CompiledStateGraph:
    graph = StateGraph(MainState)

    # 适配层：闭包捕获外层的 get_llm()，注入 model
    def intent_node(state: MainState) -> dict:
        """state -> dict，但内部调 route_intent(state 字段, model, tools)"""
        question = state.get(StateField.USER_QUESTION, "")
        model = get_llm()        # ← 从外层闭包拿
        tools = list_tools()     # ← 从外层闭包拿
        result = route_intent(question, model, tools)
        return {StateField.INTENT: result.intent, ...}

    def inspire_diverge_node(state: MainState) -> dict:
        """桥接：(state) -> dict  ⟹  inspire_diverge(state, model)"""
        return inspire_diverge(state, get_llm())

    # 注册时用适配节点
    graph.add_node("intent_route", intent_node)
    graph.add_node("inspire_diverge", inspire_diverge_node)
```

**为什么用闭包而不是把 model 写进 State？**

1. **model 不是"状态"**——LLM 实例在整次运行中不变，放 State 里会被 LangGraph 的 checkpoint 序列化（搞笑且浪费）
2. **依赖注入比 State 传递更干净**——适配节点是 glue code（胶水代码），职责就是"把东西接在一起"，不应该污染业务逻辑
3. **和 sagt_agent 的模式一致**——`sagt_agent` 也是用 `config[ConfigurableField.configurable]` 传依赖，不走 State

类比：适配节点就像电源转换插头——中国插座（LangGraph 的 `state -> dict`）和美国电器（业务函数的 `state, model -> dict`）之间需要一个转换器，这个转换器本身不"做"任何事情，只是让两边能对接。

### 📌 知识点 4：Prompt 职业设定 — 用现实中存在的职业比抽象描述更有效

**在哪**：`inspire_converge.py` L31-47（CONVERGE_SYSTEM_PROMPT）和 L50-70（EVIDENCE_GATE_PROMPT）

这是本次最有价值的 prompt engineering 洞察。

**原来的设定（泛化）：**
- 收敛合成角色：「灵感对话伙伴」
- 证据门角色：「质量检查员」

**改进后的设定（具体职业）：**
- 收敛合成角色：「**科学作家**（Science Writer）——类似 Steven Johnson 或万维钢」
- 证据门角色：「**学术期刊审稿人**（Peer Reviewer）——类似 Nature 或 Science 的同行评审专家」

**为什么具体职业比抽象描述更有效？**

1. **LLM 的训练数据里有大量该职业的写作样本**。Steven Johnson 写过《Where Good Ideas Come From》，万维钢在得到 App 有大量"串联跨领域知识"的文章——模型在预训练时读过这些文本，知道「科学作家」应该怎么说话。

2. **具体职业自带风格约束**。说"科学作家"，模型自动知道：用平实中文、不做学术八股、要有叙事感、跨领域连接是核心卖点。这些不需要在 prompt 里逐条列出。

3. **审稿人角色天然适合做 gate check**。「判断论证是否扎实」比「检查质量」更具体——审稿人不会因为"不同意观点"而拒稿（同行评审的规范），这恰好对应了我们"不否定说教"的约束。

**一个可以推广的 prompt 技巧**：

> 当你需要 LLM 扮演某个角色时，问自己：**现实中有没有一种职业，它的日常工作就是你希望 LLM 做的事？** 如果有，直接用那个职业名 + 一个该职业的代表人物/机构，效果几乎一定比你自己编的角色描述好。

类比：你想让朋友帮忙审一篇文章——说"你帮我看看质量"（抽象）远不如说"你假装你是 Nature 的审稿人，只看论证结构不看观点"（具体职业场景）有效。

### 📌 知识点 5：专属管线 vs 通用管线 — 什么时候该分叉

**在哪**：`main_graph.py` L388-406（条件路由 + 边连接）

```
通用路径：inspire → tool_execute → result_synthesis → memory_update → END
专属路径：inspire → inspire_diverge → inspire_converge → memory_update → END
```

**决策逻辑：什么时候值得为一种 intent 建专属管线？**

三条判断标准（满足两条就该分叉）：

| 标准 | inspire 是否满足 |
|------|-----------------|
| 搜索策略不同？ | ✅ 多角度并发 vs 单工具精准 |
| 回复 prompt 不同？ | ✅ "I think... and..." 风格 vs 事实陈述 |
| 后处理不同？ | ✅ 需要证据门自检 vs 不需要 |

如果三条全中——就像 `inspire` 这样——那专属管线是合理的。如果只中一条（比如只是 prompt 不同），在 `result_synthesis` 里根据 intent 选不同 prompt 就够了，不值得分叉。

**管道分叉的代价**：
- `main_graph.py` 多两条边、两个节点（约 15 行代码）
- 需要理解两条路径的开发者心智负担
- `inspire_converge` 需要补做 PG 消息保存（因为不经过 `result_synthesis`）

**收益**：
- 每个节点职责单一，不会被 if-else 分支撑爆
- 灵感引擎的专属逻辑（证据门、并行搜索）不会影响通用路径

类比：超市结账——大多数人走普通收银台（通用管线），但买烟酒的走专门柜台（专属管线），因为要查身份证。让普通柜台也查身份证会增加所有人的等待时间。

### 📌 知识点 6：PG 消息保存的职责转移

**在哪**：`inspire_converge.py` L257-262

```python
# ⑤ 保存消息到 PG（result_synthesis 的职责，inspire 路径在此补做）
try:
    save_message(session_id, "user", question)
    save_message(session_id, "assistant", draft)
except Exception as e:
    logger.error("PG 消息保存失败: %s", e)
```

**为什么需要在 inspire_converge 里补做？**

通用路径的流程是 `result_synthesis → memory_update`，PG 消息保存由 `result_synthesis` 负责。但 `inspire` 路径 **跳过了 `result_synthesis`**——它走的是 `inspire_converge` 直接到 `memory_update`。

如果 `inspire_converge` 不做 PG 保存，那所有 inspire 对话都不会写入 `messages` 表——聊天记录里会缺一块。

**设计取舍**：

| 方案 | 做法 | 问题 |
|------|------|------|
| 在 converge 里做 | ✅ 当前方案 | 保存逻辑分散在两处 |
| 让 inspire 也走 result_synthesis | 在 result_synthesis 里按 intent 选 prompt | result_synthesis 会变臃肿；而且 inspire 的专属 prompt + 证据门逻辑无处安放 |
| 把保存提升到 memory_update | memory_update 统一保存 | memory_update 不应该知道"消息"的概念——它只管记忆文件 |

最终选在 converge 里做，并在注释里明确标注"这是 result_synthesis 的职责"。**这是"管道分叉"的代价之一**——被跳过的节点的职责，需要下游节点补上。

类比：你去办事大厅，正常流程是 A窗口→B窗口→C窗口。但你走绿色通道时跳过了B窗口（result_synthesis），那B窗口负责的盖章就得在C窗口或其他地方补盖。

### 📌 知识点 7：多角度查询 — 从"搜索一个问题"到"为问题生成多个搜索角度"

**在哪**：`inspire_diverge.py` L46-64（DIVERGE_QUERY_PROMPT）和 L106-138（`generate_diverge_queries`）

这是灵感引擎和普通工具调用的核心区别之一：

```
普通工具调用：用户问题 → 直接用问题搜 → 返回结果
灵感发散：   用户问题 → LLM 生成 N 条不同角度查询 → 每条 × M 个工具并发搜索 → 收集全部结果
```

**为什么需要多角度？**

灵感类问题的本质是"寻找启发"，而不是"找答案"。比如用户问"从认知科学的角度，记忆提取有什么可以借鉴到个人知识管理的？"

- 只搜原问题 → 可能返回一堆不相关的认知科学论文
- 多角度拆解 → "记忆提取 RAG 系统启发"、"间隔重复 知识管理"、"认知负荷 信息组织"、"跨界类比 记忆宫殿 数字工具" → 每个角度往不同方向探索，结果覆盖面更广

**DIVERGE_QUERY_PROMPT 的设计要点**：

```
角度示例：同域深入 / 跨域类比 / 反直觉视角 / 实践案例 / 学术理论
```

这五个角度不是随便列的——它们覆盖了灵感类问题的主要"启发来源"：
- **同域深入**：在自己的领域里往深挖
- **跨域类比**：从其他领域找类比（最有价值的灵感来源）
- **反直觉视角**：挑战默认假设，往往产生意外发现
- **实践案例**：真实的、可操作的例子
- **学术理论**：背后的原理，提供深度

### 📌 知识点 8：证据门（Evidence Gate）— LLM 自检输出质量

**在哪**：`inspire_converge.py` L50-70（EVIDENCE_GATE_PROMPT）和 L154-198（`check_evidence_gate`）

```
流程：LLM 生成回复 → 另一个 LLM 调用检查回复质量 → 返回 passed/reason
```

**为什么叫"证据门"？**

这是一个"门控"（gate）模式——回复必须先通过质量检查，不通过则标记 `gate_passed=False`。Phase 1 的策略是"不通过也仍然输出，只记录"（软门），Phase 2 会升级为"不通过时触发补搜"（硬门）。

**证据门的三个标准**：

1. **有至少 1 个跨领域连接**——防止 LLM 只在单一领域里打转
2. **有至少 1 个引用检索证据或明确标注"基于已有知识延伸"**——防止凭空编造
3. **风格是叠加延伸，不是否定说教**——我们的核心风格约束

**为什么用独立的 LLM 调用来做 gate，而不是在 prompt 里加一句"请自我检查"？**

因为"自我检查"在同一个 context window 里效果很差——LLM 生成回复时已经"尽力了"，让它同时检查自己的回复就像让学生给自己的试卷打分。独立的 LLM 调用（不同的系统 prompt、不同的角色设定）相当于找另一个人来审稿，更可能发现问题。

**类比**：写完论文自己检查一遍 vs 送给同行评审（peer review）。前者容易漏，后者是独立的、结构化的审查。

### 📌 知识点 9：ThreadPoolExecutor + as_completed — 并行搜索编排

**在哪**：`inspire_diverge.py` L179-253（`execute_parallel_searches`）

3 条 query × 2 个工具 = 6 个独立搜索任务。如果串行执行，每个 500ms 就是 3 秒。用线程池并行，总耗时 = 最慢那个任务的时间（~500ms）。

```python
with ThreadPoolExecutor(max_workers=6) as executor:
    # 1. 提交所有任务，拿到 Future 对象
    future_map = {
        executor.submit(_run_single_search, tool, query, ctx): (tool, query)
        for tool, query in tasks
    }

    # 2. as_completed：谁先完成就先处理谁
    for future in as_completed(future_map):
        tool, query = future_map[future]
        result = future.result()  # 阻塞等这一个完成
        results.append(result)
```

**为什么用线程池而不是 asyncio？**

因为工具函数（`rag_search`、`exa_search`）是**同步阻塞**的 HTTP 调用——它们在等网络响应时，Python 线程会被 GIL 释放，其他线程可以跑。这种情况用 `ThreadPoolExecutor` 最简单，不需要把所有工具改成 `async def`。

**as_completed vs 等所有完成**：

`as_completed` 的优点是**先完成的先处理**——不需要等最慢的那个任务才统一处理。对于打 trace 日志和写 artifacts 来说这差别不大，但如果后续要在 converge 里"流式"地逐步使用先到的证据，这个模式就很重要。

类比：在 6 个窗口同时排队（submit），不按编号顺序叫号，而是哪个窗口先办完就先叫你（as_completed）。

### Step 4：`agent/trace.py`（L288-326，新增 39 行）

两个 facade 函数，给底层 `log_event()` 穿上灵感引擎的语义外衣：

```python
def trace_inspire_diverge(
    trace_id: str, session_id: str,
    *,                          # 业务字段强制关键字
    queries: list[str],         # 多角度查询列表
    evidence_count: int,        # 收集到的证据条数
    latency_ms: int,            # 发散阶段总耗时
) -> None:
    """灵感引擎发散：记录多角度查询和检索证据。"""
    log_event(trace_id, session_id, "inspire_diverge",
              queries=queries, evidence_count=evidence_count, latency_ms=latency_ms)

def trace_inspire_converge(
    trace_id: str, session_id: str,
    *,
    gate_passed: bool,          # 证据门是否通过
    gate_reason: str,           # 通过/不通过原因
    latency_ms: int,
) -> None:
    """灵感引擎收敛：记录证据门结果和耗时。"""
    log_event(trace_id, session_id, "inspire_converge",
```
