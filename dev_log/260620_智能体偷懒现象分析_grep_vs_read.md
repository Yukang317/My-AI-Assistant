# 智能体"偷懒"现象深度分析 — grep 统计 vs 逐文件阅读

**日期**：2026-06-20
**触发**：用户追问——Claude 智能体配 DeepSeek v4 读代码时只读 `TODO(human)` 注释不读实际代码，是模型问题还是策略问题？
**结论**：主要原因是 Claude Code 的**智能体策略**（~70%），次要原因是 LLM 模型特性（~30%）。

---

## 一、智能体策略层面（主要原因 ~70%）

### 1.1 工具选择的"效率陷阱"

智能体在面对"统计项目状态"这种任务时，天然倾向于高效工具：

```bash
grep -c "TODO(human)" rag/    # ← 一行命令，毫秒级返回数字
```

而不是逐一 Read 文件。因为 Claude Code 的智能体在决策时遵循"最小成本原则"——能用 grep 就不用 Read，能用搜索就不用遍历。这在查找函数定义、查引用等场景是正确的，但在"判断功能是否实现"这种需要语义理解的任务上是灾难性的。

**核心矛盾**：工具效率 ≠ 信息准确性。grep 告诉你"这文件有 5 个 TODO 注释"，但完全无法告诉你"这 5 个 TODO 下面的代码已经完整实现了"。

### 1.2 上下文窗口的"经济压力"

| 方法 | 消耗的上下文 | 信息质量 |
|------|------------|---------|
| `grep -c "TODO(human)"` | ~200 tokens | 只有数字，没有语义 |
| 逐文件 Read（14 个文件 × ~270行/文件） | ~50,000+ tokens | 完整的语义理解 |

智能体天然倾向于前者，因为上下文窗口有限（即使 128K，用多了推理质量也会下降），更多上下文 = 更慢响应 + 更高 API 成本。系统提示中隐含了"高效利用上下文"的指令。

### 1.3 Explore Agent 的设计特性

Explore Agent 文档明确写了：**"It reads excerpts rather than whole files, so it locates code; it doesn't review or audit it."** —— 它定位代码位置但不审查代码质量。当用 Explore Agent 去"了解项目状态"时，它故意只读摘要和关键行——这是功能（为了速度），不是 bug。

---

## 二、LLM 模型层面（次要原因 ~30%）

### 2.1 DeepSeek v4 的 MoE 注意力模式

DeepSeek 系列使用 MoE（Mixture of Experts）架构：

- **稀疏激活**：每个 token 只激活部分专家网络，不是全参数参与
- **对注释的注意力可能高于代码**：注释是自然语言，与预训练语料分布更接近，模型处理注释时"更舒服"
- **长上下文的注意力衰减**：当 Read 返回几百行代码时，模型对文件开头和结尾的注意力更高，对中间大段实现代码的注意力可能衰减

### 2.2 所有 LLM 的共性问题：信号显著性偏差

- `# TODO(human)` 是**高显著性信号**——它看起来像指令、像待办事项
- 实际的实现代码是**低显著性信号**——连续的常规代码，没有特殊标记
- LLM 扫描长文本时会优先提取高显著性信号，形成"摘要判断"——"这个文件还有很多 TODO"——而忽略了下面的实际代码

类比人类：快速浏览一本书时，会注意到所有高亮标记和批注，但可能跳过了正文内容。

---

## 三、两者的交互放大效应

```
智能体策略选择 grep（只返回标记行）
    ↓
模型只看到 TODO(human) 的行和行号
    ↓
模型基于不完整信息推断："这些函数还没实现"
    ↓
推断结果写入了上次会话成果.md
    ↓
后续会话读取这个文档，形成错误的"事实锚定"
```

这是**信息级联（Information Cascade）**：第一次低质量判断写入文档，后续会话把它当作事实，进一步放大错误。

---

## 四、权重分解

| 维度 | 原因 | 占比 |
|------|------|------|
| 智能体策略 | 工具选择偏向效率（grep 而非 Read），Explore Agent 只读摘要 | ~40% |
| 智能体策略 | 上下文窗口经济压力，节省 token | ~30% |
| LLM 模型 | MoE 稀疏激活 + 对注释的信号显著性偏差 | ~20% |
| LLM 模型 | 长上下文注意力衰减 | ~10% |

**核心答案**：主要是 Claude Code 智能体策略的"效率优先"设计导致的（工具选择 + 上下文经济），DeepSeek v4 的模型特性起到了放大作用但不是根本原因。

---

## 五、实践建议（已落实到 CLAUDE.md）

1. **禁止用 grep 统计代替代码审查**：判断功能是否实现，唯一的方法是逐文件 Read 完整内容
2. **善用智能体类型**：Explore Agent 定位代码，但判断实现状态必须自行 Read
3. **Mode E 特殊处理**：读到 `# TODO(human):` 后必须往下读确认是否有实现代码，只有 `pass`/`raise NotImplementedError`/空行才算未实现
4. **信任用户记忆**：文档说"未实现"但用户记得跑通过 → 优先核实代码
