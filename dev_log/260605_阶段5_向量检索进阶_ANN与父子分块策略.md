## 📌 重点内容

### 一、ANN（Approximate Nearest Neighbor）近似最近邻搜索

#### 核心概念：牺牲一点点精度，换来巨大的速度

**精确搜索（KNN）**：
- 做法：把查询向量和**所有**向量逐一计算距离，找出最近的 top_k 个
- 复杂度：O(N) 次距离计算（N 是向量总数）
- 问题：如果 N=1000 万，一次搜索就要算 1000 万次距离，太慢了（几百毫秒到几秒）

**近似搜索（ANN）**：
- 做法：提前对向量建索引（如 IVF、HNSW、PQ），搜索时**只探查一部分候选**
- 结果：**99% 的情况下能找到真正的最近邻**，但速度提升 100 倍（毫秒级）
- 本质：用"可能漏掉极个别相关向量"的微小代价，换取实时响应的能力

#### 形象比喻

```
KNN = 你在图书馆里一本一本地翻所有书来找"关于AI的书"
ANN = 你先看"计算机"书架，再快速扫一眼旁边的"数学"书架，大概率就找到了
```

#### 为什么 RAG 必须用 ANN？

你的向量库可能有几十万甚至上亿条文本块：
- ❌ 如果用精确 KNN：单次查询几百毫秒，用户会抱怨卡顿；多个并发请求直接打爆 CPU/GPU
- ✅ 使用 ANN 索引（如 Milvus 里的 IVF_FLAT）：搜索时间降到 **10 毫秒以内**，召回率保持在 **95% 以上**

对于 RAG 来说，这个召回损失可以接受（因为后续 LLM 对漏掉一两个不太相关块不敏感）。

#### 代码中的体现

```python
search_params = {"nprobe": Config.MILVUS_SEARCH_NPROBE}
results = self.client.search(
    anns_field="content_vector",   # 指定在哪个向量字段上做 ANN
    search_params=search_params,   # {"nprobe": 8} 意味着只看 8 个簇
    ...
)
```

- **`anns_field`**：告诉 Milvus 这个字段里的向量需要做 ANN 搜索
- **`search_params`** 里的 **`nprobe`** 是 IVF 索引特有的参数：
  - `nprobe=1`：只查距离查询点最近的那 1 个簇 → 极快，但可能漏掉相关向量
  - `nprobe=10`：查最近 10 个簇 → 更准，但稍慢
  - 典型值：8～64，可以根据 QPS 要求调整

#### 常见 ANN 算法对比

| 算法 | 原理 | 速度 | 召回率 | 内存 | 适用场景 |
|------|------|------|--------|------|----------|
| **IVF_FLAT** | 先 K-means 分簇，再暴力搜几个簇 | 快 | 高 | 高（存原始向量） | 百万级，精度优先 |
| **IVF_SQ8** | 同上，但向量用 8bit 标量量化压缩 | 快 | 中高 | 中 | 内存紧张，容忍一点精度损失 |
| **HNSW** | 多层图结构，类似跳表 | 极快 | 极高 | 很高 | 千万级，要求极高 QPS |
| **PQ** | 乘积量化，向量分段压缩 | 中等 | 中 | 极低 | 十亿级，内存极度受限 |

**总结一句话**：
> **ANN 是 RAG 系统的"加速引擎"：它不精确计算所有向量的距离，而是通过索引结构快速锁定可能相似的候选，让百亿级向量库的检索速度从秒级降到毫秒级。**

### 二、父子分块（Parent-Child Chunking）详解

#### 什么是父子块？

假设你有一篇很长的文章，比如 5000 字。为了能放到向量数据库里检索，你会把它切成小块：

**第一步：切"父块"（Parent Chunk）**
- 每个父块大概 **2048 个 token**（大约 1500 个汉字）
- 你的文章会被切成多个父块：`父块0`、`父块1`、`父块2`……

**第二步：把每个父块再切成更小的"子块"（Child Chunk）**
- 每个子块大概 **512 个 token**
- 目的：子块更小，查询时能更精确地命中某个具体段落；但为了给 LLM 足够上下文，最终返回的是子块所属的整个父块

**示例结构**：
```
父块 A（2048 tokens）
├── 子块 A1（512 tokens）
├── 子块 A2（512 tokens）
├── 子块 A3（512 tokens）
└── 子块 A4（512 tokens）   ← 一共 4 个子块
```

#### 父子分块常用数目

**工业界标准参数**（中文社区实际应用）：

| 参数 | 推荐范围 | 说明 |
|------|---------|------|
| **子块大小** | 256 ~ 512 tokens | 小粒度，精确匹配细节 |
| **父块大小** | 1024 ~ 2048 tokens | 大粒度，提供完整上下文 |
| **块重叠（Overlap）** | 块大小的 10%-20% | 缓解边界信息断裂 |

**海外社区常见配置**：
- 父块：2000 tokens
- 子块：400 tokens
- 比例：2000 / 400 = 5

**你的项目配置**：
- 父块：2048 tokens
- 子块：512 tokens
- 理论比例：2048 / 512 = **4 个子块/父块**

#### "不均匀"是常态，而不是意外

**重要认知**：每个父块拥有的子块数量**不固定**，由原始文本的实际长度决定。

**原因1：文档长度各异**
- 一份常见的企业级 Word 或 PDF 文档，篇幅经常是几千甚至几万字
- 绝大多数父块的最后一个子块都是不满的
- 例如：父块总长度 1500 tokens → 只能切出 3 个子块（512+512+476）

**原因2：为了语义完整而"主动调整"**
- 专业的分块算法（如递归分块 Recursive Chunking）在切分时会优先寻找句子、段落等自然的语义边界来切割
- 这就像切蛋糕时，会刻意避开在蛋糕上精美的奶油字中间下刀，哪怕这样切出来的块大小不一
- 这会进一步加剧子块大小的"不均匀"，但能让每个子块都包含完整、自洽的语义信息

**语义边界的重要性示例**：
```
假设有这样一句话：
"2023年第一季度财报显示营业收入同比增长15%，但净利润同比下降8%。"

❌ 在句子中间切断：
   子块1: "2023年第一季度财报显示营业收入同比增长15%，但"
   子块2: "净利润同比下降8%。"
   
✅ 以句子为单位切断：
   子块1: "2023年第一季度财报显示营业收入同比增长15%，但净利润同比下降8%。"
   
如果用户问"营收增长但利润下降的原因"，包含完整因果关系的子块就能被一次性检索到。
```

**结论**：从工程实践角度看，**"不均匀"是绝对的主流**。一个父块几乎总是会被切分成"N个完整的子块 + 1个可能不完全填满的子块"。

#### Token 和字符的区别

**关键误区**：2048 和 512 是 **Token 数**，不是字符数！

**Token 是什么？**
- Token 是模型理解文本的基本语义单位
- 一个 token 可能是一个完整的词，也可能是词的一部分
- OpenAI 的 `text-embedding-3-small` 模型，其上下文窗口限制就是 8191 个 **token**，而不是字符数

**字符数因语言而异，比例很不固定**：
- 英文：1 个 token ≈ 4 个英文字符
- 中文：1 个 token ≈ **0.75 个中文字符**（反过来：1 个中文字符 ≈ 1.33 个 token）

**换算示例**：
```
2048 个 token（中文）≈ 2048 × 0.75 ≈ 2730 个中文字符
512 个 token（中文）≈ 512 × 0.75 ≈ 384 个中文字符
```

**为什么必须以 Token 作为衡量单位？**
- 如果只用字符数，在不同语言间就会失去统一的衡量标准
- 导致有的块特别大，有的块特别小
- Token 是模型实际处理的单位，直接用 Token 切分最准确

### 三、父子分块解决的问题

#### 核心矛盾：检索粒度 vs 上下文长度

**如果直接用父块（比如 1024 tokens）作为检索单元**：

| 问题 | 说明 |
|------|------|
| **语义稀释** | 向量检索时，父块的语义向量会被其中大量不相关的句子"稀释" |
| **匹配不精确** | 即使块里有 1 句是完美答案，也可能因为整个块的平均相似度不高而排在后面 |

**示例**：
> 文档父块（1024 tokens）里有一段："张三在 Q3 的销售额是 120 万"，其他大部分内容是团队介绍、市场分析等。
> 
> 查询："张三 Q3 销售额"
> 
> 直接用父块检索：向量检索会把整个父块的语义平均一下，可能觉得这个块主要是"团队和市场"，排名不高。

**如果直接用子块（比如 256 tokens）作为检索单元并直接返回**：

| 问题 | 说明 |
|------|------|
| **上下文太短** | 子块上下文太短，LLM 看不到前后文 |
| **信息不完整** | 回答可能缺少背景、指代不明、信息不完整 |

**父子分块的精髓**：
- ✅ 用 **子块**（小粒度）去做检索 → **精确命中**包含关键细节的那一段
- ✅ 返回 **父块**（大粒度）给 LLM → 提供**足够的上下文**

这是 **"精确召回 + 完整供给"** 的经典模式。

### 四、混合检索（Hybrid Search）解决的问题

#### 单一检索方法的盲区

混合检索解决的是另一个问题：**单一检索方法的局限性**

| 检索方法 | 擅长 | 不擅长 |
|---------|------|--------|
| **向量检索** | 语义相似（如"销售额"≈"营收"） | 忽略关键词精确匹配（如"张三"这个词具体出现的位置） |
| **BM25** | 关键词匹配（精确命中"张三"） | 不懂语义（"销售额"和"营收"可能视为不同词） |

**RRF（Reciprocal Rank Fusion）**：融合两者的排序，综合优势。

#### 混合检索 vs 父子分块：正交的优化维度

它们不是互斥的，而是 **两个正交的优化维度**：

| 维度 | 解决的问题 | 你的方案中是否使用 |
|------|-----------|------------------|
| **检索粒度** | 精确命中细节 vs 上下文完整 | ✅ 父子分块 |
| **检索方法** | 单一方法的盲区（语义 vs 关键词） | ✅ 混合检索 + RRF |

**两者叠加的效果**：
- 细粒度召回（子块）
- 粗粒度上下文（父块）
- 多路融合排序（BM25 + 向量 + RRF）

这是工业级 RAG 系统的常见组合，不是过度设计。

### 五、父子分块的适用情况

#### 什么时候父子分块很有意义？

**场景1：查询包含细节实体**
- "张三在第 3 季度的销售额是多少"
- "合同中第 5.2 条的违约责任是什么"
- "某某说了什么话"
- "某个数字是多少"

这种情况下，父子分块会明显优于直接用父块检索。

**场景2：文档长度普遍很长（>2000 tokens）**
- 用户经常问"某个细节出现在哪段上下文"
- 需要精确定位到具体段落，同时保留周围语境

**场景3：你不知道未来查询分布**
- 保留父子分块是一个更稳健的默认设计
- RAG 系统面对的用户查询往往是混合的：既有主题问题，也有细节问题

#### 什么时候父子分块"没必要"？

**场景：查询全是长文本、主题级别的，不涉及细节实体**

例如：
- "这篇文章讲了什么？"
- "总结作者的主要观点。"
- "这份报告的核心结论是什么？"

这种情况下，父块本身的语义向量已经足够匹配整个主题，子块检索不会带来额外收益。此时你可以直接用父块检索，简化系统。

#### 折中方案

如果你觉得当前系统复杂度太高，可以考虑：

> **放弃父子分块，直接用较小的父块（比如 512 tokens）作为检索和返回单元。**

**优点**：
- 检索粒度足够精确（512 tokens 已经比较小了）
- 上下文对大多数问题也够用
- 系统简单，不需要维护父子关系和去重

**代价**：
- 对于需要超长上下文的问题（比如"从这份合同中找到所有违约责任条款"），512 tokens 可能不够，需要拼接多个块
- 但那可以通过多轮检索或重排序解决

## 📖 只需要了解的点

### 1. 覆盖率阈值方案的缺陷

你可能想过：如果某个父块被命中的子块数量很少（比如只有 1 个），就不要返回整个父块，而是直接返回那 1 个子块。

**这个方案不可行，原因**：

1. **你不知道一个父块一共有几个子块**
   - 你只知道命中了几个子块，但不知道这个父块总共含有几个子块
   - 无法计算覆盖率（命中数 / 总数）

2. **即使记录了总数，覆盖率阈值也不一定合理**
   - 一个父块共有 8 个子块，你只命中了 1 个（覆盖率 12.5%）
   - 但是，那 1 个子块恰恰是整个父块里最核心的一段
   - 按照覆盖率规则，你会返回子块（512 tokens），而丢掉父块里可能很有用的上下文
   - 这反而可能降低回答质量

**结论**：覆盖率阈值是一个"看起来很美，实际很难用好"的方案，主流 RAG 系统不这么干。

### 2. 去重的必要性

**当前代码的问题**：
```python
for hit in results[0]:
    pid = hit["entity"]["parent_id"]
    output.append({
        ...,
        "parent_content": parent_content_map.get(pid, ""),
    })
```

如果同一个父块的 3 个子块都被检索命中（比如查询匹配了子块 A1、A3、A4），那么输出列表里会有 **3 条记录**，每条记录的 `parent_content` 都是同一个父块的全部内容。

**结果**：同一段父块文本重复了 3 次，白白浪费 tokens，而且 LLM 看了三遍相同的内容，没有意义。

**解决方案**：按 `parent_id` 去重，只保留相似度最高的那条记录。

### 3. 为什么不需要担心"返回太多无关内容"？

你担心的场景：一个父块只有 1 个子块被命中，你却返回了整个父块（2048 tokens），其中 3/4 的内容可能是无关的。

**客观事实**：

1. **现代 LLM 对无关内容的容忍度很高**
   - 即使一段文本里 70% 的内容无关，只要相关的 30% 足够清晰，LLM 依然能正确回答
   - 这是 LLM 与老式信息抽取系统的本质区别

2. **子块已经精确命中了相关内容**
   - 父块只是把这个相关段落的周围内容也拿过来，相当于"扩展了视野"，而不是"灌入垃圾"
   - 比如，子块是"苹果公司发布新 iPhone"，父块里还可能包含"这款 iPhone 的售价、发布日期"——这些对你回答"iPhone 有什么新功能？"是有帮助的

3. **如果真有大量垃圾，应该用 Reranker 去过滤**
   - Reranker 是一个小模型，会重新计算查询与每个父块的真实相关性
   - 然后只保留最相关的 top-k
   - 这比"覆盖率"规则可靠得多
