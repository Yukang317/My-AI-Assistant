## 一、grep vs Read 底层执行逻辑对比

### grep 的执行流程

`grep -c "TODO(human)" rag/` 底层流程：

```
shell fork+exec grep 子进程
  → 遍历目录找文件
  → 对每个文件: open() → read() 逐字节读（内核态，每次 4K/8K buffer）
  → 遇 \n 切行 → 正则引擎匹配 "TODO(human)"
  → 匹配 → 计数+1；不匹配 → 跳过整行
  → -c 模式：只输出 <文件名>:<数字>，不输出行内容
```

**关键**：grep 是**过滤（filter）操作**——它从全量数据中筛出匹配正则的子集，丢弃所有不匹配的内容。信息在这个阶段永久丢失，不进入 LLM 上下文。

### Read 工具的执行流程

```
LLM 决定调用 Read(file_path="...")
  → Claude Code 运行时在沙箱中执行文件读取（Python open().read() 或类似）
  → 文件完整内容（含行号）注入 LLM 上下文窗口
  → LLM 处理全部内容
```

**关键**：Read 是**传输（transfer）操作**——把文件完整内容从磁盘搬到 LLM 上下文。信息完整，但 LLM 注意力机制可能选择性忽略某些行。

### 核心区别

| 维度 | `grep -c` | Read 工具 |
|------|----------|----------|
| 执行者 | 操作系统 C 程序 | Claude Code 运行时 + LLM |
| 输出目标 | 终端 → LLM 看到结果 | 文件内容 → LLM 上下文窗口 |
| 信息完整度 | 仅匹配行 | 文件全部行 |
| 有无上下文 | 零上下文 | 完整上下文 |
| 能否看到 TODO 下方的代码 | ❌ 正则没匹配，代码根本不在结果里 | ✅ 代码在上下文里，但注意力可能忽略 |
| LLM 参与时机 | 收到结果后 | 读到内容后直接处理 |

### 为什么 grep 看不到注释下方的代码

以 Mode E 产生的代码为例：

```python
def search(query):          # 第1行 → grep: 不匹配，跳过
    """搜索"""              # 第2行 → grep: 不匹配，跳过
    # TODO(human): 实现搜索  # 第3行 → grep: ✅ 命中！
    # 提示：.....           # 第4行 → grep: 不匹配，跳过
    results = store.search( # 第5行 → grep: 不匹配，跳过 ← 实际代码！
        query_emb, limit=10 # 第6行 → grep: 不匹配，跳过
    )                       # 第7行 → grep: 不匹配，跳过
    return results          # 第8行 → grep: 不匹配，跳过
```

grep 只输出第 3 行。第 5-8 行的实现代码对 grep 完全不可见——它们没匹配正则。而 LLM 收到的 grep 结果里没有这些代码，所以 LLM 根本不知道它们存在。

## 二、"偷懒"现象根因分析

**结论**：70% 智能体策略 + 30% LLM 模型特性。

### 智能体策略层面（主要）

1. **工具选择的效率陷阱**：grep 毫秒级返回数字 vs Read 需要多次调用 + 消耗大量上下文。智能体天然倾向低成本路径。
2. **上下文窗口经济压力**：grep ~200 tokens vs 逐文件 Read ~50,000+ tokens。更多上下文 = 更慢响应 + 更高成本 + 推理质量下降。
3. **Explore Agent 设计定位**：文档写明了 "It reads excerpts rather than whole files, so it locates code; it doesn't review or audit it." 这是功能设计，不是 bug。

### LLM 模型层面（次要）

1. **DeepSeek v4 MoE 稀疏激活**：每个 token 只激活部分专家，对自然语言注释的注意力可能高于代码。
2. **信号显著性偏差**（所有 LLM 共有）：`# TODO(human)` 是高显著性信号（像指令），实际代码是低显著性信号（常规文本）。LLM 扫描长文本时优先提取高显著性信号形成"摘要判断"。

### 交互放大效应（信息级联）

```
grep 只返回标记行 → 模型推断"未实现" → 写入上次会话成果.md → 后续会话当作事实 → 错误放大
```

### 权重分解

| 维度 | 原因 | 占比 |
|------|------|------|
| 智能体策略 | 工具选择偏向效率 | ~40% |
| 智能体策略 | 上下文窗口经济压力 | ~30% |
| LLM 模型 | MoE + 信号显著性偏差 | ~20% |
| LLM 模型 | 长上下文注意力衰减 | ~10% |
