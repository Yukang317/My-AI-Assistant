**内容**：定位 `/chat/graph` 请求后前端不显示回复的根因，修复两个文件

## 〇、做了什么 & 为什么

**背景**：用户在前端测试"我的沟通偏好是什么？"时，两次请求都显示"思考中"2-3 分钟没有回复。上一轮会话初步排查认为是 DeepSeek API 慢 + SSE 伪流式问题，但追踪日志（`logs/agent_trace.jsonl`）揭示了真正的根因。

**核心发现**：不是 API 慢，是 `result_synthesis.py` 第 67 行 `msg["role"]` 的 **类型不兼容崩溃**——`add_messages` reducer 把 `load_context` 返回的 dict 自动转成了 `HumanMessage` 对象，而 `result_synthesis` 用 dict 下标语法访问对象，导致 `'HumanMessage' object is not subscriptable`。

**修复**：`load_context.py` + `result_synthesis.py`，源头和消费端双重防护。

## 二、根因详解

### 数据流追踪

```
load_context()
  └→ db_load_messages(session_id) → list[dict] 如 [{"role":"user","content":"..."}, ...]
  └→ return {StateField.HISTORY_MESSAGES: recent}
       ↓
  add_messages REDUCER（state.py:88）
  └→ Annotated[list[BaseMessage], add_messages]
  └→ 自动把 dict 转成 HumanMessage / AIMessage 对象！
       ↓
  result_synthesis() 读取 state
  └→ history = state.get(StateField.HISTORY_MESSAGES, [])
  └→ 此时 history 里是 [HumanMessage(...), AIMessage(...)]
  └→ msg["role"] → CRASH! HumanMessage 不支持下标操作
```

### 根本矛盾

| 模块 | 期望类型 | 实际类型 |
|------|---------|---------|
| `load_context` 产出 | `list[dict]` | `list[dict]` |
| `add_messages` reducer 转换后 | `list[BaseMessage]` | `list[HumanMessage\|AIMessage]` |
| `result_synthesis` 第 67 行假设 | `dict`（用 `msg["role"]`） | `HumanMessage` → 💥 |

`add_messages` 是 LangGraph 内置 reducer，设计意图是"方便"——自动把 dict 转成消息对象。但当下游代码混用两种访问方式时，类型不一致就成了选择性 bug（有历史消息时炸，无历史消息时不炸）。

## 三、修复内容

### 修复 1：`load_context.py` — 源头保证类型一致

```python
# 把 PG 返回的 dict 显式转成 LangChain 消息对象
from langchain_core.messages import HumanMessage, AIMessage

langchain_messages = []
for msg in recent:
    role = msg.get("role", "")
    content = msg.get("content", "")
    if role == "user":
        langchain_messages.append(HumanMessage(content=content))
    elif role == "assistant":
        langchain_messages.append(AIMessage(content=content))

return {
    StateField.HISTORY_MESSAGES: langchain_messages,  # 现在全是 LangChain 对象
    ...
}
```

### 修复 2：`result_synthesis.py` — 消费端兼容两种类型

```python
# 兼容 dict 和 LangChain 消息类型
for msg in history:
    if isinstance(msg, HumanMessage):
        messages.append(msg)          # LangChain 对象直接复用
    elif isinstance(msg, AIMessage):
        messages.append(msg)
    elif isinstance(msg, dict):       # 兜底：如果还是 dict
        role = msg.get("role", "")
        content = msg.get("content", "")
        if role == "user":
            messages.append(HumanMessage(content=content))
        elif role == "assistant":
            messages.append(AIMessage(content=content))
```

## 四、讲解修复原理 — 为什么改了那两个文件就恢复正常了？

修复前后走的是**两条完全不同的 SSE 数据流路径**：

```
修复前 ❌（错误路径）：
  前端 → POST /chat/graph → StreamingResponse
    → generate_graph_stream_response()
      → await future (run_graph 在线程池里跑)
        → load_context: list[dict] → add_messages 转成 [HumanMessage, AIMessage]
        → intent_route → tool_execute ✅
        → result_synthesis: msg["role"] → 💥 'HumanMessage' object is not subscriptable
        → 异常抛出 → _invoke_with_trace 记录 trace_run_end(error=...) → re-raise
      → await future 收到异常 → try-except 捕获
      → yield {"error": "Agent Graph 处理出错：...", "finished": true}  ← 只有 error 事件！
    → 前端收到 error SSE → data.error 为真 → throw → catch 块
    → 理论上应显示 "网络错误：..."，但实际显示"思考中"（前端 bug，待修）

修复后 ✅（成功路径）：
  前端 → POST /chat/graph → StreamingResponse
    → generate_graph_stream_response()
      → await future (run_graph 在线程池里跑)
        → load_context: 源头转成 [HumanMessage, AIMessage] ← 类型一致
        → ... → result_synthesis: isinstance(msg, HumanMessage) ✅
        → LLM 生成回复 → {FINAL_RESPONSE: "你的沟通偏好：..."}
      → await future 拿到回复文本
      → yield {"content": "你的沟通偏好：...", "finished": false}  ← content 事件！
      → yield {"content": "", "finished": true}                     ← 结束事件
    → 前端: data.content !== undefined → accumulated += data.content
    → bubble.innerHTML = renderMarkdown(accumulated) → ✅ 正常显示
```

**一句话总结**：修复前 `run_graph` 在 `result_synthesis` 崩溃，SSE 只产出 error 事件；修复后 `run_graph` 成功完成，SSE 产出 content 事件，前端正常渲染。

## 五、MEMORY.md dedup 逻辑 — 对话知识点笔记

以下知识点来自本次会话早期对 `memory_update.py` 中 `_dedup_bullets` 函数的学习讨论。

### Q1：Jaccard 是什么？为什么阈值设为 0.75？

**背景**：`SIMILARITY_THRESHOLD: float = 0.75`，用于判断两条 bullet 是否"足够相似"以视为重复。

**解答**：Jaccard 相似度 = `|A ∩ B| / |A ∪ B|`，即两个集合的**交集大小除以并集大小**。

在这个项目里，`_bullet_similarity` 把每条 bullet 拆成字符集合（如 `"喜欢小步教学讨厌说教"` → `{'喜','欢','小','步','教','学','讨','厌','说'}` ），两条消息共有的字符越多、独有的字符越少，Jaccard 值越接近 1.0。

**为什么是 0.75**：经验值。相当于"两条 bullet 共享了 75% 的字符"。比 0.5 严格（避免过度合并），比 0.9 宽松（允许 LLM 换个说法）。实际去重是**四层判定**：归一化相等 → 子串包含 → 短语子集 → Jaccard，Jaccard 是兜底。

### Q2：`re.compile()` 是啥？`_PHRASE_SEP` 和 `_PUNCT` 分别干什么？

**解答**：`re.compile(pattern)` 把正则表达式**预编译**成一个"正则对象"，之后调用 `.split()` / `.sub()` 时不用每次重新解析正则——在本项目这种每次对话都跑一次的场景里能省几毫秒。

两个正则的分工：
- **`_PHRASE_SEP`**：切分短语。`[、，,;；/。！!？?\s]+` 匹配中英文标点+空白符，用于把 `"喜欢小步教学、讨厌说教"` 拆成 `{"喜欢小步教学", "讨厌说教"}`。
- **`_PUNCT`**：剔除标点。`_PUNCT.sub("", t)` 把所有标点替换为空字符串，只留实义字符做比较——因为标点对"这两句话是不是同一个意思"没有贡献。

### Q3：`_dedup_bullets` 里 `kept` 一开始是空列表，`for i, existing in enumerate(kept)` 怎么执行？

**背景**：`_dedup_bullets` 遍历每条 bullet 和 `kept` 里已保留的条目逐一比对相似度。

**解答**：第一条 bullet 遍历时 `kept = []`，`for i, existing in enumerate(kept)` **直接跳过**（空列表 for 循环不执行任何迭代），于是 `matched` 保持 `False`，走 `kept.append(bullet)` 把第一条加入。这保证了第一条 bullet 总是被保留，后续 bullet 再跟它比对——相当于以第一条为基准。

### Q4：阈值已经判断相似度了，为什么还要加 `len(bullet) > len(existing)` 长度比较？

**解答**：阈值 0.75 是近似值，不是精确判定。当两条 bullet 相似度 ≥ 0.75 时要**择优保留**——保留更长（信息更全）的那条。

例如 `"讨厌说教"` vs `"喜欢小步教学、讨厌说教"`：Jaccard ≈ 0.47（不够 0.75 阈值），但会被更早的"短语子集"判定为重复。此时保留更长的后者，因为前者只是后者的子集。长度比较确保**去重时不会丢失信息**。

### Q5：`_dedup_bullets` 是 O(n²) 的，生产环境 MEMORY.md 大了会不会慢？

**解答**：当前 `MAX_MEMORY_CHARS = 2000`（约 500 token），就算全是 bullet 也不过几十条。O(n²) 在几十条数据上是**毫秒级**。况且 `_dedup_bullets` 只是工程层兜底（拦最明显的换说法重复），阶段 7.2 会用 LLM 做真正的语义总结压缩——那时候 n 会更小。**过早优化是万恶之源**，先跑通再谈性能。

### Q6：`merge_patch_into_memory` 的去重逻辑是全局的还是按 section 的？

**解答**：**按 section 的**。每个补丁只对自己所属的 `## section` 分区执行 `_dedup_bullets`：

```python
for patch in patches:
    section = patch["section"]  # 如 "状态/背景"
    # 只截取这个 section 的 lines
    section_lines = lines[section_index:insert_index]
    # 只收集这个 section 内的 bullet
    existing_bullets = [line for line in section_lines[1:] if ...]
    existing_bullets.append(bullet)  # 加新 bullet
    deduped_bullets = _dedup_bullets(existing_bullets)  # 仅此 section 去重
```

**这意味着**：当用户说"我是太原理工毕业的"→ 补丁打到"状态/背景"→ 只去重"状态/背景"分区的条目。"沟通偏好"分区即便有 3 条重复也不会被清理——因为这次没有新补丁打到"沟通偏好"。

**已确认**：用户实测"对了，我还是太原理工大学毕业的本科生"后，MEMORY.md 的"状态/背景"更新了，但"沟通偏好"三行重复依旧存在——这就是按 section 去重的直接证据。
