# 日志记录和笔记记录风格

**来源**：`学习过程与学习过程中的项目/学习过程中的RAG项目实战/开发日志笔记.md`（2374 行参考日志）
**用途**：开发日志和笔记的写作风格参考，供 dev-logger skill 和手动记录时参考。

---

## 一、核心原则：像学生在记笔记，不是在写文档

日志的本质是"给自己看的思考记录"，不是"给外人看的正式文档"。把当时的困惑、理解过程、试错经历原汁原味保留下来。

---

## 二、八大写作风格要素

### 1. 对话式语气

- 用"你"而不是"用户"，像是在对自己说话
- 语言口语化，允许"就是""其实""对吧"等口头禅
- 示例："`split('.')`：字符串方法，按 `.` 分割文件名——这个你之前用过对吧？"

### 2. 解释为什么（不只是"做了什么"）

每条记录都回答两个问题：做了什么 + 为什么这样做。

- ❌ 差："使用 `tempfile` 创建临时文件"
- ✅ 好："为什么需要 tempfile？上传的文件是二进制数据流在内存里，没有文件路径。LangChain 的加载器认路不认内存，所以必须写到临时文件让加载器能找到。"

### 3. 问题驱动（以问题为线索组织内容）

不是按"代码结构"组织，而是按"遇到的问题"组织：

```
模块 3：core/document_processor.py
  → 遇到的问题：
    1. 为什么需要 tempfile 创建临时文件？
    2. 三个分割器都在 __init__ 里，会把文本切割三次吗？
    3. chunk_size 和 separators 的优先级关系？
```

每个问题展开为：**背景 → 解答 → 关键代码（如有）→ 类比（如有）**

### 4. 自提疑问（最有价值的部分）

把自己当时疑惑的问题记录下来并给出完整解答，格式：

```
#### 自提疑问 1：检索到的子文档合不成一个父块怎么办？

**首先明确"合不成一个父块"的常见场景**：
1. 检索到的子文档来自多个不同的父文档
2. 子文档的 parent_doc_id 无效 / 找不到对应的父文档
3. 只有零散的子文档，没有对应的完整父文档

**其次代码中对结果的强依赖**：
...
```

### 5. 代码 + 注释风格

代码块里带注释，用口语化中文解释"这一行在干什么"：

```python
# ✅ 好的注释风格
with tempfile.NamedTemporaryFile(delete=False, suffix=f".{uploaded_file.name.split('.')[-1]}") as tmp_file:
    tmp_file.write(uploaded_file.getvalue())  # 把上传文件的内容写入临时文件
    tmp_path = tmp_file.name  # 保存临时文件的绝对路径

# ❌ 差：没注释，或者用英文术语堆砌
```

### 6. 踩坑实录

每个 bug 记录完整三要素：**现象 → 原因 → 修复**：

| # | Bug 现象 | 根因 | 修复 |
|---|---------|------|------|
| 1 | 文档处理失败 `TypeError: function missing required argument 'year'` | `content = Column(MEDIUMINT, ...)` — MEDIUMINT 是整数类型，存不了长文本 | 改为 `MEDIUMTEXT` |
| 2 | 同上，第二次失败 `Incorrect integer value` | 数据库表中 content 列仍是旧的整数类型 | `ALTER TABLE` 自动修复 |

### 7. 类比解释

用生活化类比解释技术概念，降低理解成本：

- "`session.flush()` 就像把购物车里的东西放到收银台上——还没结账（commit），但东西已经不在购物车里了"
- "`parent_chunk.document` 就像点击父块对象中的超链接，跳转到文档对象"
- "索引就像书的目录：没有目录时你得逐页翻（全表扫描），有目录直接翻到对应页"

### 8. 概念卡片

遇到新概念时，用简短的一句话解释或小表格：

```
- `engine`：你家到超市的"路"，所有数据库操作都走这条路
- `SessionLocal`：超市的"购物车工厂"，每次操作领一个新购物车
- `Base.metadata.create_all`：告诉超市"我需要这些货架"，自动搭好
```

## 三、分层结构模板

一个完整模块的记录建议按以下层次组织：

```
### 模块 N：路径/文件名（一句话定位）

**作用**：
- 一句话说明这个模块在整个系统里的角色

#### 遇到的问题：

##### 1. 问题标题
**背景**：为什么会有这个问题
**拆解**：逐步解释涉及的变量/函数/逻辑
**关键代码**：贴出相关代码段
**类比理解**（可选）：生活化类比

##### 2. 下一个问题...

#### 整体逻辑总结（可选）：
`步骤A` → `步骤B` → `步骤C` → `结果`
```

## 四、日志 vs 笔记的区别

| 维度 | 开发日志（dev_log/） | 学习笔记（参考日志风格） |
|------|---------------------|------------------------|
| 粒度 | 一次会话的成果摘要 | 逐模块、逐函数的详细拆解 |
| 代码量 | 核心片段（≤60行） | 完整代码 + 逐行注释 |
| 问题深度 | 关键决策和 Bug | 每个变量、每行语法都追问 |
| 适用场景 | 项目进度追踪 | 新知识学习、模块深啃 |

**本项目的 dev_log 走"核心讨论+代码+Bug"级别**，不需要参考日志那种逐行拆解（除非是 Mode E 教学过程中用户的提问记录）。

---

## 五、关键禁忌

1. **不要写"已实现""已完成"就结束**——要说清楚怎么实现的、为什么这样实现
2. **不要跳过中间推理过程**——Bug 修复要保留"尝试了A→失败→尝试B→成功"的完整路径
3. **不要用抽象专业名词解释专业名词**——每个概念都要落到本项目具体代码/数据流上
4. **不要把日志写成 API 文档**——不是给外人看的接口说明，是给自己看的思考笔记
5. **不要复制大段无人声的代码**——代码块里必须有口语化注释
