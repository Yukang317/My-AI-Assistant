## 一、三个参考项目对比

你学过的三个 reference：

| 项目 | 模式 | PDF 方案 | Office 方案 | 亮点 | 缺陷 |
|------|------|----------|------------|------|------|
| MilDoc `parser/` | 策略模式 | PyPDF2 / MinerU VLM | markitdown | 策略模式 + 多编码兼容 | 协调器管太多（MinIO+分块+MD5一把抓） |
| LlamaIndex RAG `core/` | 管道模式 | pymupdf4llm | ❌ 不支持 | 稳定 ID 去重 + 双管道分块 | 强依赖 LlamaIndex，无 Office |
| RAG实战 `rag_project/` | if/elif 分支 | PyPDFLoader | Docx2txtLoader | 父子分块内置 + tempfile | 无抽象接口、不可扩展 |

### 各项目的"精华"提取

**从 MilDoc 取**：策略模式架构 + 多编码 fallback（utf-8→gbk→gb2312→latin-1→errors='ignore'）
**从 LlamaIndex 取**：pymupdf4llm（PDF→Markdown，含图片+表格）+ 稳定文档 ID 设计
**从 RAG实战 取**：tempfile 中转模式 + 父子分块思路

### 为什么 parser 不用框架

parser 的职责是 `bytes → str`，是最底层的原子操作。框架的价值在**编排层**——当需要协调多个模块、管理状态、处理重试时才有意义。把 bytes 解码成字符串不需要框架。

## 二、自提疑问 & 解答

### Q1：策略模式嵌套策略模式会不会显得没必要？

**背景**：你想在 PDF 解析时再做一层策略细分——PyPDF2（纯文本）、pymupdf4llm（表格/图片）、MinerU（高级场景）。但如果要先"意图识别"来判断 PDF 内容类型，就多了一层嵌套。

**结论**：你的直觉是对的，嵌套策略模式在现阶段没必要。原因：
- 嵌套会导致组合爆炸（3 文档类型 × 3 PDF 子策略 = 9 种路径）
- "意图识别"是 Agent 的事，不是 parser 的事
- 正确做法是**平面化注册**：所有解析器注册到协调器同一层，按优先级排序即可

```python
# 平级注册，不是嵌套
self.parsers = [
    DocxParser(),        # markitdown
    MarkdownParser(),    # 多编码兼容
    PDFParser(),         # pymupdf4llm（主力，覆盖 90%）
    TextParser(),        # 兜底
]
# 未来加 MinerU 只需加一行：
#   MinerUParser(),
```

### Q2：LlamaIndex 和 LangChain 各自的优势是什么？为什么企业级系统要一起用？

**背景**：我之前说"不建议同时深度使用"，你说得对——企业级系统确实是二者配合使用。

**结论**：

| 维度 | LlamaIndex | LangChain |
|------|-----------|-----------|
| 设计哲学 | **数据驱动**，围绕"如何让数据被高效索引与检索" | **流程驱动**，围绕"如何编排应用逻辑" |
| 核心抽象 | Document, Node, Index, QueryEngine | Chain, Agent, Tool, Prompt |
| 在 RAG 中的位置 | **上游 R（Retrieve）**：数据加载→分块→索引→检索 | **中游 AG（Augmented Generation）**：Prompt 组装→工具调用→结构化输出→引用标注 |
| 最佳用途 | 文档摄入管道、混合检索、分块策略 | 对话管理、Agent 编排、多轮上下文融合 |

而 **LangGraph** 是下游的 **编排层**：意图路由→条件分支→HITL→子图调度。三者不是竞品，是上下游协作关系。

### Q3：框架应该在什么时机引入？

**原则**："疼了才引入"——当原生方式写起来很痛苦时，框架的价值就显现了。

具体判断表：

| 阶段 | 用什么 | 为什么 |
|------|--------|--------|
| Step 1-5 | 纯原生 Python | 每个模块只做一件事，框架是累赘 |
| Step 6 (chunker) | LangChain 分块器 | 语义分隔符、重叠、多级分块手写要 200+ 行 |
| Step 7 (indexer) | 先原生，后 LlamaIndex | MVP 阶段 pipeline 简单，需缓存去重时再升级 |
| Step 8 (retrieval) | 先手写 RRF，后 LlamaIndex | 理解原理后再用框架 |
| Step 9 (rag_service) | LangChain | 就该用框架——Prompt 组装+LLM调用+输出解析 |
| 未来 Agent | LangGraph | 你已有 SAGT 深度实战经验 |

### Q4：LangGraph 在项目中的正确定位？

**结论**：LangGraph 是整个个人 AI 助理的**核心骨架**（主程序），RAG 只是它身上的一个器官（子图/工具）。

你的 SAGT 项目已经验证了"主图 + 6 子图"的模式。个人助理可以完全复用：
- 主图：意图检测 + 条件路由
- 子图：RAG 子图、日程子图、邮件子图、学习教练子图...
- 每个子图可以独立开发、独立测试、独立部署

## 三、关键技术决策

| 决策 | 选择 | 理由 |
|------|------|------|
| PDF 解析库 | pymupdf4llm | 比 PyPDF2 质量高 10 倍——保留表格+图片+公式，输出 Markdown |
| DOCX 解析库 | markitdown | 一个库覆盖 doc/docx/xls/xlsx/ppt/pptx，输出 Markdown |
| Parser 架构 | 纯策略模式 + 原生库 | 不引入框架，bytes→str 不需要 LlamaIndex |
| 框架引入时机 | "疼了才引入" | 底层原子操作原生，编排层用框架 |
| 开发路线 | 四层金字塔 | 基础架构→RAG底座→智能检索→Agent智能体 |
