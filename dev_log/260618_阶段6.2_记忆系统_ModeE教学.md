### 为什么先做记忆系统

阶段 6 技术方案里，记忆用 **Markdown 文件** 而非向量库（借鉴 OpenClaw / Claude Code 思路）：
- 可读、可改、可版本控制
- Agent 每轮对话前 `load_context` 读入 ~500 token 用户画像
- 对话后 `memory_update` 增量写回，形成「读 → 用 → 写」闭环

### 记忆在整体架构中的作用

```
用户提问
  → load_context（读 MEMORY.md + PG 历史）     ← 读记忆
  → intent_route → tool_execute（可选）
  → result_synthesis（LLM 回复，注入 memory_context）
  → memory_update（LLM 抽补丁 → 合并 → 写 MEMORY.md）  ← 写记忆
  → END
```

MEMORY.md **不替代** PostgreSQL（完整聊天记录）和 Milvus（知识库检索），它是轻量「用户画像快照」。

## 二、代码全貌（核心逻辑）

### 2.1 常量与 Prompt

```python
MEMORY_PATH = "/root/assistant/personal_assistant/memory/MEMORY.md"
MAX_MEMORY_CHARS = 2000          # ~500 token
AUTO_APPLY_MEMORY = True         # True=直接写盘；False=半自动待确认

# EXTRACT_SYSTEM_PROMPT：告诉 LLM 只输出 JSON 数组，section 限定 4 个值
# EXTRACT_USER_TEMPLATE：注入本轮 user_input + assistant_reply
```

### 2.2 补丁数据结构

LLM 输出格式：

```json
[{"section": "当前焦点", "content": "用户在做 personal_assistant 项目"}]
```

合法 section：`当前焦点` / `沟通偏好` / `状态/背景` / `待探索`

### 2.3 数据流（单轮对话）

```
State.user_question + State.final_response + State.memory_context
    ↓ extract_memory_patch（调 LLM）
list[dict] patches（空则跳过）
    ↓ merge_patch_into_memory（按 ## section 定位，追加 bullet）
    ↓ truncate_memory（超 2000 字截断）
    ↓ write_memory_file（落盘）
State.memory_context 更新
    ↓ 下轮 load_context 再读入
result_synthesis 的 System Prompt 注入新画像
```

### 2.4 merge_patch_into_memory 核心思路

- 按**行**处理 Markdown，不用字符串盲目拼接
- 找 `## {section}` 标题行 → 确定 section 范围（到下一个 `##` 之前）
- 新 bullet 追加到 section 末尾；section 不存在则在文末新建
- 跳过重复 bullet；删除模板占位符 `- （待记录）`
- 合并后调用 `truncate_memory`

### 2.5 memory_update 节点逻辑

```python
def memory_update(state, model):
    user_input = state[USER_QUESTION]
    assistant_reply = state[FINAL_RESPONSE]
    memory_context = state[MEMORY_CONTEXT]

    if 缺关键字段: return {MEMORY_CONTEXT: memory_context}  # 省 token

    patches = extract_memory_patch(...)
    if not patches: return {MEMORY_CONTEXT: memory_context}

    if not AUTO_APPLY_MEMORY:
        return {MEMORY_CONTEXT: memory_context, "memory_pending_patches": patches}

    merged = merge_patch_into_memory(memory_context, patches)
    write_memory_file(merged)
    return {MEMORY_CONTEXT: merged}
```

### 2.6 main_graph.py 适配层

```python
def memory_update_node(state: MainState) -> dict:
    return memory_update(state, get_llm())

graph.add_node("memory_update", memory_update_node)  # 注册适配层，不是业务函数
graph.add_edge("result_synthesis", "memory_update")
graph.add_edge("memory_update", END)
```

## 四、自提疑问 & 解答

### Q1：LangGraph 的「适配层」是什么？为什么 `result_synthesis_node` 和 `memory_update_node` 要包一层？

**背景**：LangGraph 注册节点时，只认 `(state) -> dict` 这种签名——**只传 state，不传 LLM**。但 `result_synthesis(state, model)` 和 `memory_update(state, model)` 都需要第二个参数 `model` 来调 `model.invoke()`。

**解答（流水线类比）**：

| 角色 | 是什么 |
|------|--------|
| `state` | 流水线上的「工单」——含 user_question、memory_context、final_response 等 |
| `model`（LLM） | 车间里的「机器」——不是工单的一部分，但某些工序必须用 |
| 适配层 | 「工序包装员」——LangGraph 只给工单，包装员自己去 `get_llm()` 拿机器，再交给真正的业务函数 |

```python
# LangGraph 只能这样调：
def memory_update_node(state):
    return memory_update(state, get_llm())  # 包装员帮你注入 LLM
```

**为什么 `load_context` 不需要适配层？** 因为它的签名本来就是 `(state) -> dict`，只要一个参数。

**为什么用 `get_llm()` 而不是在业务文件里 new？** `get_llm()` 是模块级单例，整个 Graph 共用同一个 LLM 实例和 `.env` 配置；换模型只改一处。

**同类适配层**：`intent_node`、`result_synthesis_node`、`memory_update_node` 都是同一模式。

### Q2：`merge_patch_into_memory` 为什么要按「行」处理而不是字符串拼接？

**解答**：MEMORY.md 是结构化 Markdown（`## 标题` + bullet 列表）。按行可以：
- 精确定位 section 起止位置
- 在 section 末尾插入新 bullet，不破坏其他 section
- 去重、删占位符时只动目标 section 的行

直接 `current_memory + new_content` 无法保证 section 对齐。

### Q3：`truncate_memory` 为什么要在 suffix 里预留空间？

**解答**：`max_chars=2000` 是**全文上限**。如果先截 2000 字再拼 suffix，总长度会超过 2000。正确做法：

```python
cut_len = max_chars - len(suffix)  # 正文 + suffix 合计不超上限
```

### Q4：半自动模式 `AUTO_APPLY_MEMORY=False` 是干什么的？

**解答**：阶段 6.2 后期计划——LLM 抽出的补丁不直接写盘，而是把 `memory_pending_patches` 放进 State，等前端/API 用户确认后再落盘。避免 LLM 记错信息直接污染 MEMORY.md。

## 五、知识点记录

### claude（本次新增概念）

| 知识点 | 内容 |
|--------|------|
| 记忆三层定位 | MEMORY.md=轻量画像快照；PG=完整聊天历史；Milvus=知识库检索，三者职责不同 |
| 补丁式增量更新 | LLM 只输出「新增条目」JSON，merge 函数负责合并，避免每轮重写整文件 |
| section 限定 | 4 个固定 section 防止 LLM 乱建段落，保持 MEMORY.md 结构稳定 |
| LangGraph 适配层 | `(state) -> dict` vs `(state, model) -> dict` 签名不一致时的标准桥接模式 |
| get_llm() 单例注入 | Graph 层注入依赖，业务节点不耦合 LLM 配置 |
| 占位符清理 | 模板 `- （待记录）` 在首次写入真实内容时应删除 |
| AUTO_APPLY 开关 | 全自动写盘 vs 半自动 pending 确认，同一套 extract/merge 逻辑复用 |

### deepseek（问题驱动笔记）

#### Q1：`parse_patch_json` 为什么要剥 ```json 包裹？

LLM 经常输出 markdown 代码块而不是裸 JSON。函数先检测 ` ``` ` 开头，去掉首尾 fence 行，再 `json.loads`。解析失败或结构不对时**安全返回 `[]`**，不让 Graph 崩溃。

#### Q2：`extract_memory_patch` 的 System Prompt 为什么要注入已有记忆？

避免 LLM 重复输出已在 MEMORY.md 里的内容。Prompt 明确说「不要重复已有内容」，无新信息时输出 `[]`。

#### Q3：为什么 memory_update 放在 result_synthesis **之后**？

必须等 `final_response` 生成完毕才有「用户说了什么 + AI 回了什么」的完整对话对。放在 result_synthesis 之前拿不到 assistant_reply。

#### Q4：注册节点时为什么必须用 `memory_update_node` 而不是 `memory_update`？

```python
graph.add_node("memory_update", memory_update)       # ❌ LangGraph 只传 state，缺 model 参数
graph.add_node("memory_update", memory_update_node)  # ✅ 适配层内部 get_llm()
```

直接注册业务函数会报：`TypeError: memory_update() missing 1 required positional argument: 'model'`
