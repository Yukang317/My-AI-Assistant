### Mock 辅助函数

```python
def build_mock_tool(*return_results: ToolResult):
    """构造假工具函数，按顺序返回预设 ToolResult，最后一个无限重复"""
    call_count = [0]  # 用列表绕过 UnboundLocalError

    def mock_tool(ctx: ToolContext, question: str) -> ToolResult:
        idx = min(call_count[0], len(return_results) - 1)  # 防越界
        result = return_results[idx]
        call_count[0] += 1
        return result

    return mock_tool
```

```python
def build_mock_llm(*responses: str) -> MagicMock:
    """构造假 LLM，每次 invoke() 返回 MagicMock(content=预设字符串)"""
    call_count = [0]

    def side_effect(prompt: str):
        idx = min(call_count[0], len(responses) - 1)
        content = responses[idx]
        call_count[0] += 1
        return MagicMock(content=content)  # 包装成有 .content 的对象

    mock_llm = MagicMock()
    mock_llm.invoke.side_effect = side_effect
    return mock_llm
```

### patch 关键代码

```python
with patch("agent.main_graph.get_tool", return_value=mock_tool), \
     patch("agent.main_graph.get_llm", return_value=mock_llm), \
     patch("agent.main_graph.check_bound", return_value=(True, "")), \
     patch("agent.main_graph.get_tool_bound", return_value="READ_ONLY"):
    result_dict = tool_execute(state)
```

**为什么 `return_value=` 不能省**：`patch("path", mock_obj)` 把目标**替换为** mock_obj，所以 `get_tool("rag_search")` 等于调用 `mock_obj("rag_search")`，不是调用 `mock_obj`。`patch("path", return_value=mock_obj)` 才是让 `get_tool(...)` **返回** mock_obj。

## 二、自提疑问 & 解答

### Q1：`build_mock_llm` 为什么返回 `MagicMock(content=...)` 而不是直接返回字符串？

**背景**：真实 LLM 的 `model.invoke(prompt)` 返回一个对象，后续代码通过 `response.content` 取文本。如果 mock 直接返回字符串 `"STOP"`，那 `response.content` 就会报 `AttributeError`——字符串没有 `.content` 属性。

**解答**：用 `MagicMock(content="STOP")` 包装——`mock.content` 返回 `"STOP"`，完美模拟真实 LLM 返回对象的接口。

### Q2：`patch("agent.main_graph.get_tool", mock_tool)` vs `patch(..., return_value=mock_tool)` 有什么区别？

**背景**：初版代码两个 patch 都漏了 `return_value=`，导致测试失败。

**解答**：
- `patch(path, obj)` → 把目标**替换为** obj。`get_tool("rag_search")` 实际调用 `obj("rag_search")`
- `patch(path, return_value=obj)` → 目标函数仍存在，只是**返回** obj。`get_tool("rag_search")` 正常执行并返回 obj

mock 的函数（`get_tool`、`get_llm`）需要 `return_value=`；mock 的值（`check_bound` 返回的元组、`get_tool_bound` 返回的字符串）也需要 `return_value=`。

### Q3：为什么测 token 预算但不测 timeout？

**背景**：token 预算检查（第5.5步）在 `_should_continue_by_observation` 之外，不是观察者函数的规则。

**解答**：
- `budget_exceeded` 在 while 循环体内但在 `_should_continue_by_observation` **之前**——它是独立的停止点，和观察者的 5 条规则平行。但它只需要 mock 工具返回大数据 + state 设小预算，mock 成本极低
- `timeout`（规则3）需要 mock `time.monotonic()` 这个 C 级内置函数，`patch` 搞不定，必须引入 `freezegun` 之类的时间 mock 库。为了一条极低频路径引入新依赖不划算

### Q4：为什么 9 个停止点只测 4 个？

**完整停止点全景**：

```
while 循环体：
  ├─ 安检失败 → bound_rejected / tool_setup_error     ❌ 没测（Harness 层，非 Loop 层）
  ├─ token 预算超限 → budget_exceeded                  ✅ 用例4
  └─ _should_continue_by_observation:
       ├─ 规则1 次数到上限 → reach_max_turns            ✅ 用例3
       ├─ 规则2 success=False → tool_failed             ❌ 没测（用例1的镜像）
       ├─ 规则3 超时 → timeout                          ❌ 没测（mock 时间太贵）
       ├─ 规则4 data>80 → enough_information             ✅ 用例1
       └─ 规则5 LLM判断
            ├─ CONTINUE → 继续（用例2测了这个分支）
            ├─ STOP → llm_stop                          ❌ 没测（逻辑太简单）
            └─ 异常 → llm_error                         ❌ 没测（异常路径）
```

**取舍标准**：改了 `main_graph.py` 最容易悄悄搞坏的路径 > 边界条件 > 异常路径。
