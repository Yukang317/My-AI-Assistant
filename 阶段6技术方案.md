## 〇、核心设计理念

阶段 6 不做"又一个 RAG 聊天机器人"，而是搭一个**可扩展的 Agent 框架**。借鉴四个典型智能体系统的设计精华，落实到个人 AI 助理的起步阶段。

### 四条指导原则

| 原则 | 来源 | 含义 |
|------|------|------|
| Agent 循环是小的部分，Harness 是大的部分 | Claude Code | 核心 while-true 循环 ~20 行，但权限/压缩/编排/可观测性层是真正的工程价值 |
| 能力扩展 = 写文档，不是写代码 | OpenClaw / Codex | 工具用 TOOL.md 说明书描述，LLM 读说明书知道怎么用，惰性加载 |
| 不信任模型，用确定性代码守住底线 | Codex BOUND / Harness | deny-first 权限、exit 2 hooks、沙箱执行——模型可以犯错，但系统不让错误落地 |
| 文件系统 > 向量数据库（做记忆） | Claude Code / OpenClaw | Agent 自身记忆用 Markdown 文件，可读可改可版本控制；向量库留给文档 RAG |

## 二、分阶段推进计划

### 阶段 6.1：Agent 框架骨架（当前）

**目标**：搭好框架，跑通一个完整的 Agent 循环

| 模块 | 文件 | 说明 |
|------|------|------|
| 状态定义 | `agent/state.py` | 主图 State + StateField 枚举 |
| 主图 | `agent/main_graph.py` | START → load → route → execute → synthesize → END |
| 上下文加载 | `agent/nodes/load_context.py` | PG 加载历史消息 + 读取 MEMORY.md |
| 意图路由 | `agent/router.py` | LLM 分析意图 → 返回目标工具名 |
| 结果合成 | `agent/nodes/result_synthesis.py` | 整合工具结果 + 保存消息 + 流式输出 |
| 工具基类 | `agent/tools/base.py` | 工具接口定义 |
| 工具注册表 | `agent/tools/registry.py` | 名称 → 函数的映射 |
| BOUND 约束 | `agent/bound.py` | DANGER_ZONES / NEVER_DO / IRON_LAWS |
| 示例工具 | `agent/tools/rag_search.py` | 把现有 RAG 检索封装为工具 |
| FastAPI 端点 | `step23_rag.py`（新增 /chat/graph） | 新的 Graph 聊天端点 |

### 阶段 6.2：工具扩展 + 记忆系统

- 网页搜索工具
- 文档管理工具（从现有端点迁移到 Graph）
- MEMORY.md 自动维护（LLM 定期提炼用户画像和偏好）
- 会话记忆文件（`memory/sessions/{id}.md`）

### 阶段 6.3：自我检查 + 多角色

- Codex 风格的验证循环（执行 → 检查 → 修复）
- Harness 风格的双角色（Router + Executor 分离）
- Skills 惰性加载（TOOL.md 按需注入 prompt）

### 阶段 6.4+：多智能体 + 工作流

- 多 Agent 协作（Planner / Executor / Reviewer）
- 工作流编排（DAG 依赖执行）
- 长期记忆向量检索

## 三、关键设计细节

### 3.1 工具注册机制（可插拔）

```python
# agent/tools/registry.py
TOOLS = {
    "rag_search": {
        "func": rag_search,
        "description": "搜索个人知识库，返回相关文档片段",
        "skill_md": "tools/rag/TOOL.md",   # 可选：详细说明书
        "bound": ["READ_ONLY"],             # BOUND 分类
    },
    "web_search": {
        "func": None,  # 6.2 实现
        "description": "搜索互联网获取实时信息",
        "skill_md": None,
        "bound": ["READ_ONLY", "NEEDS_CONFIRM"],
    },
}
```

新增工具 = 写一个函数 + 注册一行（+ 可选的 TOOL.md 说明书）。

### 3.2 BOUND 约束层

```python
# agent/bound.py
BOUND = {
    "DANGER_ZONES": ["/etc", "~/.ssh", "*.env", "*.key"],
    "NEVER_DO": ["删除文件", "执行shell命令", "修改系统配置"],
    "IRON_LAWS": ["回复必须使用中文", "不确定时反问用户", "不编造信息"],
}

def check_bound(action: str, target: str) -> bool:
    """每次工具调用前检查。返回 True = 允许，False = 拒绝。"""
```

### 3.3 记忆文件格式


## 四、与 sagt_agent 的对比

| 维度 | sagt_agent（参考） | personal_assistant（阶段6） |
|------|-------------------|---------------------------|
| 架构 | 主图 + 6 子图 | 主图 + 工具注册表（更轻量） |
| 扩展方式 | 每个功能是独立子图 | 每个功能注册到工具表 |
| 状态管理 | TypedState + StateField 枚举 | 同样的模式 |
| HITL | 企微 interrupt 确认 | 阶段 6.1 不涉及，预留接口 |
| 意图路由 | LLM 判断 → 子图 | LLM 判断 → 工具 |
| 工具调用 | 无 while-true 循环 | while-true 循环（更灵活） |

## 七、风险评估

| 风险 | 等级 | 应对 |
|------|------|------|
| LangGraph API 变化 | 低 | 使用稳定版，参考 sagt_agent 的固定版本 |
| while-true 循环死循环 | 中 | max_turns 硬限制 + token 预算上限 |
| LLM 意图判断不稳定 | 中 | fallback 到 general_chat，不强依赖路由准确性 |
| 工具注册表膨胀 | 低 | 惰性加载 TOOL.md，只在需要时读入上下文 |
