- etcd command 必须用 exec 模式（每个参数独立 `- ` 列表项），否则 Docker 把整个字符串当可执行文件名

**etcd 深度讲解 Q&A**：

- 4 个环境变量控制"呼吸系统"：写→膨胀→压缩(COMPACTION)→快照(SNAPSHOT)→上限(QUOTA)
- PG 16 Alpine 要求 `POSTGRES_PASSWORD` 不能为空

系统性梳理了 RAG 系统的组件关系和核心概念：

**用户 5 个核心问题**：
1. `MilvusVectorStore()` 和 `EmbeddingService()` 在 RAG 流程中是什么角色？
2. 为什么从 Milvus 读取全量 child_chunks 构建 BM25？
3. 为什么 `VectorRetriever` 只需传入依赖而 BM25 需要写构建逻辑？

**核心知识点**：
- **RAG 两大模块职责分离**：DocumentIndexer（文档摄入，离线）↔ RagService（知识检索，在线），共享底层组件但互不包含
- **懒加载模式**：全局变量 `_rag_service = None`，首次调用时初始化（~1-2 秒），后续直接复用（~0ms）
- **BM25 vs 向量检索**：BM25 内存索引（重启丢失需重建）vs Milvus 持久化索引（重启依然存在）
- **从 Milvus 反查构建 BM25**：单一数据源原则，简化架构，适合学习项目（生产环境通常独立持久化）
- **已识别 Bug**：`update_bm25_after_uploads()` 调用不存在的 `update_index()`，应为 `add_chunks()`

## 核心知识点记录

| 知识点 | 一句话 |
|--------|--------|
| RAG 两大模块职责分离 | DocumentIndexer 写入（离线）↔ RagService 读取（在线），共享底层组件但互不包含 |
| 懒加载 vs 直接初始化 | 懒加载保护启动流程：Milvus 挂了不影响普通聊天，首次初始化 ~1-2 秒，后续 ~0ms |
| BM25 内存索引 vs 向量持久化 | BM25 重启丢失需从 Milvus 重建；向量存在 Milvus 重启后直接可用 |
| 从 Milvus 反查构建 BM25 | 单一数据源原则，避免数据不一致，适合学习项目（生产环境通常独立持久化） |
| `object_key` vs `filename` | object_key 是全路径（含年月目录），天然唯一 |
| `update_bm25_after_uploads` 的两条路径 | 已初始化→增量更新；未初始化→跳过（下次全量构建包含） |
| Python import 全局解析 | 一个 import 失败会级联瘫痪所有依赖模块 |
| etcd 在 Standalone 中的角色 | Milvus Standalone 只是把分布式组件压缩进一个进程，内部仍用 etcd 做元数据协调 |
| Docker Compose command exec vs shell | YAML 列表格式是 exec 模式（每个参数独立），字符串是 shell 模式（/bin/sh -c 包装） |
| PG 16 Alpine 密码要求 | 不允许空密码，必须设置 `POSTGRES_PASSWORD` |

| **Claude Code** | while-true 无限循环 + 7 层权限 + 5 层上下文压缩 | Agent 循环只占 1.6% 代码，98.4% 是 harness 基础设施；deny-first 权限；文件记忆不用向量库 |
| **OpenClaw** | 星型拓扑（Gateway 单入口）+ Skills 惰性加载 + 4 层文件记忆 | 能力扩展 = 写 Markdown 说明书；SOUL.md 定义人格；NO_REPLY 沉默权 |
| **Codex** | SKILL.md 操作手册 + 自我检查闭环 + BOUND 约束 | 扫描→检测→修复→验证→循环；DANGER_ZONES/NEVER_DO/IRON_LAWS 用 hooks 硬拦截 |
| **Harness** | 三角色分工（Planner/Generator/Evaluator）+ 自进化 + 四重约束 | 上下文重置而非压缩；可证伪修改记录；约束层防止 Agent 越界 |

**四条共性规律**：
1. Agent 循环本身不值钱（~20 行），值钱的是外面的 harness——权限、压缩、编排、可观测性
2. 能力扩展 = 写文档，不是写代码——SKILL.md / SOUL.md / CLAUDE.md 都是同一思路
3. 不信任模型，用确定性代码守住底线——deny-first、exit 2 hooks、沙箱执行
4. 文件系统 > 向量数据库做记忆——Markdown 文件可读可改可版本控制

**最终架构（融合四家精华）**：

```
FastAPI → Main Graph → while-true loop → 工具注册表（可插拔）
                    ↓                     ↓
              BOUND 约束层           MEMORY.md 文件记忆
```

**阶段 6 分步计划**：
- 6.1：Agent 框架骨架（目录结构 + 主图 + 状态定义 + 意图路由 + 工具注册 + BOUND + 示例工具）
- 6.2：工具扩展（网页搜索 + 文档管理迁移）+ 记忆系统（MEMORY.md 自动维护）
- 6.3：自我检查闭环 + 双角色分工（Router/Executor）
- 6.4+：多智能体 + 工作流编排

**关键设计决策**：
- **工具 = 函数 + TOOL.md 说明书 + 注册**（不写完整子图，更轻量）
- **BOUND 约束从第一天就写进去**（DANGER_ZONES / NEVER_DO / IRON_LAWS）
- **记忆用 Markdown 文件起步**（memory/MEMORY.md + sessions/ + daily/）
- **while-true 循环**：max_turns 硬限制 + token 预算上限防死循环
