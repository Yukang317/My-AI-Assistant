### 本会话：Step 2-3 编码完成（6月2日）

#### Step 2：`rag/embedding.py` — Embedding 服务（C 模式）

实现了 `EmbeddingService` 类，支持 local/API 双模式：
- `__init__`：根据 mode 分发到 `_init_local_model` 或 `_init_api_client`
- `_init_local_model`：加载 BGE-large-zh-v1.5 模型（SentenceTransformer, device="cpu"）
- `_init_api_client`：初始化 OpenAI 兼容客户端（预留）
- `embed(texts)`：批量向量化，`normalize_embeddings=True`（L2 归一化 → Milvus IP 索引最优）
- `unload()`：释放模型内存（~1.3GB），含 `gc.collect()` 防御性回收

**本次核心讨论 3 个问题**：

| 问题 | 结论 |
|------|------|
| `gc.collect()` 是什么？ | Python 两层内存回收：引用计数（实时）+ GC（定期扫循环引用）。`del` 触发引用计数释放大部分内存，`gc.collect()` 手动触发 GC 清理循环引用——防御性措施，ECS 3.5GB 小内存不留隐患 |
| `encode(texts, normalize_embeddings=True)` 参数含义 | texts 支持单条/批量（批量更快，利用 SIMD）；normalize 做 L2 归一化，使向量模长=1，Milvus 用 IP（内积）索引直接等价余弦相似度，速度快一倍 |
| GPU vs CPU 部署？笔记本 GPU 算完传 ECS？ | **行业标准三档**：个人→CPU 同机推理（0 元/月），创业→Embedding API（几十元/月），企业→GPU 推理服务（几千+/月）。**笔记本 GPU 传 ECS 不是标准做法**——管道断裂、增量索引麻烦、笔记本不是 24h 服务器。BGE 在 CPU 上索引 100 篇 2~5 分钟，查询 0.1~0.3 秒，个人场景绰绰有余 |

**代码 Review 发现 6 个 bug**（缺逗号、API_URL→API_BASE、text→texts、f-string 漏 f、mode→model、elif/else 缩进错误），全部修复。

#### Step 3：`rag/storage/minio_client.py` — MinIO 客户端封装（C 模式）

实现了 `MinioClient` 类，封装 S3 兼容对象存储的五项操作：
- `_ensure_bucket()`：bucket 不存在时自动创建
- `upload(file_data, object_key, content_type)`：BytesIO 包装 → put_object
- `download(object_key)`：get_object → try/finally 读数据 + 释放连接
- `delete(object_key)`：remove_object
- `exists(object_key)`：stat_object 轻量检查

**代码 Review 发现 3 个主要 bug**：
1. `download()` 方法：`.data()` 误当方法调用 + 先 close 再 read 顺序反了 → try/finally 正确模式
2. `__init__` 里 `self.bucket = self.__ensure_bucket()` 无返回值 → `self.bucket` 变 None
3. `exists()` 方法是空的 → stat_object + try/except

**额外踩坑**：`__ensure_bucket`（双下划线）vs `_ensure_bucket`（单下划线）不匹配 → 名称改写导致 AttributeError；`from io import BytesIO` 但用了 `io.BytesIO` → NameError。
