## 三、协作模式（重要！）

### 原模式（已废弃）
~~"施工图纸"三步走：Claude 出图纸 → 用户手写代码 → Claude review~~

**废弃原因**：图纸粒度太粗，用户无法从零写出代码。

### 新模式：四步渐进式

```
A. 我写    →  Claude 写好完整可运行代码，逐行注释（风格参照用户的开发日志）
B. 你读懂  →  用户逐行阅读，在注释里写上自己的理解，跑起来验证
C. 你改    →  Claude 提一个小改动需求，用户动手改代码
D. 你扩展  →  Claude 提一个新功能，用户参照已有代码自己写
```

### 为什么这个模式适合
- 用户的开发日志本质上就是在做 B 步骤（读代码、理解、写注释）
- 用户做不了 A（从零写代码），但 B 能力很强
- C 和 D 是在**熟悉代码的基础上做增量改动**，比从零写容易得多
- 逐渐积累"自己写过的代码"，知识变成自己的

### 当前模式状态
**已演进到 C/D**：Python 后端用户自己写，JS 前端 Claude 写（用户不熟悉 JS）。RAG 模块具体模式待定。

### 三存储分工（重要概念，用户已理解）

```
原始文件  →  MinIO     (127.0.0.1:9000, bucket: personal-assistant)
向量搜索  →  Milvus    (127.0.0.1:19530)
文档台账  →  PostgreSQL (127.0.0.1:15432, database: personal_assistant)
```

- **MinIO**：文件柜，存 PDF/Word/MD 等原始文件
- **Milvus**：搜索引擎，存向量 + chunk 文本，做语义相似度检索
- **PostgreSQL**：档案目录卡片，存 documents 表（文档身份证）+ chunk_parent_map 表（父子块对应关系）
- **注意**：Milvus 内部也依赖一个 MinIO（docker-compose 管理），跟用户存文件的 MinIO 是同一个实例的不同 bucket，互不干扰

### 刻意不用的技术（现阶段）
- **LangGraph**：简单请求-响应不需要图编排，等需要多步推理时再引入
- **Redis**：暂无缓存需求（阶段6 语义缓存时会引入）
- **企业微信 SDK**：先做 Web 端
- **GraphRAG / 知识图谱**：个人文档量小，复杂的图检索收益低，远期再考虑
- **Agentic RAG（多步推理）**：等阶段 7-8 引入 Agent 时自然获得

### 长期技术演进方向
1. LangGraph → 多步 Agent 推理
2. MCP 协议 → 工具生态集成
3. 企业微信/飞书 SDK → 多平台接入
4. 长期记忆 + 用户画像 → 个性化
